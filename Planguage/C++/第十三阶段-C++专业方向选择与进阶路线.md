# 第十三阶段：C++ 专业方向选择与进阶路线

> 前置要求：完成 C++ 基础主线，至少掌握现代 C++、工程化、并发和性能分析。  
> 学习方式：选择一条主方向长期深入，可搭配一条辅助方向。  
> 阶段目标：把语言知识转化为具体领域的设计、调试、交付和性能能力。

---

## 一、从语言学习转向领域学习

完成 C++26 知识主线不代表已经“学完 C++”。真正的专业能力来自：

```text
C++ 语言能力
    + 领域知识
    + 工具链
    + 真实项目
    + 调试与性能证据
    + 长期维护经验
```

从本阶段开始，不再以记忆语言特性为核心，而以交付可靠领域系统为核心。

### 1.1 选择方向的五个问题

1. 你更喜欢贴近硬件，还是贴近产品界面？
2. 你更看重低延迟、吞吐、实时性、可视化还是跨平台体验？
3. 目标岗位或项目最常用哪些库和平台？
4. 你能获得哪些真实数据、设备或部署环境？
5. 哪个方向能让你持续完成至少三个逐级项目？

### 1.2 不要同时深入所有方向

建议采用 T 型结构：

- 横向：通用 C++、测试、构建、并发、性能和调试。
- 纵向：一个主要专业方向。
- 辅助：与主方向强关联的第二方向。

常见组合：

- 系统网络 + 高性能计算。
- 游戏引擎 + 图形渲染。
- 嵌入式 + 实时系统。
- 桌面应用 + 跨平台工程。
- 编译器 + 静态分析。

## 二、所有方向共同的专业基础

### 2.1 计算机基础

- 数据结构与算法。
- 操作系统：进程、线程、虚拟内存、调度、文件系统。
- 计算机网络：TCP/IP、DNS、HTTP、TLS。
- 计算机组成：缓存、流水线、SIMD、内存层次。
- 数据库：事务、索引、并发控制。

### 2.2 工程能力

- Git 分支、代码审查和可回滚提交。
- CMake、包管理和可复现构建。
- 单元、集成、端到端和压力测试。
- Sanitizer、静态分析、fuzzing。
- CI/CD、版本和发布管理。
- 日志、指标、trace 和故障复盘。

### 2.3 设计能力

- API 和所有权设计。
- 错误模型与异常边界。
- 并发不变量和关闭协议。
- 数据格式及兼容性。
- 性能预算与容量规划。
- 威胁建模与不可信输入处理。

### 2.4 证据化交付

每个专业项目都应保存：

- 需求与非目标。
- 架构和关键取舍。
- 测试策略。
- 性能基线和 profiler 证据。
- 已知限制。
- 构建、运行和部署说明。
- 复盘与下一步。

## 三、方向一：系统与基础设施

### 3.1 适合人群

喜欢操作系统、网络、服务器、数据库基础设施、低延迟服务和复杂故障诊断。

### 3.2 核心知识

- 进程、线程、信号和 IPC。
- 虚拟内存、mmap、页缓存。
- 文件描述符和异步 I/O。
- TCP/UDP、拥塞、半包和粘包。
- epoll、kqueue、IOCP、io_uring。
- TLS、证书与安全边界。
- 序列化、RPC、服务发现。
- 限流、熔断、重试和背压。
- 高可用、幂等与故障恢复。

### 3.3 C++ 技能重点

- RAII 包装系统句柄。
- 协程和异步生命周期。
- 原子、内存模型和并发队列。
- 零拷贝、scatter/gather I/O。
- buffer 所有权和生命周期。
- endian、协议解析和 fuzzing。
- P99/P999 延迟分析。

### 3.4 工具与库

- Boost.Asio / standalone Asio。
- gRPC、Protobuf、FlatBuffers 或 Cap'n Proto。
- OpenSSL/BoringSSL。
- Linux perf、eBPF、strace、Wireshark。
- Windows ETW、WPA、WinDbg。

库的选择应依据目标平台和团队，不要求全部掌握。

### 3.5 递进项目

1. RAII 文件与 socket 包装器。
2. 支持多客户端的 TCP 聊天服务器。
3. 带长度帧、超时和背压的协议库。
4. 线程池/协程 HTTP 服务。
5. 持久化 KV 存储或简化消息队列。
6. 带指标、trace、限流和优雅关闭的服务框架。

### 3.6 验收标准

- 不阻塞事件循环。
- 所有连接有超时、取消和关闭路径。
- 协议解析通过 fuzzing。
- 过载时内存有界。
- 部分读写、断连和系统错误正确处理。
- 有吞吐、尾延迟和资源占用报告。

## 四、方向二：游戏开发与实时图形

### 4.1 适合人群

喜欢实时交互、图形、物理、内容管线和帧级性能优化。

### 4.2 数学基础

- 向量、矩阵和四元数。
- 坐标系与变换。
- 投影、视锥和裁剪。
- 插值、曲线和数值积分。
- 碰撞检测基础。

### 4.3 渲染知识

- GPU 管线、shader 和材质。
- 顶点/索引缓冲。
- 纹理、采样和 mipmap。
- 光照、阴影、PBR。
- frame graph、同步和资源屏障。
- CPU/GPU 并行与帧延迟。

### 4.4 引擎架构

- 游戏循环和固定时间步。
- ECS 与数据导向设计。
- 资源加载和热更新。
- 场景、动画和物理。
- Job System。
- 编辑器、序列化和内容构建。

### 4.5 工具与 API

- SDL、GLFW。
- OpenGL 用于入门。
- Vulkan、Direct3D 12、Metal 用于现代显式图形 API。
- RenderDoc、PIX、Nsight Graphics。
- Unreal Engine 或自研小引擎。

### 4.6 递进项目

1. 2D 游戏与固定时间步。
2. 基础 3D 查看器。
3. PBR 渲染器。
4. ECS 与并行 Job System。
5. 资源流式加载和热重载。
6. 小型编辑器或完整可玩 Demo。

### 4.7 验收标准

- 帧时间而非只看平均 FPS。
- CPU/GPU profile 分离。
- 资源生命周期和 GPU 同步正确。
- 不在主线程进行不可控阻塞 I/O。
- 固定步模拟可复现。
- 有低端目标硬件性能预算。

## 五、方向三：嵌入式与实时系统

### 5.1 适合人群

喜欢硬件、确定性、资源受限环境、控制系统和设备可靠性。

### 5.2 核心知识

- MCU、寄存器、GPIO、UART、SPI、I2C、CAN。
- 中断、DMA、定时器和看门狗。
- 链接脚本、启动代码和内存映射。
- 裸机与 RTOS。
- 优先级反转、截止时间和抖动。
- 功耗、闪存和 RAM 预算。
- 故障安全、升级和恢复。

### 5.3 C++ 子集与模式

- freestanding C++。
- 编译期配置和强类型寄存器。
- 固定容量容器。
- 无动态分配或分阶段分配。
- 异常/RTTI 策略按平台决定。
- ISR 中禁止阻塞和不安全操作。
- RAII 临界区和外设句柄。

### 5.4 标准与规范

根据行业了解：MISRA C++、AUTOSAR C++、功能安全和安全编码规范。规范是约束框架，不代替领域风险分析和验证。

### 5.5 工具

- 交叉编译器和 CMake toolchain。
- JTAG/SWD、GDB、逻辑分析仪、示波器。
- QEMU 或厂商模拟器。
- 静态分析和栈深度分析。
- HIL（硬件在环）测试。

### 5.6 递进项目

1. GPIO 驱动与强类型寄存器接口。
2. 无动态分配的串口协议栈。
3. RTOS 传感器采集系统。
4. CAN 节点与诊断协议。
5. 双分区固件升级和回滚。
6. 带故障注入的实时控制系统。

### 5.7 验收标准

- 最坏执行时间和内存有预算。
- 中断与线程共享状态同步正确。
- 断电和通信错误有恢复策略。
- 编译器、链接器和目标硬件配置可复现。
- HIL 和故障注入覆盖关键路径。
- 不把桌面环境假设带入设备代码。

## 六、方向四：高性能计算与科学计算

### 6.1 适合人群

喜欢数值算法、并行计算、模拟、机器学习基础设施和硬件性能。

### 6.2 数学与数值基础

- 线性代数。
- 数值稳定性和条件数。
- 浮点误差和舍入。
- 优化方法。
- 微分方程和离散化（按领域）。
- 随机算法和统计。

### 6.3 并行层次

```text
单核 SIMD → 多核线程/OpenMP → 多机 MPI → GPU CUDA/HIP/SYCL
```

先优化算法复杂度和数据移动，再扩展并行度。

### 6.4 C++ 技能重点

- mdspan、simd 和 linalg。
- 表达式模板与惰性计算。
- AoS/SoA 和块状布局。
- NUMA、cache blocking。
- reduction 和可复现浮点结果。
- 异构内存和数据传输。

### 6.5 工具与库

- BLAS/LAPACK、Eigen、Kokkos 等。
- OpenMP、MPI。
- CUDA、HIP 或 SYCL。
- VTune、Nsight Systems/Compute、perf。
- Python 绑定可使用 pybind11，便于科学工作流。

### 6.6 递进项目

1. 标量与 SIMD 向量运算库。
2. cache-blocked 矩阵乘法。
3. OpenMP 并行数值算法。
4. MPI 分布式 stencil 或矩阵任务。
5. CUDA/SYCL 求解器。
6. CPU/GPU 多后端科学计算库。

### 6.7 验收标准

- 有数值正确性和误差标准。
- 报告硬件、编译器和数据规模。
- 测量计算与数据传输时间。
- 有强缩放和弱缩放结果。
- 处理 reduction 顺序差异。
- 标量参考实现用于验证优化版本。

## 七、方向五：桌面与跨平台应用

### 7.1 适合人群

喜欢可视化产品、工具软件、编辑器、跨平台用户体验和长期维护。

### 7.2 核心知识

- GUI 事件循环和主线程规则。
- Model/View 或状态管理。
- 布局、输入法、DPI 和可访问性。
- 本地化、Unicode 和时区。
- 文件格式、设置和自动保存。
- 插件系统和沙箱边界。
- 安装、更新、签名和崩溃报告。

### 7.3 工具与框架

- Qt 是完整跨平台方案。
- wxWidgets、JUCE、Dear ImGui 等用于不同场景。
- 平台原生 API：WinUI/Win32、Cocoa 等。
- CMake、CPack、平台打包工具。

### 7.4 架构重点

- UI 与业务逻辑分离。
- 后台任务不阻塞事件循环。
- 对象销毁后回调不悬空。
- 可取消任务和进度反馈。
- 文档修改状态、自动保存和崩溃恢复。

### 7.5 递进项目

1. 设置持久化的小工具。
2. 带后台扫描的文件管理器。
3. 文本/图像编辑器。
4. 插件化数据查看器。
5. 跨平台更新与崩溃恢复系统。
6. 可发布的完整桌面产品。

### 7.6 验收标准

- UI 操作流畅且无长时间阻塞。
- 高 DPI、多语言和键盘操作可用。
- 后台任务可取消并安全关闭。
- 文件写入具有事务性。
- Windows/macOS/Linux 至少目标平台 CI 验证。
- 安装、升级和卸载流程可测试。

## 八、方向六：编译器、静态分析与开发工具

### 8.1 适合人群

喜欢语言、类型系统、程序分析、IDE、重构工具和构建工具链。

### 8.2 编译器基础

- 词法和语法分析。
- AST。
- 名称查找和作用域。
- 类型检查与推导。
- 中间表示 IR。
- 控制流图和数据流分析。
- 优化和代码生成。
- 调试信息和目标文件。

### 8.3 工具方向

- Clang AST Matchers。
- LibTooling。
- clang-tidy 自定义检查。
- LLVM IR 和 Pass。
- Language Server Protocol。
- Tree-sitter 等解析工具。

### 8.4 静态分析知识

- CFG。
- reaching definitions。
- liveness。
- abstract interpretation。
- 符号执行。
- 污点分析。
- interprocedural analysis。
- 误报、漏报和可解释诊断。

### 8.5 递进项目

1. 表达式解释器。
2. 带变量和类型检查的小语言。
3. 字节码虚拟机。
4. clang-tidy 自定义规则。
5. LLVM 优化 Pass。
6. LSP 或自动重构工具。

### 8.6 验收标准

- 语法错误可恢复并提供位置诊断。
- AST/IR 有明确不变量。
- 测试包含正例、反例和黄金输出。
- 静态分析报告误报/漏报特征。
- 变换保持语义并通过差分测试。
- 能处理真实项目而不只玩具输入。

## 九、辅助方向：安全工程

安全不应只作为独立岗位知识，所有方向都应覆盖：

- 输入长度和整数溢出。
- 内存安全与生命周期。
- 权限最小化。
- 密钥和敏感数据处理。
- 依赖与供应链安全。
- 模糊测试和漏洞响应。
- 威胁建模。

建议项目至少完成一次：攻击面枚举、fuzzing、依赖审计、漏洞修复和回归测试。

## 十、如何制定个人路线

### 10.1 三层目标

为选定方向分别设定：

- 入门：能完成单机、单组件项目。
- 进阶：能处理并发、故障、性能和跨平台。
- 专业：能设计架构、建立指标并维护生产系统。

### 10.2 十二个月示例

1. 第 1～2 月：补齐领域理论和工具。
2. 第 3～4 月：完成第一个小项目。
3. 第 5～7 月：完成带测试和性能报告的中型项目。
4. 第 8～10 月：加入真实平台、设备或开源项目约束。
5. 第 11～12 月：重构、发布、写设计文档和复盘。

### 10.3 每个项目必须增加的新难度

- 第一个：功能正确。
- 第二个：错误和资源正确。
- 第三个：并发或实时约束。
- 第四个：性能和容量目标。
- 第五个：部署、兼容和长期维护。

不要连续写五个只有界面不同、技术难度相同的练习。

## 十一、作品集标准

一个高质量项目仓库应包含：

- 清楚的 README 和演示。
- 架构图及设计决策记录。
- 一键配置、构建和测试。
- 固定依赖和支持矩阵。
- 单元、集成和失败路径测试。
- CI 状态。
- benchmark/profile 报告。
- Sanitizer/静态分析说明。
- release 和变更日志。
- 已知限制与后续计划。

代码量不是核心指标。一个边界清晰、测试充分、可复现的 5,000 行项目，通常比无法构建的 50,000 行项目更有说服力。

## 十二、开源协作路线

1. 从构建项目和运行测试开始。
2. 阅读贡献指南与架构文档。
3. 修正文档、测试或小缺陷。
4. 为问题建立最小复现。
5. 提交范围小、动机清晰的变更。
6. 响应代码审查并补充测试。
7. 逐步承担模块维护。

不要从大规模重写开始。真实协作能力包括理解现有约束、兼容性和维护者取舍。

## 十三、专业能力自测

### 通用问题

1. 该方向最关键的三个非功能需求是什么？
2. 典型系统的生命周期和错误边界在哪里？
3. 哪些数据不可信？
4. 如何定义性能和容量指标？
5. 如何复现线上或设备故障？
6. 如何发布、升级和回滚？
7. 哪些工具能观察系统内部行为？
8. 哪些优化会破坏可移植性或正确性？
9. 如何测试最坏情况而非只测试正常流程？
10. 项目停止维护后，其他人能否接手？

### 完成检查清单

- [ ] 已选择一个主方向和可选辅助方向。
- [ ] 制定至少六个月路线。
- [ ] 完成一个小型和一个中型项目。
- [ ] 中型项目具备 CI、测试和诊断。
- [ ] 有真实 profiler 或容量数据。
- [ ] 有一份架构决策记录。
- [ ] 处理过一次故障注入或 fuzzing 发现的问题。
- [ ] 完成至少一次开源贡献或正式代码审查。
- [ ] 能解释项目最重要的三个取舍。
- [ ] 下一项目比当前项目增加明确的新约束。

## 十四、常见误区

1. 继续收集语法知识，却不做领域项目。
2. 同时学习六个方向，任何一个都不深入。
3. 只做功能演示，不处理错误和关闭。
4. 只看平均性能，不看尾延迟和最坏情况。
5. 只在自己的机器上构建。
6. 项目没有测试、文档和可复现数据。
7. 为使用新标准特性而使用，不考虑工具链。
8. 把框架 API 熟练等同于领域理解。
9. 忽略安全、部署、升级和回滚。
10. 只写新项目，从不维护和重构旧项目。
11. 只展示成功优化，不记录失败实验。
12. 追求代码量而不是问题难度和交付质量。

## 十五、长期学习方法

保持三个循环：

```text
知识循环：阅读 → 实验 → 解释
工程循环：设计 → 实现 → 测试 → 发布 → 维护
性能循环：基线 → profile → 假设 → 优化 → 验证
```

持续关注：

- WG21 标准与编译器实现动态。
- 主方向的重要库、会议和论文。
- 真实事故报告和性能复盘。
- 开源项目的设计与代码审查。

完成第十三阶段没有统一终点。真正的阶段成果，是你能在一个领域中独立定义问题、选择技术、实现系统、验证质量，并对长期维护负责。
