# Ceph 性能分析与调试实战：从慢请求到源码定位的方法论

## 这篇文章要解决什么问题

到现在为止，我们已经把 Ceph 的很多核心模块都拆开讲过了：

- MON 的 quorum 和 Paxos
- OSD 的请求调度与 PG 执行
- PrimaryLogPG 的主副本写路径
- BlueStore 的提交语义
- CephFS 的 MDS、caps、缓存一致性与排障
- RGW 的请求链和元数据模型

这些内容如果都理解了，你会发现自己又会进入一个新的阶段：

- 懂原理了，但线上慢了时，还是不知道第一步该看什么

这其实是很典型的进阶门槛。

因为真正的性能分析和问题定位，不是把某个模块原理背熟就自动会了，而是要建立一条可复用的方法链：

- 先怎么判断是“真的慢”还是“观测口径混乱”
- 先看 health、日志还是 perf counter
- 什么时候该看 admin socket
- `dump_ops_in_flight`、`dump_historic_ops` 到底应该怎么读
- 什么时候该从命令行切到源码入口
- 怎样避免一上来就陷入无穷多指标、日志和函数名

如果只允许用一句话先给结论，那就是：

**Ceph 性能分析真正有效的方法，不是先背一堆命令，而是建立一条稳定的定位链：先把症状分型，再用 health 与日志确定“谁在慢”，再用 admin socket 和 perf counter回答“具体卡在哪一层”，最后再顺着守护进程入口、慢请求追踪点和核心子系统调用链进入源码；也就是说，排障的关键不是信息多，而是能不能按层收缩问题空间。**

这一篇的目标，就是把这条方法链真正讲清楚。

## 先建立第一条边界：Ceph 调试不是“先开 debug 日志”，而是先分清症状类型

这是理解整篇文章的第一步，也是最重要的一步。

很多人一遇到 Ceph 性能问题，第一反应就是：

- 开大日志
- 看 Grafana
- 搜 slow request
- 到处跑 `ceph daemon ...`

这些动作本身都不算错，但如果没有先分型，通常会很快掉进：

- 指标太多
- 日志太杂
- 看了很多信息却没有收敛

### 更成熟的第一步应该是什么

- 先把“慢”翻译成更具体的症状类型

比如：

- 集群级健康告警在报慢
- 某个 OSD 慢请求很多
- 某个 MDS metadata 操作卡住
- 某条客户端 IO 路径尾延迟抬高
- 整体吞吐下降但没有明显 slow request

### 这意味着什么

说明性能分析的第一原则不是：

- 先找工具

而是：

- **先定义症状。**

只有这一步做对了，后面的观测和源码定位才会有方向。

## 第 1 层：为什么 health 告警和日志最适合回答“谁在慢”

一旦症状先被定义出来，最先值得看的通常不是细粒度 counter，而是：

- health 告警
- 关键日志

### 为什么

因为它们最适合先回答一个粗粒度但特别关键的问题：

- 到底是谁在慢？

比如：

- 是集群级 `SLOW_OPS`
- 是 `MDS_SLOW_REQUEST`
- 是 BlueStore slow op
- 是某个 OSD 一直在抱怨 blocked request

### 这意味着什么

说明 health 和日志在方法链里的作用更像：

- 第一层分流器

它们未必能直接给你最终根因，但很适合先把问题缩到：

- OSD
- MDS
- MON
- BlueStore
- 网络/后端

这几个方向中的一个。

## 第 2 层：为什么 slow request 是 Ceph 排障里最值得优先抓的“现场证据”

Ceph 的性能问题里，一个最有价值的现象词就是：

- slow request

### 为什么它特别值钱

因为它不是抽象的“感觉变慢了”，而是守护进程自己在说：

- 某些请求长时间没有完成

这就把问题从：

- 模糊体感

收缩成：

- 一组具体的、可跟踪的、带生命周期信息的操作

### 所以 slow request 真正的价值是什么

- 它给了你一条可以反向追踪的操作线索

这比单纯看平均延迟曲线更适合继续深挖。

## 第 3 层：为什么 `dump_ops_in_flight` / `dump_historic_ops` 比泛看日志更有结构

继续往 slow request 深一层，你会发现 Ceph 给了非常关键的结构化入口：

- `dump_ops_in_flight`
- `dump_historic_ops`

### 为什么这两个入口特别重要

因为它们不像普通日志那样是线性文本流，而更像：

- 当前或历史请求的结构化快照

### 它们分别更适合看什么

## `dump_ops_in_flight`

- 当前还没完成、仍在执行中的请求

## `dump_historic_ops`

- 已发生过的慢操作历史样本

### 这意味着什么

说明它们在方法链里真正回答的是：

- 请求到底卡在哪个阶段

而不是：

- 系统总体慢不慢

这一步很关键，因为从这里开始你会从“宏观现象”进入“具体操作路径”。

## 第 4 层：为什么看 slow op 时最重要的不是总时长，而是“最后一个 event”

这是读 `dump_ops_in_flight` 时最值钱的一个实战直觉。

很多人第一次看 slow op，会先被：

- duration
- age
- request id

吸引。

这些当然有价值，但更关键的往往是：

- 最后一个 event

### 为什么

因为它最接近在回答：

- 当前请求最后卡在了什么地方

比如它可能暗示：

- 在等 PG
- 在等副本
- 在等磁盘提交
- 在等锁
- 在等 MDS
- 在等 OSD

### 所以最准确的理解方式是

- 总时长告诉你“问题严重”
- 最后一个 event 告诉你“问题更可能在哪”

这就是为什么它在方法论里特别关键。

## 第 5 层：为什么 admin socket 是 Ceph 调试里最像“现场检查口”的东西

继续往守护进程内部看，另一个特别重要的概念就是：

- admin socket

### 为什么它这么关键

因为它让你不必通过：

- 修改代码
- 重启服务
- 抓一大堆通用日志

就能直接向守护进程问一些非常内部的问题：

- 当前有哪些 ops
- perf counter 现在是多少
- 某些子系统状态是什么
- 是否能临时调整 debug level

### 所以 admin socket 在方法链里的位置是什么

- 它是“从外部监控切入守护进程内部状态”的最短通道之一

这也是为什么：

- `ceph daemon ...`

在 Ceph 调试里会反复出现。

## 第 6 层：为什么 `perf schema` 和 `perf dump` 最好成对使用

perf counter 是 Ceph 调试里另一个核心入口，但也最容易被误用。

### 为什么会误用

因为很多人会直接：

- `perf dump`

然后看到一大堆数字，不知道该看什么。

更成熟的方式通常是：

- 先 `perf schema`
- 再 `perf dump`

### 为什么

因为：

- `schema` 回答“有哪些指标、它们是什么类型、语义大概是什么”
- `dump` 回答“当前实时值是多少”

### 这意味着什么

说明 perf counter 的使用顺序不应该是：

- 先看值再猜名字

而应该是：

- 先理解指标面，再看当前数据

这点非常实用。

## 第 7 层：为什么 perf counter 最适合回答“哪个子系统异常活跃或异常阻塞”

继续往 perf counter 再深一点，最值得建立的直觉是：

- perf counter 不是为了直接告诉你“根因”

它更擅长回答：

- 哪个子系统的行为模式异常了

### 比如它最适合暴露什么

- 某类请求数暴涨
- 某类延迟累计异常
- 某个队列堆积
- 某个后台线程/模块明显更忙

### 这意味着什么

说明 perf counter 在调试方法链里的作用更像：

- “缩小到具体子系统”

它不是万能诊断器，但非常适合从：

- OSD 很慢

继续收缩成：

- PG peering 慢
- recovery 压力大
- BlueStore 提交延迟高
- 某个前端请求类型异常

这类更窄的结论。

## 第 8 层：为什么 perf histogram 在分析尾延迟和分布问题时比单值 counter 更有价值

再进一步，很多性能问题单看 average 或累计计数根本不够。

比如：

- 平均时延还行，但尾延迟很差
- 某类请求只有少数特别慢

这时更有价值的往往是：

- perf histogram

### 为什么

因为 histogram 回答的是：

- 分布

而不是：

- 单一平均值

### 这意味着什么

说明当你怀疑问题是：

- 尾延迟
- 某类大小/某类请求分布异常

时，histogram 往往比普通 counter 更贴近真实问题形态。

这一步对“为什么用户感觉卡，但均值看起来不差”的问题特别有效。

## 第 9 层：为什么日志在 Ceph 里只能精确地“临时打开”，不能长期蛮开

日志当然很重要，但也必须正面讲一个现实：

- Ceph debug 日志是有成本的

### 为什么

因为高 debug level 可能带来：

- 更大 IO
- 更高 CPU
- 更快的日志膨胀
- 甚至反过来影响线上性能

### 这意味着什么

说明在 Ceph 里，日志更适合：

- 带着问题、带着时间窗、带着子系统范围去临时打开

而不适合：

- 一上来全开、长期开

### 所以更成熟的思路是什么

- 先用 health / perf / admin socket 收缩问题空间
- 再临时抬高特定子系统 debug level

这样才更稳。

## 第 10 层：为什么“宏观观测”和“守护进程自检”必须配合，而不是二选一

很多团队容易在观测上走两个极端：

## 极端一

- 只看 Grafana / Prometheus

## 极端二

- 只盯守护进程日志和 admin socket

其实都不完整。

### 为什么

因为这两类工具回答的是不同层次的问题：

- 宏观观测回答“系统整体哪里不正常”
- 守护进程自检回答“这个守护进程内部现在卡在哪”

### 所以最成熟的组合方式是什么

- 先用宏观观测做范围收缩
- 再用守护进程自检做子系统定位

这比只依赖其中一种方式效率高得多。

## 第 11 层：为什么源码定位不应该从“全文搜索某个日志字符串”开始

一旦进入源码定位，很多人会本能地：

- 搜慢日志字符串
- 然后顺着结果到处跳

这当然有时能用，但很容易把自己带进：

- 零散函数
- 无上下文的辅助路径
- 与主链关系不明的边角逻辑

### 更稳的源码入口应该是什么

- 守护进程主入口
- admin socket 命令注册点
- slow op/OpTracker 通用底座
- 核心子系统入口类

### 为什么

因为你真正要找的是：

- “这个问题在源码里的主战场”

而不是：

- 某个偶然打印日志的角落

这就是为什么源码定位也必须先分层。

## 第 12 层：为什么守护进程入口、admin socket 注册点、OpTracker 三者是最稳的源码定位起点

如果把源码定位再压缩一下，我最推荐优先抓三类入口：

## 守护进程主入口

- 比如 `ceph_osd.cc`
- `ceph_mds.cc`
- `rgw_main.cc`

## admin socket 命令注册点

- 比如 OSD/MDS 的命令处理入口

## `TrackedOp` / `OpTracker` 这类慢请求共用底座

- 它们连接了“线上慢请求现象”和“代码里的请求生命周期”

### 为什么这三类特别稳

因为它们天然对应了排障链的三步：

- 这个进程是谁
- 我可以向它问什么
- 它怎样追踪请求生命周期

这会比盲搜随机函数高效很多。

## 第 13 层：为什么从慢请求到源码定位，真正的关键是“问题翻译”

讲到这里，可以把方法论再收束成一句更本质的话：

- 调试不是找更多信息，而是做更好的问题翻译

### 具体在翻译什么

- 用户说“卡” -> 你翻译成 slow request / 队列积压 / 尾延迟
- health 告警说慢 -> 你翻译成具体守护进程和子系统
- `dump_ops_in_flight` 的 event -> 你翻译成代码路径里的阶段
- perf counter 的异常 -> 你翻译成某个子系统行为模式
- 最后再翻译成源码入口和调用链

### 这说明什么

真正成熟的 Ceph 调试不是：

- 把所有工具都背下来

而是：

- 能把问题一层层翻译成更接近代码现实的形式

这才是方法论的核心。

## 第 14 层：一条最实用的 Ceph 性能分析顺序应该长什么样

讲到这里，可以把最实用的方法压缩成一套固定顺序。

## 第一步：先定义症状

比如：

- 集群慢
- OSD 慢
- MDS 慢
- 客户端尾延迟高
- 吞吐下降

## 第二步：看 health 和关键日志

回答：

- 谁在慢
- 慢到什么程度
- 是否已有明确告警类别

## 第三步：用 admin socket 看结构化现场

比如：

- `dump_ops_in_flight`
- `dump_historic_ops`
- 其他子系统状态

## 第四步：用 perf schema + perf dump 缩到子系统

回答：

- 哪类 counter 异常
- 哪个模块更忙或更堵

## 第五步：必要时短时打开针对性 debug 日志

回答：

- 关键流程在这个时间窗里到底发生了什么

## 第六步：从守护进程入口、admin socket 注册点和 OpTracker 进入源码

回答：

- 这条慢请求路径在代码里到底走到哪了

这套顺序很朴素，但非常稳。

## 第 15 层：为什么“先缩小问题空间”比“先找根因”更重要

这是整篇最想让读者留下的认知。

很多人一上来就想：

- 直接找到根因

但 Ceph 这种系统太复杂了，直接追根因往往会让你同时面对：

- OSD
- MON
- MDS
- 网络
- BlueStore
- CRUSH
- 客户端

这几乎不可能高效。

### 更成熟的做法是什么

- 先缩小问题空间

比如先缩到：

- 只剩某个 daemon
- 只剩某个 PG/请求阶段
- 只剩某个 perf 子系统
- 只剩某个代码入口

### 这意味着什么

说明 Ceph 调试的真正核心不是：

- “一眼看穿根因”

而是：

- **把大问题稳定地压缩成小问题。**

这才是方法论真正值钱的地方。

## 第 16 层：把整条 Ceph 调试主线压缩成一张图

如果这一篇只记一张图，我建议记下面这张：

```text
症状
  ->
health / 日志
  ->
锁定 daemon 与慢请求类型
  ->
admin socket
  ->
dump_ops_in_flight / dump_historic_ops / 结构化现场
  ->
perf schema + perf dump + histogram
  ->
锁定异常子系统

  ->

源码调用链
```

这张图里最关键的一句话是：

- **Ceph 调试不是从源码往外猜，而是从症状往里逐层收缩。**

## 用一句话重新概括这篇

如果把这篇全部内容压缩成一句尽量准确的话，我会这样说：

**Ceph 性能分析与调试最有效的方法，不是先背命令或先搜源码，而是先把“慢”分型，再通过 health 和日志锁定异常守护进程，用 admin socket 的结构化现场和 perf counter/perf histogram 缩小到具体子系统，必要时短时提高相关 debug 日志，最后再从守护进程入口、admin socket 命令注册点和 OpTracker 进入源码；也就是说，真正高效的定位方法不是信息越多越好，而是问题空间收缩得越稳定越好。**

## 把整篇压缩成一条最短骨架

如果你只想记一条骨架，可以记成下面这样：

```text
先定义症状
  ->
用 health/日志判断谁在慢
  ->
用 admin socket 看慢请求现场
  ->
用 perf counter/histogram 缩到子系统
  ->
短时开针对性日志
  ->
再进源码入口和调用链
```

只要这条骨架记住了，Ceph 的性能分析和源码定位通常就不会一上来乱掉。

## 初学者最容易混淆的 10 个点

### 1. 认为性能分析首先是背命令

不对。首先是分型问题。

### 2. 认为一慢就该直接开大日志

不对。日志应该最后精确地开。

### 3. 认为 slow request 只是告警文案

不对。它是最值钱的现场证据之一。

### 4. 认为 `dump_ops_in_flight` 主要看总时长

不对。通常更该先看最后一个 event。

### 5. 认为 perf counter 直接 dump 就够了

不对。通常应该先看 schema。

### 6. 认为 histogram 只是高级附加项

不对。尾延迟和分布问题里它往往更关键。

### 7. 认为宏观监控和守护进程自检选一个就行

不对。它们回答不同层次的问题。

### 8. 认为源码定位应该从搜索日志字符串开始

不对。更稳的是从进程入口和 admin socket 注册点开始。

### 9. 认为调试目标是马上找到根因

不对。更现实的目标是先缩小问题空间。

### 10. 认为工具越多，定位就越快

不对。顺序和分层比工具数量更重要。

## 这一篇最应该留下的 5 个直觉

### 直觉一：先分症状，再选工具

这是第一原则。

### 直觉二：slow request 是连接现象与代码的关键桥

这点必须立住。

### 直觉三：admin socket 是守护进程内部状态的短通道

非常值钱。

### 直觉四：perf counter 更适合定位“哪个子系统异常”

不是直接给最终根因。

### 直觉五：源码定位本质上是问题翻译和空间收缩
