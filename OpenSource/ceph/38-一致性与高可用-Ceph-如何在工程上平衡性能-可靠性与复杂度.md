# 一致性与高可用：Ceph 如何在工程上平衡性能、可靠性与复杂度

## 这篇文章要解决什么问题

写到这里，我们其实已经把 Ceph 的很多关键模块都拆开讲过了：

- `MON` 的 quorum、选举和 Paxos
- `OSD` 的主副本写路径
- `PG` 的 peering、recovery、backfill
- `CRUSH` 的故障域和放置规则
- `BlueStore` 的本地持久化
- `RGW multisite` 的异步同步
- `CephFS` 的 MDS、caps 和缓存一致性

但如果只是一篇篇分开看，读者很容易产生另一种新的困惑：

- Ceph 到底算强一致，还是最终一致？
- 为什么有些地方看起来特别“硬”，比如 MON quorum；有些地方又明显接受异步收敛，比如 RGW multisite？
- Ceph 的高可用，到底是靠 Paxos、靠副本、靠 PG，还是靠 CRUSH？
- 为什么系统里同时存在“多数派”“主副本”“故障域隔离”“后台恢复”这些看起来不完全同类的机制？
- Ceph 到底是怎样在性能、可靠性和实现复杂度之间做取舍的？

如果只允许用一句话先给结论，那就是：

**Ceph 并没有在整个系统里套用一种统一的一致性模型，而是按层分工：监控与集群地图这类控制面状态用 `MON + quorum + Paxos` 追求强一致，RADOS 数据面用 `PG + primary/replica + peering + durable ack` 保证已确认写不丢，CRUSH 和故障域规则负责把高可用转化成数据放置结构，而跨站点对象网关等更高层能力则接受异步复制与最终收敛；也就是说，Ceph 的核心能力不是“绝对统一的一致性哲学”，而是按问题类型做分层取舍。**

这一篇的目标，就是把这条工程取舍主线真正讲清楚。

## 先建立第一条边界：不要问“Ceph 是不是强一致”，要先问“你在问 Ceph 的哪一层”

这是理解整篇文章的第一步，也是最重要的一步。

很多关于 Ceph 的争论，一开始就把问题问坏了：

- Ceph 是强一致吗？

这个问题本身不够精确。

更准确的问题应该是：

- Ceph 的哪一层在讨论一致性？

### 为什么

因为 Ceph 至少同时有下面几层完全不同的问题：

## 控制面一致性

- monitor 集群地图
- quorum
- Paxos 提交

## 数据面写入一致性

- primary / replica
- PG peering
- durable commit

## 高可用放置结构

- CRUSH
- failure domain
- acting/up set

## 跨站点或高层服务一致性

- RGW multisite
- CephFS 客户端缓存回收

### 这意味着什么

说明 Ceph 不是一个“全系统统一只用一种一致性模型”的系统。

它真正的风格是：

- **不同层解决不同问题，并用不同强度的一致性与可用性策略。**

这条边界一定要先立住。

## 第 1 层：为什么 MON 要追求强一致，而且几乎不能退

如果从 Ceph 全局图里找最强一致的一层，我会先指向：

- `MON`

### 为什么

因为 monitor 集群维护的是：

- monmap
- osdmap
- crush map
- mgrmap
- fsmap
- 各类集群控制状态

这些状态一旦不同 monitor 之间不一致，后面的整个系统都会失去统一参考系。

### 所以 MON 层真正不能接受什么

- 每个 monitor 都“各自觉得差不多”
- 地图版本分叉
- 没有明确多数派就继续往前推进

### 这就是为什么 Ceph 在这一层选择了

- quorum
- Paxos
- 单调 epoch

也就是说：

- **控制面地图必须先统一，后面的 OSD/客户端才有共同现实。**

这就是 MON 强一致几乎不能退的原因。

## 第 2 层：为什么 `Paxos` 在 Ceph 里解决的不是“所有一致性”，而是“集群地图一致性”

这是一个很重要的澄清。

很多人一听 Ceph 用 Paxos，就会误以为：

- 整个 Ceph 的一切一致性都靠 Paxos

其实不是。

更准确地说：

- Paxos 主要服务于 monitor 维护的那部分控制面状态

### 它解决的核心问题是什么

- 新的 map 或监控状态变更，怎样被 monitor 多数派一致接受

### 它不直接解决什么

- 某个对象写请求怎样复制到副本 OSD
- 某个 PG 怎样完成 peering
- 某个 RGW bucket 怎样跨站点同步

### 这说明什么

说明 Paxos 在 Ceph 里很重要，但它不是：

- 统治全局的一把万能钥匙

它是：

- 控制面强一致的核心机制

这条边界特别关键。

## 第 3 层：为什么数据面不直接用“MON 那一套强一致复制”来处理对象写入

继续往下，就会看到 Ceph 很重要的一次工程取舍。

既然 MON 那一层用 quorum + Paxos 很可靠，那为什么数据面不也这么做？

### 最核心的原因是

- 成本太高

对象写路径是高频、海量、持续发生的。

如果每次数据写入都走：

- monitor 多数派一致提交

那系统几乎不可能获得现在这种吞吐和扩展性。

### 所以 Ceph 在数据面选了什么

- `PG + primary/replica + peering + durable ack`

也就是：

- 不把每次对象写都升级成全局控制面共识问题

而是把它限定在：

- 一个 PG 的 acting set 内部处理

### 这说明什么

说明 Ceph 在这里做了非常明确的分层：

- 集群控制状态走 monitor 强一致
- 数据对象写入走 PG 级主副本复制一致性

这是性能和可靠性之间非常关键的一次平衡。

## 第 4 层：为什么 `PG` 是 Ceph 数据面一致性的真正边界

这一点前面几篇其实已经讲过，但在这一篇里必须重新收束一下。

Ceph 不是直接以“对象”为一致性协调单位，而是选择：

- `PG`

### 为什么

因为如果每个对象都单独维持：

- primary
- 副本集合
- 历史协商
- 恢复边界

系统状态规模会爆炸。

### 所以 PG 真正在做什么

- 给一组对象提供共同的放置、primary、副本、一致性收敛和恢复边界

### 这意味着什么

说明 Ceph 数据面的一致性不是：

- 每个对象自己谈一遍

而是：

- **对象先归属于 PG，再由 PG 作为一致性与恢复边界统一处理。**

这就是为什么 PG 是 Ceph 灵魂级概念之一。

## 第 5 层：为什么主副本复制不是“弱一致折中”，而是 Ceph 数据面的核心现实主义设计

再进一步，很多人会把 primary/replica 模式简单理解成：

- 为了性能，做了点妥协

这个说法太轻了。

更准确地说：

- 主副本复制是 Ceph 在可扩展性、恢复能力和实现复杂度之间找到的核心现实主义解

### 为什么

因为它带来了几个非常关键的结构优势：

- 写入协调点明确
- 副本组边界明确
- 历史收敛点明确
- 失败后恢复入口明确

### 这意味着什么

Ceph 不是在数据面追求：

- “人人平等同时提案”的全对等写一致性

而是明确接受：

- 一个 primary 负责协调

这样才能把写路径、peering、recovery 都做成可工程化的系统。

这不是随便的取舍，而是 Ceph 数据面真正的骨架。

## 第 6 层：为什么 `peering` 才是 Ceph 数据面“一致性恢复”的关键，而不只是故障后的小流程

很多人第一次接触 PG peering 时，会把它看成：

- OSD 重启后顺便跑一下的流程

这会严重低估它的重要性。

更准确地说：

- peering 是 Ceph 数据面判断“当前谁拥有 authoritative history、谁可以成为可写 primary、哪些副本还能参与一致性集合”的核心过程

### 它真正解决什么

- 历史收敛
- primary 合法性
- acting set 合法性
- 过去 interval 的分歧处理

### 所以最准确的说法是

- peering 不是恢复附属流程
- 它是 PG 数据面重新建立一致性现实的核心过程

这点非常关键。

## 第 7 层：为什么“已确认写不丢”是 Ceph 数据面最关键的可靠性承诺之一

继续把一致性和可靠性拉到用户最关心的问题上，可以把最重要的承诺概括成：

- **已确认写不丢**

### 为什么这句很重要

因为它比抽象地说“强一致”更贴近 Ceph 数据面的真实承诺边界。

Ceph 在 PG 主副本复制路径里，核心保证之一就是：

- 在 acting set 达到相应 durable 条件之前，不向客户端宣布成功

### 这意味着什么

说明 Ceph 数据面真正看重的是：

- 确认语义必须和可恢复的持久状态绑定

而不是：

- 只要 primary 内存里看见了就算完成

这就是为什么：

- apply
- committed
- ondisk

这些语义会在前面的写路径文章里被反复区分。

## 第 8 层：为什么 Ceph 接受“degraded 可继续服务”，但又不接受“确认语义变松”

这也是 Ceph 一致性和高可用平衡里特别有代表性的点。

很多系统在副本不完整时会选择：

- 整体不可用

Ceph 则经常允许：

- `active+degraded`
- 在某些条件下继续服务

### 这说明什么

说明 Ceph 在可用性上做了积极选择：

- 不因为副本暂时不齐就完全停摆

但它同时又非常强调：

- 已确认写的可靠性边界不能随便放松

### 所以它的取舍更准确地说是

- 尽量维持业务可继续推进
- 但不随便污染确认语义

这就是工程上的平衡感。

## 第 9 层：为什么高可用不只靠复制，还要靠 `CRUSH + failure domain`

讲到这里，就必须把“高可用”从“复制成功”再往外扩一层。

因为复制本身并不能自动保证高可用。

### 为什么

如果所有副本都落在：

- 同一台主机
- 同一个机架
- 同一电源域

那即使你有 3 副本，也可能一起死。

### 所以 Ceph 真正做了什么

- 用 `CRUSH`
- 用 `failure domain`
- 用规则把“副本不能相关故障”编码进放置算法

### 这说明什么

说明高可用在 Ceph 里不是单靠：

- “副本数 >= 3”

而是靠：

- **副本数 + 正确的故障域分布。**

这就是为什么 CRUSH 在一致性/高可用话题里绝不能被当成单纯的定位算法。

## 第 10 层：为什么 `CRUSH` 解决的是“高可用结构”，而不是“写一致性”

继续把概念拆清楚，还要再强调一个边界：

- CRUSH 非常关键

但它解决的不是：

- 写确认语义
- 历史收敛

而是：

- 副本应该放在哪
- 故障相关性怎样被避免

### 这意味着什么

说明在 Ceph 里：

- `Paxos` 解决控制面强一致
- `PG + peering + primary/replica` 解决数据面写入与恢复一致性
- `CRUSH + failure domain` 解决结构化高可用

这三类东西虽然都和“可靠”有关，但职责完全不同。

这点非常值得在脑子里分开。

## 第 11 层：为什么 stretch mode 说明 Ceph 不会把“普通副本规则”硬说成适合跨站点

再往跨站点高可用看，Ceph 的工程态度其实很务实。

它没有简单地说：

- 反正 CRUSH 已经有故障域了，跨机房也一样

而是专门引入了：

- `stretch mode`

### 为什么

因为跨站点环境会引入更尖锐的问题：

- 网络分区
- 两站点视角不一致
- 单站点残存副本是否还能安全继续确认写

### 这说明什么

说明 Ceph 非常清楚：

- 普通单集群复制规则，并不自动等于跨站点一致性规则

于是它愿意：

- 增加额外限制
- 增加额外 monitor/quorum 约束
- 增加 peering 规则

来换取更稳的跨站点行为。

这就是另一种典型的工程取舍。

## 第 12 层：为什么 RGW multisite 又明显不是同一强度的一致性模型

如果再往上看 RGW multisite，就会发现另一个很有代表性的转折。

### 为什么

因为 RGW multisite 面对的不是：

- 一个 PG acting set 内的同步复制

而是：

- 跨站点对象服务复制

这时系统明显接受了更多：

- 异步传播
- 最终收敛
- sync status 追赶

### 这意味着什么

说明 Ceph 并没有强行让对象网关跨站点复制也遵守：

- 和单集群 RADOS 数据面完全同等的强确认语义

而是明确接受：

- 这是另一类问题，应该用另一类模型处理

这正体现了 Ceph 的分层现实主义。

## 第 13 层：为什么 CephFS 客户端缓存一致性也说明 Ceph 不是“处处最强”，而是“按语义尽量强”

再看 CephFS，会发现又是另一种取舍。

CephFS 的目标不是：

- 把一切缓存都关掉换强一致

而是：

- 通过 caps、recall、session、flush 机制，在保住较强语义的同时尽量保住缓存性能

### 这说明什么

说明 CephFS 追求的也不是：

- 无条件最强、最直接、最简单的同步语义

而是：

- 在文件系统缓存命中和一致性之间做精细平衡

这再次说明 Ceph 全家桶从来不是单一教条，而是：

- 各层按语义目标分别取舍

## 第 14 层：为什么 Ceph 真正的高可用不是“永不故障”，而是“故障后仍有正确的继续方式”

谈高可用时，很多系统介绍都会给人一种错觉：

- 高可用 = 不会坏

Ceph 明显不是这种叙事。

它更真实的目标是：

- 故障一定会发生
- 但系统要知道如何继续正确地活下去

### 这体现在什么地方

- monitor 失去少数节点仍能 quorum
- PG 副本降级仍可继续服务
- peering 后重新找到 authoritative history
- recovery/backfill 把系统逐渐拉回健康状态
- CRUSH 让重建目标仍然遵守故障域

### 所以 Ceph 的高可用观更准确地说是

- 不是“没有故障”
- 而是“故障发生后仍能按规则收敛到正确状态”

这点特别关键。

## 第 15 层：为什么复杂度在 Ceph 里不是副作用，而是被主动拿来交换可靠性和扩展性的成本

讲到这里，就必须正面承认一件事：

- Ceph 很复杂

但这种复杂度并不只是：

- 历史包袱

更大程度上，它来自很多主动的工程交换：

- 用 PG 降低对象级状态爆炸，但引入 peering/recovery 复杂度
- 用 primary/replica 获得更可控的数据面一致性，但引入主副本协商复杂度
- 用 CRUSH 去中心化放置，但引入规则和拓扑设计复杂度
- 用 MON quorum 保证地图强一致，但引入 Paxos 和 epoch 管理复杂度
- 用 multisite 异步复制换跨站点可用性，但接受收敛与同步状态复杂度

### 这说明什么

说明 Ceph 的复杂度很多时候不是偶然，而是：

- 为了扩展性、可靠性和去中心化能力付出的显式成本

这点必须正面承认。

## 第 16 层：一张最实用的“层次 -> 一致性/高可用策略”图

如果这一篇只记一张图，我建议记下面这张：

```text
控制面:
  MON + quorum + Paxos
  -> 强一致的集群地图

单集群数据面:
  PG + primary/replica + peering + durable ack
  -> 已确认写不丢 + 故障后可恢复收敛

放置与结构高可用:
  CRUSH + failure domain
  -> 副本按故障域分布，避免相关性故障

跨站点对象服务:
  RGW multisite + metadata/data sync
  -> 异步复制 + 最终收敛

文件缓存一致性:
  CephFS caps/session/flush
  -> 在缓存性能与语义之间精细平衡
```

这张图里最关键的一句话是：

- **Ceph 的一致性和高可用不是一种机制解决全部问题，而是按层分工。**

## 第 17 层：为什么“强一致 vs 最终一致”这个二元对立并不足以描述 Ceph

这是整篇最想纠正的误区。

很多人习惯把分布式系统压成一个二选一问题：

- 强一致
- 最终一致

但 Ceph 明显不是这样。

### 为什么

因为在 Ceph 里你会同时看到：

- monitor 地图强一致
- 单集群对象写确认的严格 durability 边界
- 跨站点 RGW 的异步收敛
- CephFS 缓存一致性的协议化协调

### 所以更准确的说法应该是

- Ceph 是一个“多层一致性策略并存”的系统

它真正厉害的地方，不是统一把所有问题压成一种答案，而是：

- 为不同问题选不同强度的正确答案

这就是它的工程价值。

## 第 18 层：把整篇压缩成一句最核心的话

如果这一篇只想留下一句最核心的话，我会写成这样：

- **Ceph 的强大，不在于它到处都追求同样强的一致性，而在于它知道哪些状态必须强一致，哪些数据写必须把确认和持久化绑定，哪些高可用必须落实到故障域结构，哪些跨站点能力则可以接受异步复制与最终收敛。**

这句话其实就是整篇的核心。

## 用一句话重新概括这篇

如果把这篇全部内容压缩成一句尽量准确的话，我会这样说：

**Ceph 并不是用一种统一模型解决所有一致性与高可用问题，而是按层分工：控制面通过 `MON + quorum + Paxos` 保证集群地图强一致，单集群数据面通过 `PG + primary/replica + peering + durable ack` 保证已确认写不丢并在故障后重新收敛，`CRUSH + failure domain` 把高可用落成结构化放置规则，而像 `RGW multisite` 这样的跨站点能力则接受异步复制与最终收敛；因此 Ceph 的本质不是简单“强一致”或“最终一致”，而是一套面向不同问题分层取舍的工程体系。**

## 把整篇压缩成一条最短骨架

如果你只想记一条骨架，可以记成下面这样：

```text
先区分控制面、单集群数据面、放置结构和跨站点服务
  ->
控制面用 MON/Paxos 追求强一致
  ->
数据面用 PG/主副本/peering 保证已确认写不丢
  ->
CRUSH/failure domain 保证结构化高可用
  ->
跨站点对象服务接受异步收敛
  ->
Ceph 的核心是按层取舍，而不是一刀切
```

只要这条骨架记住了，Ceph 的一致性与高可用主线基本就不会再混。

## 初学者最容易混淆的 10 个点

### 1. 认为 Ceph 全系统只有一种一致性模型

不对。不同层明显用了不同策略。

### 2. 认为 Paxos 解决了 Ceph 的所有一致性问题

不对。它主要解决 monitor 控制面状态一致性。

### 3. 认为数据面也应该走 MON 那一套多数派提交

不对。那样成本会高到不可接受。

### 4. 认为 PG 只是定位中间层

不对。它还是数据面一致性与恢复边界。

### 5. 认为主副本复制只是“性能妥协”

不对。它是数据面工程化可行性的核心结构。

### 6. 认为 peering 只是故障后的小流程

不对。它是重新建立数据面一致性现实的核心过程。

### 7. 认为高可用只等于副本数够多

不对。还必须看故障域是否隔离正确。

### 8. 认为 CRUSH 解决了写一致性

不对。它解决的是结构化高可用与放置规则。

### 9. 认为 RGW multisite 的一致性和单集群 RADOS 写确认是一回事

不对。它明显接受异步收敛。

### 10. 认为 Ceph 复杂度只是历史包袱

不对。很多复杂度是为可靠性和扩展性主动付出的成本。

## 这一篇最应该留下的 5 个直觉

### 直觉一：问一致性前，先问 Ceph 的哪一层

这是第一原则。

### 直觉二：控制面和数据面用的是不同一致性机制

这点必须立住。

### 直觉三：已确认写不丢是 Ceph 数据面最关键的可靠性边界之一

比抽象口号更重要。

### 直觉四：高可用必须落实到 CRUSH 和 failure domain 结构

不只是副本数问题。

### 直觉五：Ceph 的核心价值是按层取舍，而不是追求单一教条

这是整篇最重要的认识。
