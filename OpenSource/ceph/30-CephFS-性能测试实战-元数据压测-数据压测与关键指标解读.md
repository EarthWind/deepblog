# CephFS 性能测试实战：元数据压测、数据压测与关键指标解读

## 这篇文章要解决什么问题

前面几篇我们已经把 CephFS 的架构、MDS、客户端、FUSE、内核客户端都讲得比较完整了。

但一旦进入真正的使用阶段，读者几乎都会碰到同一个现实问题：

**CephFS 到底该怎么测？**

这件事看起来像：

- 跑个 `fio`
- 跑个 `mdtest`
- 看一眼吞吐和时延

好像就结束了。

但只要你真的做过几次 CephFS 压测，很快就会发现结果特别容易混：

- 为什么 `mdtest` 很快，但业务还是慢？
- 为什么顺序吞吐很好，小文件创建却糟糕？
- 为什么同一个集群换成 kernel client 和 `ceph-fuse` 差很多？
- 为什么第一次跑很快，第二次更快，第三次又不一样？
- 为什么有时测出来的是客户端缓存命中，不是 CephFS 本体能力？
- 为什么 MDS 指标看起来不高，但后端 OSD 已经被打满？

如果只允许用一句话先给结论，那就是：

**CephFS 性能测试首先不是“挑一个工具跑一下”，而是先分清你到底在测元数据面、数据面、客户端缓存效果，还是整个端到端路径；只有把 `mdtest/smallfile`、`fio`、kernel client / FUSE、`cephfs-top`、MDS 指标和后端 OSD 指标放到同一个框架里，测试结果才真正有解释力。**

这一篇的目标，就是把这个框架真正讲清楚。

## 先建立第一条边界：CephFS 不是只有“一种性能”

这是理解整篇文章的第一步。

很多人说：

- CephFS 性能怎么样？

这个问题其实不完整。

因为 CephFS 至少同时有三层完全不同、又经常互相影响的性能问题：

## 元数据性能

- 目录遍历
- lookup
- create
- unlink
- rename
- stat

## 数据性能

- 顺序读写吞吐
- 随机读写
- 页缓存命中后的本地体验
- 回写和 fsync 行为

## 缓存与一致性相关性能

- cap 命中
- dentry lease 命中
- page cache 命中
- recall / trim / flush 对时延的影响

### 为什么一定要先分这三层

因为你如果只看一个总吞吐，很可能根本不知道自己测到了什么。

CephFS 不是块设备，不是对象网关，也不是传统单机文件系统，它是：

- 元数据面 + 数据面 + 分布式客户端缓存

的组合体。

所以性能测试也必须按这三层拆开。

## 第 1 层：为什么 `mdtest`、`smallfile`、`fio` 不能互相替代

这是最容易混淆的一点。

很多人会问：

- CephFS 压测是不是直接 `fio` 就行？

不行，至少不完整。

### `mdtest`

- 更偏元数据高压测试

它更适合回答：

- create / stat / readdir / unlink 这类元数据操作能扛多大压力

### `smallfile`

- 更偏海量小文件 workload

它更适合回答：

- 小文件创建、读写、删除、目录扩散这类真实文件系统 workload 表现如何

### `fio`

- 更偏数据面测试

它更适合回答：

- 顺序吞吐
- 大块随机读写
- page cache / writeback 相关数据路径表现如何

### 所以最重要的结论是什么

- **`mdtest`、`smallfile`、`fio` 测的是三个不同侧面，它们不能互相替代。**

这条边界一定要先立住。

## 第 2 层：为什么 CephFS 性能测试应该先问“我要测元数据还是数据”

这是最实用的第一步。

在真的写任何命令前，先问自己：

- 这次测试主要在看什么？

## 如果你关心的是元数据面

优先应该选：

- `mdtest`
- `smallfile`

### 你在观察什么

- create / mkdir / unlink / rename 速率
- 目录层级操作的时延
- MDS reply latency
- session/caps 相关行为

## 如果你关心的是数据面

优先应该选：

- `fio`

### 你在观察什么

- 顺序吞吐
- 随机读写
- 平均时延与尾时延
- page cache 命中与 writeback 行为

### 为什么这个区分必须先做

因为 CephFS 的数据面和元数据面本来就由不同组件主导：

- 元数据更偏 `MDS`
- 数据更偏客户端 + OSD

如果你不先区分，后面所有结果都很容易解释错。

## 第 3 层：为什么 `mdtest` 是理解 CephFS 元数据性能的首选工具

如果只谈元数据压测，仓库里最明确、最值得先讲的工具就是：

- `mdtest`

### 为什么

因为它天然适合制造：

- 大量 create
- stat
- unlink
- 目录操作

这类纯元数据压力。

而 CephFS 的元数据能力，恰恰最需要靠这种 workload 来揭示。

### 它特别适合回答什么问题

- 单 MDS 下元数据并发能力如何
- 多客户端同时创建小文件时 MDS 能扛多少
- 多 MDS 是否真的带来收益
- 大量目录项操作是否出现明显热点

### 所以 `mdtest` 的定位最准确应该是

- CephFS 元数据面基准工具

## 第 4 层：为什么 `smallfile` 比很多人想象得更接近真实业务

如果说 `mdtest` 更偏“元数据压力机”，那：

- `smallfile`

就更像：

- 小文件业务场景模拟器

### 为什么它很有价值

因为很多真实 CephFS 业务并不是：

- 连续大文件吞吐

而是：

- 海量小文件
- 多目录并发
- 创建后马上读写
- 删除和重建交织

### 所以它最适合回答的问题是

- 真实文件业务在小文件场景下的体感会怎样
- 目录扩散是否合理
- MDS 会不会很快出现热点

### 这说明什么

说明如果你的目标 workload 本来就偏：

- 小文件 / 目录风暴 / metadata-heavy

那只跑 `fio` 基本没有说服力。

## 第 5 层：为什么 `fio` 在 CephFS 里更像“数据路径测试器”

讲完元数据面，再看：

- `fio`

它当然可以跑在 CephFS 挂载点上，但它最擅长回答的问题，仍然主要是：

- 数据 IO 路径

### 比如它很适合测

- 大文件顺序读写吞吐
- 随机读写
- 不同块大小下的时延
- 页缓存与直刷边界对结果的影响

### 但为什么它不能代表 CephFS 全部性能

因为它对下面这些问题通常不够敏感：

- 目录创建风暴
- rename / unlink / readdir 压力
- 海量文件 stat
- dentry / caps 相关元数据压力

### 所以 `fio` 的正确定位应该是

- CephFS 数据面测试主力

而不是：

- CephFS 全能基准工具

## 第 6 层：为什么 kernel client 和 `ceph-fuse` 一定要分开测

这是 CephFS 测试里特别容易被忽视的一点。

很多人会想：

- 反正都挂成文件系统了
- 跑同样 workload，结果应该差不多吧

这个想法非常危险。

### 为什么

因为：

- kernel client 直接接 Linux VFS / page cache / writeback
- `ceph-fuse` 走的是用户态 FUSE 前端 + Ceph 客户端

两条路径在：

- 页缓存参与方式
- 上下文切换
- 请求进入客户端语义层的成本
- metadata cache 行为

上都可能表现不同。

### 所以测试时必须先问

- 生产里到底走哪条客户端路径

### 更准确地说

- 不同客户端路径的测试结果不能直接混在一起比较

这条边界非常重要。

## 第 7 层：为什么“第二次跑更快”通常意味着你测到了缓存

这是性能测试里最大的坑之一。

在 CephFS 上，同一个 workload 第二次更快，往往不是什么神奇优化，而是：

- dentry lease 命中
- cap 命中
- page cache 命中
- 本地 inode/dentry 缓存还在

### 这意味着什么

意味着你必须先想清楚：

- 你是要测 cold path
- 还是 warm cache path

### 如果不区分，会怎样

你就会得到一种很常见但没法复现的结果：

- 第一次 2000 ops/s
- 第二次 8000 ops/s
- 然后完全不知道自己到底优化了什么

所以 CephFS 测试最重要的一条方法论是：

- **必须显式区分冷缓存测试和热缓存测试。**

## 第 8 层：为什么 `cephfs-top` 是 CephFS 性能测试里最值得先开起来的观测工具

如果只推荐一个 CephFS 专属观测入口，我会优先推荐：

- `cephfs-top`

### 为什么

因为它非常适合把：

- 客户端维度
- 吞吐
- 时延
- 缓存相关指标

这些信息放在同一个视角里看。

### 最有价值的指标包括

- `chit`
- `dlease`
- `rtio`
- `wtio`
- `rsp`
- `wsp`
- `rlatavg`
- `wlatavg`
- `mlatavg`

### 这说明什么

说明 CephFS 性能测试不是只靠压测工具输出结果，还要靠 CephFS 自己的观测面回答：

- 客户端到底命中了多少缓存
- 元数据时延是不是开始抬头
- 当前更像 metadata bound 还是 data bound

这点特别重要。

## 第 9 层：为什么客户端指标和 MDS 指标一定要放在一起看

这是 CephFS 测试里最容易少看一半的问题。

只看客户端输出，你可能知道：

- 吞吐变慢了
- 时延变高了

但不知道根因在哪。

### 如果同时看客户端指标

你会知道：

- `cap_hits/cap_miss`
- `dentry_lease_hits/miss`
- `avg_metadata_latency`
- `avg_read_latency`
- `avg_write_latency`

### 如果同时看 MDS 指标

你会知道：

- reply latency
- session 数
- cache 规模
- caps recall 压力
- 慢请求与慢 metadata IO

### 所以最重要的结论是什么

- **CephFS 压测结果不能只看 workload 输出，必须至少配套看客户端指标和 MDS 指标。**

否则很多现象你只能看到“慢了”，却解释不了“为什么慢”。

## 第 10 层：为什么有时问题根本不在 MDS，而在后端 OSD

这一点特别容易误判。

很多人一看到 CephFS 慢，就会先怀疑：

- MDS
- caps
- 多 MDS 配置

这些当然可能有关，但并不总是根因。

如果你跑的是：

- 大文件顺序读写
- 大块 `fio`

很多时候真正先撞墙的，是：

- OSD
- 网络
- BlueStore
- 底层盘

### 这意味着什么

CephFS 测试不能只看文件系统层，还必须联动看：

- OSD 吞吐
- objecter 指标
- 网络带宽
- 底层盘能力

### 所以一个非常实用的测试原则是

- **CephFS 的数据面问题，经常不是 MDS 问题，而是后端 RADOS 问题。**

## 第 11 层：为什么 workload 是否超过 `mds_cache_memory_limit` 非常关键

这一点在元数据压测里尤其重要。

如果你的工作集远小于：

- `mds_cache_memory_limit`

会发生什么？

- 大量元数据都能稳定留在 MDS cache 里
- 命中率会非常高
- 看起来一切都很快

### 这说明什么

说明你测到的更像是：

- “缓存很舒服时的表现”

而不是：

- “真实工作集压力下的表现”

所以如果要做更有代表性的元数据测试，必须问：

- 工作集规模是否真的逼近或超过 MDS cache 预算

这点非常值钱。

## 第 12 层：为什么大目录和单热点目录会严重扭曲测试结论

CephFS 性能测试里另一个常见坑是：

- 测试设计不小心把 workload 集中到极少数目录

### 这会带来什么

- 热目录
- dirfrag 压力
- subtree 偏斜
- 单 MDS 热点

最后你测到的就不再是“CephFS 整体能力”，而是：

- 某个目录布局设计很差时的局部热点

### 这并不是说这种测试没价值

它当然有价值，但你必须明确：

- 这是热点场景测试

而不是：

- 一般性能力测试

所以 CephFS 基准设计里，目录分布是一个必须显式声明的维度。

## 第 13 层：为什么多 MDS 测试不能简单地“开了就更快”

这是又一个非常容易误导的点。

很多人一听多 MDS，直觉就会觉得：

- 元数据性能一定更高

其实不一定。

### 为什么

因为多 MDS 更适合的是：

- 多客户端并发
- 多目录并发
- 可分裂的元数据热点

而不是：

- 单客户端、单热点目录、单串行 workload

### 所以测试多 MDS 时必须先问

- 当前 workload 是否真的能被 subtree 拆分和并行化

否则就会出现很常见的误解：

- 开了多 MDS，为什么没提升？

其实不是多 MDS 没用，而是 workload 根本没给它发挥空间。

## 第 14 层：为什么 CephFS 测试一定要显式说明错误语义和 flush 边界

这点在数据测试里很容易被忽略。

如果你跑 `fio` 或应用写入时只盯：

- `write()` 成功

却不看：

- flush
- sync
- fsync

那你就很可能误解结果。

### 为什么

因为 CephFS 可能存在：

- page cache
- writeback
- delayed error reporting

所以：

- 吞吐很好

并不自动等于：

- durable 语义也一样好

### 这意味着什么

在写路径测试里，至少应该明确：

- 是否关注 writeback 吞吐
- 是否关注 sync/fsync 时延
- 是否要观察 ENOSPC/断连/回写失败的暴露时点

这对测试结论非常关键。

## 第 15 层：一套实用的 CephFS 测试顺序应该长什么样

讲到这里，可以把最实用的方法压缩成一套顺序。

## 第一步：先定义目标

比如：

- 测元数据
- 测数据吞吐
- 测小文件场景
- 测热缓存体验

## 第二步：选对工具

比如：

- 元数据优先 `mdtest`
- 小文件优先 `smallfile`
- 数据面优先 `fio`

## 第三步：明确客户端路径

比如：

- kernel client
- `ceph-fuse`

## 第四步：区分冷缓存和热缓存

比如：

- 是否清理页缓存
- 是否复用同一目录树
- 是否重复运行同一 workload

## 第五步：同步采集指标

至少包括：

- workload 输出
- `cephfs-top`
- client metrics
- MDS metrics
- 必要时 OSD/网络指标

## 第六步：按层解释结果

比如：

- 元数据慢，先看 MDS 与 caps
- 数据慢，先看 OSD 与网络
- 第二次更快，先怀疑缓存

这套顺序不花哨，但非常稳。

## 第 16 层：一张最实用的“工具与问题匹配表”

如果这一篇只想记一张最实用的表，可以记成下面这样：

```text
元数据性能:
  mdtest

小文件真实 workload:
  smallfile

大文件数据吞吐/随机 IO:
  fio

客户端即时观测:
  cephfs-top

客户端底层指标:
  ceph fs perf stats / client metrics

MDS 压力与慢请求:
  MDS metrics / health messages / dump_ops_in_flight

后端瓶颈:
  OSD / objecter / 网络 / 磁盘指标
```

这张表的价值不是背工具，而是提醒你：

- 工具一定要服务于问题类型

## 第 17 层：把 CephFS 性能测试真正压缩成一句方法论

这是整篇最想让读者留下的认知。

CephFS 不是一个“单面向”的存储系统。

它同时有：

- MDS 元数据面
- 客户端缓存面
- OSD 数据面

所以性能测试也必须按这个现实来组织。

换句话说：

- 真正靠谱的 CephFS 压测，不是跑一个 benchmark
- 而是先把 workload、客户端路径、缓存状态、指标口径和后端观测同时立起来

这就是为什么这篇文章必须把工具、指标和方法论放在同一个框架里讲。

## 用一句话重新概括这篇

如果把这篇全部内容压缩成一句尽量准确的话，我会这样说：

**CephFS 性能测试的核心，不是挑 `mdtest`、`smallfile` 或 `fio` 里的某一个工具跑一下，而是先明确在测元数据面、数据面还是缓存效果，再区分 kernel client / `ceph-fuse` 路径、冷缓存 / 热缓存状态，并把 workload 输出与 `cephfs-top`、客户端指标、MDS 指标和后端 OSD 指标联动起来解释；只有这样，测试结果才真正有可比性和诊断价值。**

## 把整篇压缩成一条最短骨架

如果你只想记一条骨架，可以记成下面这样：

```text
先分清测元数据还是测数据
  ->
选 mdtest / smallfile / fio
  ->
明确 kernel client 还是 ceph-fuse
  ->
区分冷缓存和热缓存
  ->
同步看 cephfs-top、client metrics、MDS metrics、OSD 指标
  ->
再解释吞吐、时延、命中率和热点问题
```

只要这条骨架记住了，CephFS 性能测试通常就不会跑偏得太厉害。

## 初学者最容易混淆的 10 个点

### 1. 认为 CephFS 只有一种“总性能”

不对。至少要分元数据、数据和缓存效果。

### 2. 认为 `fio` 可以代表 CephFS 全部能力

不对。它更多代表数据面。

### 3. 认为 `mdtest` 和 `smallfile` 是一回事

不对。一个更偏元数据压力，一个更偏小文件业务 workload。

### 4. 认为 kernel client 和 `ceph-fuse` 测出来可以直接横向比较

不对。路径不同，结果天然可能不同。

### 5. 认为第二次更快就是系统优化成功

不对。大概率是缓存命中更多了。

### 6. 认为 CephFS 慢一定先怪 MDS

不对。数据面慢时经常先撞 OSD、网络或磁盘。

### 7. 认为工作集不重要

不对。是否超过 `mds_cache_memory_limit` 会显著改变测试结果。

### 8. 认为大目录热点结果可以代表整体能力

不对。那往往只是热点场景测试。

### 9. 认为开多 MDS 一定更快

不对。要看 workload 是否能真正拆分。

### 10. 认为写成功就代表 durable 没问题

不对。flush/fsync 边界同样关键。

## 这一篇最应该留下的 5 个直觉

### 直觉一：CephFS 性能测试必须分层

这是第一原则。

### 直觉二：工具必须服务于问题类型

`mdtest`、`smallfile`、`fio` 各有边界。

### 直觉三：缓存状态必须显式说明

否则结果几乎不可比较。

### 直觉四：客户端指标、MDS 指标和后端 OSD 指标必须联动看

不能只看 benchmark 输出。

### 直觉五：CephFS 压测的价值不只是给数字，而是帮助定位瓶颈层

这点最重要。
