# 02. 控制面与数据面：DAOS 为什么是双平面架构

## 这一篇要解决什么问题

理解 DAOS 的第一道真正门槛，往往不是对象模型，也不是 VOS，而是一个更基础的问题：

**为什么 DAOS 不做成一个统一的大进程，而要拆成控制面和数据面？**

从仓库结构上看，这种分层非常明显：

- `src/control/` 是一套以 Go 为主的控制面代码。
- `src/engine/` 是一套以 C 为主的数据面代码。
- `daos_server` 和 `daos_engine` 不是同一个二进制，而是两个职责不同、协作紧密的进程。

如果这一层没有理解清楚，后面读 `daos_server` 启动链路、dRPC 通信、模块装载、pool/object 服务时，就很容易把不同层次的职责混在一起。

所以这一篇聚焦回答四个问题：

1. 控制面和数据面分别负责什么？
2. DAOS 为什么要把它们拆成两个平面？
3. 为什么控制面主要用 Go，而数据面主要用 C？
4. 这种双平面设计到底给性能、扩展性和可运维性带来了什么好处？

## 先给答案：双平面不是“实现习惯”，而是架构选择

`src/control/README.md` 开篇就把这个事情说得很直接：DAOS 在两个紧密集成的平面上运行，数据面负责重型传输和 I/O 工作，控制面负责进程与存储管理，并协助数据面运行。

这句话的含义非常关键。它表明：

- DAOS 并不是先写出一个存储引擎，后来再补一个管理进程。
- 控制面和数据面从一开始就是不同的角色。
- 双平面设计的目标，是让“管理控制”和“高性能数据路径”分别沿着最适合自己的方向优化。

换句话说，DAOS 并不是简单地把一个服务拆成两个名字，而是在系统边界上做了一次明确切分：

- 哪些事情属于“编排、配置、生命周期和外部管理”？
- 哪些事情属于“真正的请求执行、网络数据通路和本地存储访问”？

这就是双平面架构的核心。

## 控制面到底在做什么

### 控制面的进程形态

在 DAOS 里，控制面对应的核心二进制是 `daos_server`，主要代码在 `src/control/`，使用 Go 实现。

`src/README.md` 和 `src/control/README.md` 对它的职责描述可以总结成四类：

- 解析配置文件。
- 进行网络和存储硬件的准备、分配和管理。
- 启动并监控本机上的一个或多个 `daos_engine` 进程。
- 对外暴露管理接口，承接管理命令和编排动作。

这意味着控制面本质上是一个“节点级编排器 + 管理服务入口”，而不是直接承载高性能 I/O 的执行核心。

### 从 `main.go` 看控制面的真实入口

`src/control/cmd/daos_server/main.go` 非常适合作为控制面第一入口，因为它把控制面的工作风格暴露得很清楚。

从 `main()` 和 `parseOpts()` 可以看到几件事：

- 它先构造命令行和日志环境。
- 它在执行前检查特权 helper 是否可用，调用 `pbin.CheckHelper()`。
- 它支持配置加载、网络初始化、SCM 初始化、NVMe 初始化这些前置流程。
- 它最终把子命令分发到不同的控制面功能路径。

这说明 `daos_server` 的关注点明显偏向：

- 配置与环境准备。
- 前置检查。
- 设备和网络初始化。
- 管理命令入口。
- 运行时编排。

这些事情都不是“把对象请求写入介质”本身，但又是 DAOS 系统能够正常运行的前提。

### 控制面为什么需要特权能力

从 `main.go` 里的 `pbin.CheckHelper()`，再结合 `src/control/README.md` 对 `pbin` 包的描述，可以看出控制面有一个非常现实的工程目标：

它需要代表 `daos_server` 去完成一些必须由特权 helper 处理的操作。

这背后反映的是控制面工作的真实语境：

- 它要接触本地硬件和系统资源。
- 它要配置网络和存储环境。
- 它要管理本机上数据面进程的生命周期。

因此控制面不是“一个纯业务 API 层”，而是明显带有系统管理性质的节点守护进程。

## 数据面到底在做什么

### 数据面的进程形态

数据面对应的核心二进制是 `daos_engine`，主要代码在 `src/engine/`，使用 C 实现。

`src/README.md` 对它的定义很明确：它是一个多线程进程，负责运行 DAOS 存储引擎，处理元数据和 I/O 请求，通过 CART 通信，并通过 PMDK、SPDK 访问本地 SCM 和 NVMe。

如果把控制面看成“组织者”，那数据面就是“真正干活的人”。

它直接承担：

- 客户端与服务端请求处理。
- 服务模块的加载与执行。
- 高性能网络通信。
- 本地持久化访问。
- 多线程并发调度。

### 从 `src/engine/README.md` 看数据面的内部组织

`src/engine/README.md` 透露了数据面几个特别重要的实现特征。

第一，它不是单线程服务，而是明确采用多线程模型，并且使用 Argobots 做非阻塞处理。

第二，它不是一个把所有逻辑硬编码在一起的大程序，而是通过模块接口装载服务端代码。每个模块可以定义：

- 模块名和模块 ID
- 初始化与清理函数
- CART RPC handlers
- dRPC handlers

第三，它维护自己的线程本地存储 TLS，每个 xstream 都可以为不同模块分配私有数据。

这些信息组合起来，说明数据面的设计目标非常明确：

- 面向高并发。
- 面向模块化扩展。
- 面向低开销执行路径。
- 面向存储和网络资源的细粒度控制。

### 从 `init.c` 看数据面为什么像“模块化引擎”

`src/engine/init.c` 里最值得注意的，不只是各种初始化细节，而是那条默认模块列表：

- `vos`
- `rdb`
- `rsvc`
- `security`
- `mgmt`
- `dtx`
- `pool`
- `cont`
- `obj`
- `rebuild`

同时还能看到 `modules_load()` 通过 `dss_module_load()` 去逐个装载模块。

这说明 `daos_engine` 的本质不是“一个单体对象服务进程”，而是一个承载多种服务模块的运行时容器。也正因为如此，后续你看到 `pool`、`container`、`object`、`rebuild` 等目录时，应该把它们理解成被数据面装载和运行的服务能力，而不是若干彼此独立的小程序。

## 控制面和数据面之间到底怎么分工

理解双平面设计，最好的方式不是只记定义，而是把事情按类别拆开。

### 控制面负责什么

可以把控制面的职责概括成“把节点和系统准备好，并让数据面处于可运行状态”。

更具体一点，控制面主要负责：

- 配置解析与参数注入。
- 节点级网络和存储资源准备。
- 本机 `daos_engine` 实例的拉起、监控和管理。
- 对外管理 API 和管理命令入口。
- 节点与系统成员关系相关的管理逻辑。
- 与特权 helper、系统环境和运维流程的衔接。

### 数据面负责什么

可以把数据面的职责概括成“真正执行存储系统的核心工作负载”。

更具体一点，数据面主要负责：

- 处理客户端和服务器间的实际请求。
- 运行业务服务模块，如 `pool`、`cont`、`obj`、`rebuild`。
- 维护本地执行线程、TLS、调度和模块上下文。
- 通过 CART 承担高性能数据通路。
- 访问本地 SCM、NVMe，以及更底层的存储子系统。

### 一句话区分两者

如果要用一句话区分：

- 控制面更像“会管理节点和系统的控制中心”。
- 数据面更像“真正处理存储请求和数据路径的执行引擎”。

## 为什么不能只做一个平面

这是整篇文章最重要的问题。

从 DAOS 的实现和需求出发，至少有五个理由决定了它更适合双平面，而不是单平面。

### 1. 管理流量和数据流量的性质根本不同

管理操作通常包括：

- 配置加载
- 成员管理
- 启停控制
- 设备准备
- 状态查询

这些请求的特点是：

- 不一定高频
- 更关心可观测性和安全性
- 更适合走成熟的管理接口，例如 gRPC

而数据请求的特点完全不同：

- 高并发
- 低延迟
- 对线程模型和网络开销非常敏感
- 更适合使用贴近 HPC 场景的高性能通信栈

如果强行用一个统一平面同时处理这两类工作，就很容易让管理逻辑污染数据路径，也容易让高性能执行路径被不必要的控制逻辑拖累。

### 2. 生命周期管理天然需要“管理者”和“被管理者”

`src/README.md` 已经明确写出：控制面负责启动和监控一个或多个数据面实例。

这意味着从进程关系上看，控制面天然扮演 orchestration 的角色，而数据面天然扮演 workload execution 的角色。

如果把两者塞进一个进程里，会出现两个问题：

- 谁来负责本进程外部的设备准备、配置校验和生命周期控制？
- 当高性能数据执行路径出现问题时，管理能力如何保持相对独立？

拆成双平面后，这种角色分工就清楚了：

- 控制面负责把数据面拉起来。
- 数据面专注于把请求处理好。

### 3. 两条路径对语言和运行时的偏好不同

控制面主要使用 Go，数据面主要使用 C，这不是历史偶然，而是职责驱动的结果。

控制面更偏向：

- CLI 和服务端框架
- gRPC
- 配置和编排
- 系统管理逻辑
- 工程可维护性

这些事情 Go 做起来非常顺手。

而数据面更偏向：

- 高性能网络库
- 存储介质访问
- 内存和线程的细粒度控制
- 模块装载和低开销执行路径

这些事情 C 更适合。

如果强行统一到一种语言和一种运行时，通常意味着至少有一侧要做妥协。

### 4. 双平面更容易控制故障域

控制面和数据面拆开后，至少在逻辑上可以把两类问题分层定位：

- 管理链路问题，多半先看控制面。
- I/O、模块执行、线程模型和存储路径问题，多半先看数据面。

这对排障非常重要。因为 DAOS 不是简单应用，而是要运行在真实集群节点上的存储系统。出问题时，能够先快速判断故障更接近“编排管理”还是“数据执行”，价值非常大。

### 5. 双平面让扩展边界更清晰

一旦把控制面和数据面边界划清楚，很多设计都会更自然：

- 管理 API 可以围绕控制面演化。
- 数据服务模块可以围绕 `daos_engine` 的模块接口持续扩展。
- 本地进程间协作可以通过 dRPC 统一。
- 高性能远程通信可以通过 CART 统一。

从长期演进角度看，这种边界感会让系统更容易增长，而不是越长越缠。

## 为什么控制面用 Go，数据面用 C

这是很多第一次看 DAOS 仓库的人都会问的问题。

答案其实不复杂：**因为两边解决的问题本来就不一样。**

### Go 更适合控制面

控制面是典型的工程系统代码：

- 需要处理命令行和配置。
- 需要组织 gRPC 服务和管理接口。
- 需要做节点级资源编排。
- 需要较强的工程可维护性和开发效率。

这些方面，Go 的包管理、并发模型、服务开发体验和代码组织方式都比较合适。

### C 更适合数据面

数据面则是典型的性能敏感系统代码：

- 需要贴近高性能网络与存储库。
- 需要精细管理线程、CPU 亲和性和本地上下文。
- 需要在关键路径上尽量减少额外抽象成本。
- 需要与已有 HPC 生态和底层库直接协作。

这使得 C 成为更自然的选择。

因此，Go 与 C 的并存不是“语言风格混乱”，而是 DAOS 对不同职责采用了不同技术栈。

## 双平面之间怎么通信

双平面不是彼此隔离，而是明确分工后再协作。

`src/README.md` 和 `src/engine/README.md` 都指出，控制面与数据面通过 Unix Domain Socket 上的 dRPC 通信。

这个选择非常合理：

- 它们通常运行在同一台主机上。
- 本地进程间通信不需要走远程高性能 fabric。
- dRPC 足够轻量，适合承担控制面与数据面之间的本地协作。

而在更大的系统边界上：

- 管理通路主要走 gRPC。
- 数据通路主要走 CART。
- 控制面与数据面的本机协作走 dRPC。

这种三层分工非常有 DAOS 风格：不同边界用不同通信手段，而不是试图让一种协议包打天下。

## 双平面架构带来的直接收益

如果只看概念，双平面像是在“拆系统”；但如果从结果看，它其实是在“保护系统”。

### 对性能的保护

把高性能数据路径放进 `daos_engine`，意味着：

- I/O 关键路径不必被控制逻辑打断。
- 执行模型可以围绕 Argobots、xstream、TLS 和模块装载专门优化。
- 更容易把高并发请求压到适合的数据执行环境中。

### 对可运维性的保护

把编排、配置、设备初始化和管理 API 放进 `daos_server`，意味着：

- 管理接口更清晰。
- 系统集成与部署更容易。
- 节点管理逻辑不会和 I/O 执行细节缠在一起。

### 对可扩展性的保护

控制面和数据面边界清楚后，后续新增能力时更容易判断该放哪一层：

- 管理类、系统类能力优先考虑控制面。
- 请求执行类、服务模块类能力优先考虑数据面。

这比“所有能力都堆进一个进程”要健康得多。

## 读源码时应该如何利用这个认知

一旦理解了双平面架构，后面的源码阅读顺序就会清楚很多。

### 如果你想看控制面主线

优先读：

- `src/control/README.md`
- `src/control/cmd/daos_server/main.go`
- `src/control/server/README.md`

目标是先看明白：

- `daos_server` 如何启动
- 如何加载配置
- 如何检查 helper
- 如何准备网络和存储环境

### 如果你想看数据面主线

优先读：

- `src/engine/README.md`
- `src/engine/init.c`
- `src/engine/module.c`

目标是先看明白：

- `daos_engine` 的线程模型
- 模块是如何注册和装载的
- dRPC handler、RPC handler、TLS key 如何进入引擎

### 如果你想看两者如何衔接

优先抓住这条关系：

- 控制面负责“起、管、配”
- 数据面负责“跑、收、写”
- 本机协作走 dRPC

只要先记住这三点，后面很多目录和函数名都会更容易归位。

## 小结

DAOS 的双平面架构，表面上看只是 `daos_server` 和 `daos_engine` 的拆分，实际上代表了一种非常明确的系统设计哲学：

- 管理控制和高性能数据路径不是一回事。
- 它们应该由不同的进程、不同的语言、不同的运行时模型分别承载。
- 它们通过清晰的本地与远程通信边界协作，而不是混成一个“大而全”的服务。

所以，DAOS 为什么是双平面架构？

答案可以归纳成一句话：

**因为它既要做一个可管理、可编排、可部署的分布式存储系统，又要做一个面向高性能 I/O 的低开销数据执行引擎，而这两种目标最适合分层实现。**
