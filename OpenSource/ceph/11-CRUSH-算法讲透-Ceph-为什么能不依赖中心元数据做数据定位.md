# CRUSH 算法讲透：Ceph 为什么能不依赖中心元数据做数据定位

## 这篇文章要解决什么问题

前面几篇我们已经走过了几条很关键的主线：

- Ceph 的整体架构
- 一次写请求如何进入 OSD
- OSD 如何通过 PG、Peering、复制和恢复来执行数据面逻辑

但如果你继续追问一个更底层的问题，很快就会碰到 Ceph 最有代表性的设计之一：

**客户端和 OSD 到底是怎么知道“这份数据应该落到哪几个 OSD 上”的？**

在很多传统存储系统里，这个问题往往依赖：

- 中心元数据服务
- 放置表
- 某种集中式映射数据库

而 Ceph 最有名的地方恰恰在于：

- 它不依赖一个“每次查询都要访问”的中心数据定位服务

这背后的核心机制，就是：

- `CRUSH`

如果只允许用一句话先给结论，那就是：

**CRUSH 不是“一个把对象随机分配到磁盘的算法”，而是一套基于集群层级拓扑和放置规则、可由客户端和 OSD 独立计算的数据定位算法。**

这一篇的目标，就是把这件事讲透。

## 先建立第一条边界：CRUSH 解决的是“如何算出位置”，不是“如何保存对象内容”

很多人第一次听到 CRUSH，会自然把它和：

- 副本机制
- PG
- BlueStore
- 持久化

这些概念混在一起。

这会导致理解一直不清楚。

先把边界切开：

### CRUSH 主要解决什么

- 给定一个 PG 和一条放置规则
- 按集群拓扑和故障域要求
- 算出这份数据应该放到哪一组 OSD

### CRUSH 不直接解决什么

- 对象内容如何写入磁盘
- 日志和事务如何提交
- OSD 如何复制数据
- BlueStore 如何落盘

也就是说：

- CRUSH 解决“放哪里”
- OSD / PG / BlueStore 解决“怎么存进去”

这条边界一旦清楚，后面所有概念都会更容易串起来。

## 第二张图：Ceph 的数据定位不是“对象直接映射 OSD”，而是两段式

理解 CRUSH 的第一步，不是先啃 bucket，也不是先背 rule，而是先建立一张非常关键的总图：

```text
对象
  |
  v
哈希
  |
  v
PG
  |
  v
CRUSH rule + CRUSH map
  |
  v
raw OSD set
  |
  v
up set / acting set
```

这张图非常重要，因为它直接说明：

- Ceph 不是“对象名直接算成 OSD”
- 而是“对象先归到 PG，再由 CRUSH 把 PG 放到 OSD”

这也是为什么下一篇会专门讲：

- 为什么 PG 是 Ceph 的灵魂

但在这一篇里，你已经可以先建立最关键的直觉：

**CRUSH 输入的核心不是对象本身，而更接近“经过 pool 和哈希处理后的 PG 语义输入”。**

## 为什么 Ceph 不想依赖中心元数据定位服务

在正式讲 CRUSH 之前，先回答一个设计动机问题：

- Ceph 为什么要这样做？

原因其实很直接。

如果每次对象定位都必须先问一个中心服务：

- 这个对象在哪？

那会立刻带来几个问题：

### 1. 中心瓶颈

所有客户端和 OSD 都要去问同一个地方。

### 2. 中心故障风险

一旦这个服务出问题，整个集群定位能力就受影响。

### 3. 扩展性差

对象越多、请求越多，中心映射维护和查询压力越大。

### 4. 拓扑变化代价高

每次扩容、缩容、迁移，都可能要维护大量中心映射记录。

Ceph 想要的是另一种模型：

- 给你一张全局一致的拓扑图和规则
- 然后每个参与者都能自己算出结果

这就是 CRUSH 的设计哲学。

## 第 1 层：CRUSH map 是“拓扑图 + 规则集合”

要理解 CRUSH，最先要抓住的核心对象就是：

- `CRUSH map`

文档里对它的描述非常接近：

- 它描述的是集群拓扑和放置规则

这句话特别关键，因为它说明 CRUSH map 不是单一数据结构，而是同时包含两部分：

### 1. 拓扑

也就是：

- root
- rack
- host
- device / osd

这样的层级树

### 2. 规则

也就是：

- 在这棵树里，应该怎么挑节点
- 挑几个
- 跨哪一层故障域

所以你可以把 CRUSH map 理解成：

**“Ceph 集群存储拓扑的可计算描述”。**

## 第 2 层：bucket 是拓扑树里的内部节点

在 CRUSH 的概念里，最容易先看到的对象就是：

- bucket

### bucket 到底是什么

可以先非常直白地理解：

- bucket 就是 CRUSH 树里的内部节点

比如：

- root
- row
- rack
- chassis
- host

这些都可以是 bucket 类型。

而叶子通常就是：

- 具体的 OSD

### 为什么 bucket 很重要

因为 CRUSH 不是在一堆平面 OSD 上瞎选，而是在：

- 有层级的故障拓扑里做选择

这正是它能表达：

- 跨 host 放副本
- 跨 rack 放副本
- 跨机房做隔离

这些需求的基础。

所以 bucket 不是“为了看起来有树结构”，而是为了把现实拓扑编码进算法输入里。

## 为什么 Ceph 要把物理世界抽象成层级树

这一点其实非常值得专门讲。

现实世界里的存储节点不是“同质平面集合”，而是天然有故障层级的，比如：

- 同一块盘可能坏
- 同一台主机可能掉
- 同一机架可能断电
- 同一机房可能出问题

如果算法只知道：

- 一堆 OSD 编号

那就没法表达“副本不要落在同一台机器”这种要求。

所以 Ceph 需要一棵树，把现实世界抽象成：

- 叶子是 OSD
- 内部节点是故障层级

这也是 CRUSH 真正比“简单一致性哈希”更强大的地方之一。

## 第 3 层：rule 是“怎么沿这棵树挑节点”的策略

如果 bucket 解决的是“树长什么样”，那：

- rule

解决的就是：

- 该怎么沿着这棵树挑目标

很多人第一次听 CRUSH rule，会以为它只是：

- 一个副本数设置

这显然太窄了。

更准确地说，rule 本质上是在描述：

- 从哪个 root 开始
- 选哪类 bucket
- 选几个
- 遇到冲突和失败怎么重试
- 什么时候要一直选到叶子

也就是说，rule 是 CRUSH 的“放置策略程序”。

## 第 4 层：failure domain 不是附加属性，而是规则真正生效的核心约束

如果说 CRUSH 最值得让初学者马上建立直觉的概念之一，那就是：

- failure domain

### 它到底是什么

最简单的工程化理解是：

- 你希望副本至少分散到哪一层故障域

比如：

- `host`
- `rack`
- `row`

### 为什么它是核心概念

因为 Ceph 的副本放置目标从来都不是：

- “随便找三个 OSD”

而是：

- “找三个 OSD，但它们不能都落在同一个故障层级里”

这也是为什么 failure domain 直接决定了你的高可用边界。

举个最典型的例子：

### failure domain = `host`

意味着：

- 每个副本应该尽量落在不同 host

### failure domain = `rack`

意味着：

- 每个副本应该尽量落在不同 rack

所以 failure domain 决定的不是“放得漂不漂亮”，而是：

- 某类故障发生时，你还能不能保住副本隔离

## 第 5 层：`chooseleaf` 为什么是 CRUSH 里最值得理解的动作之一

讲到这里，最关键的一个动作就出现了：

- `chooseleaf`

这是理解 failure domain 真正如何落地的关键。

### 它不是简单 `choose`

很多人会把：

- choose
- chooseleaf

看成差不多的事，但其实差别很关键。

可以先粗略理解成：

### `choose`

- 从某层 bucket 里选目标

### `chooseleaf`

- 先在某层 bucket 里选，再继续往下钻，直到拿到叶子 OSD

也就是说：

- `chooseleaf` 不是“选完就结束”
- 它带着“跨某层故障域后继续落到最终 OSD”的含义

### 为什么这正好对应 failure domain

比如你设置：

- failure domain = `host`

那么规则本质上就是：

- 每次先选不同 host
- 再从各自 host 下面挑最终 OSD

这就是 chooseleaf 语义的工程落地。

所以你完全可以把这句话记住：

**在 Ceph 里，failure domain 往往不是一个空洞配置项，而是通过 rule 里的 `chooseleaf(type=...)` 真正变成算法行为。**

## 第 6 层：对象为什么不直接输入 CRUSH，而是先变成 PG

这是 CRUSH 最容易被说错的地方之一。

很多文章会图省事，直接说：

- CRUSH 把对象映射到 OSD

这在科普层面勉强能用，但如果你开始读源码，这个说法就不够准确了。

Ceph 更真实的主线是：

1. 对象名或 locator key 先做哈希
2. 先得到 `pg_t`
3. pool 再把 raw pg 归一化到当前 `pg_num`
4. 再生成 CRUSH 输入 seed
5. 最后通过 rule 在 CRUSH map 上选 OSD

### 为什么中间要有 PG

这个问题下一篇会展开讲，但在这一篇你先可以建立两个直觉：

### 1. PG 是定位和数据管理之间的缓冲层

它让 Ceph 不必直接管理“每个对象到每个 OSD”的海量粒度映射。

### 2. CRUSH 的输入规模被 PG 稳定住了

对象无限多，但 PG 是可控数量级。

所以这一篇里最重要的认知不是所有 PG 细节，而是：

- CRUSH 真正吃的是 PG 相关输入，而不是裸对象直接平铺到 OSD

## 第 7 层：对象到 PG 的第一段映射，本质是哈希

从源码主线看，这一段大致会走到：

- `OSDMap::map_to_pg()`
- `OSDMap::object_locator_to_pg()`
- `pg_pool_t::hash_key()`
- `ceph_str_hash()`

### 这段在做什么

简单说就是：

- 取对象名或 locator key
- 按 pool 的对象哈希规则做 hash
- 形成 raw `pg_t`

如果 locator 指定了 hash，也可以直接使用。

### 为什么这里值得强调 locator key

因为 Ceph 并不是永远只拿对象名做定位，有些场景下会借助：

- locator key

来控制对象分布语义。

这说明对象到 PG 的第一段映射并不是“对象名死算”，而是带有一定灵活性的。

## 第 8 层：从 raw PG 到真正用于 CRUSH 的输入，还会经过 pool 规则处理

对象 hash 出来的 raw `pg_t` 并不意味着它已经是最后的 CRUSH 输入。

中间还会经过 pool 相关处理，比如：

- `raw_pg_to_pg()`
- `raw_pg_to_pps()`

### 为什么这一步重要

因为它说明：

- pool 不是“副本数配置盒子”那么简单
- pool 本身也参与定义对象到 PG 再到 CRUSH 输入的变换逻辑

特别是：

- `pg_num`
- `pgp_num`
- `HASHPSPOOL`

这些设置，都会影响后续定位输入。

所以如果你以后看到对象定位结果变化，不一定是 CRUSH rule 本身变了，也可能是：

- pool 的 PG 相关参数变了

## 第 9 层：`OSDMap::_pg_to_raw_osds()` 是 PG 进入 CRUSH 的关键桥

从源码主线上看，真正把 PG 交给 CRUSH 去选 OSD 的关键桥接点之一就是：

- `OSDMap::_pg_to_raw_osds()`

这一层大致会做几件事：

- 把 PG 转成 placement seed
- 取当前 pool 对应的 `crush_rule`
- 调 `crush->do_rule(...)`
- 得到 raw OSD 集合

### 这一步非常值得记住

因为它把“Ceph 业务世界”和“CRUSH 算法世界”接上了：

- 上游仍然是 pool / PG / OSDMap 语义
- 下游则进入 `CrushWrapper` 和底层 mapper

换句话说：

**如果你想找“PG 是怎么真的交给 CRUSH 的”，`_pg_to_raw_osds()` 就是最重要的桥之一。**

## 第 10 层：`CrushWrapper -> crush_do_rule()` 才是 CRUSH 真正执行的地方

很多文章讲 CRUSH 时只停留在概念上，但如果你要真正理解源码主线，就必须看到算法执行落点。

这条线大致会经过：

- `CrushWrapper::do_rule()`
- `crush_do_rule()`
- `crush_choose_firstn()` 以及 chooseleaf 相关分支

### 为什么这里很重要

因为它说明 CRUSH 不是一个神秘黑盒，而是很明确地在做：

- 根据 rule 的步骤解释执行
- 在 bucket 树上做选择
- 处理重试、冲突、out 节点等情况

也就是说，CRUSH 不是“生成一个神奇哈希值”，而是：

- 在一棵层级树上，按规则执行一段可解释的放置程序

这正是它和“平面哈希分布”最大的差别之一。

## 第 11 层：raw OSD set 还不是最终结果，后面还有 up / acting 的修正

这是很多人第一次读定位链路时最容易忽略的地方。

CRUSH 算出来的：

- raw OSD set

并不一定就是最后真正生效的数据放置集合。

后面还会叠加很多 OSDMap 语义上的修正，比如：

- `pg_upmap`
- `pg_temp`
- `primary_affinity`
- down OSD 过滤

所以更真实的主线是：

```text
CRUSH -> raw osds -> up set -> acting set
```

### 为什么这点必须讲清楚

因为很多人会把：

- “CRUSH 输出”

和：

- “当前实际 acting 集合”

混成同一件事。

但在 Ceph 里，这两者并不总是完全相同。

CRUSH 更像是：

- 理论上的原始放置结果

而 OSDMap + Peering 则会进一步把它变成：

- 当前集群真实应该生效的上层集合

## 第 12 层：`up set` 和 `acting set` 为什么不是同义词

这一点是理解 CRUSH 和 Peering 关系的关键。

### `up set`

更接近：

- 在当前 OSDMap 和 CRUSH 规则下，理论上可用的放置集合

### `acting set`

更接近：

- 当前真正对数据生效的集合

它可能会受到：

- `pg_temp`
- peering 过程中的临时调整

等影响。

### 为什么这个区分非常关键

因为它再次说明：

- CRUSH 解决的是“怎么基于规则算位置”
- 但 Ceph 的真实运行位置还要和 Peering、临时调整、故障状态一起决定

这也正是为什么我们前一篇先讲 OSD，再讲 CRUSH，会更顺。

## 第 13 层：CRUSH 为什么比“简单一致性哈希”更适合存储系统

很多人听到“去中心化定位算法”时，会下意识联想到一致性哈希。

但 CRUSH 更适合用下面这组特点来理解：

### 1. 它理解层级拓扑

不是平面节点集合。

### 2. 它理解故障域

不是只关心分布均匀。

### 3. 它支持规则编排

不是只有一种通用放置策略。

### 4. 它让每个参与者都能独立计算

前提是大家拿到同一份 map 和规则。

### 5. 它在扩容和拓扑变化时能保持“尽量少迁移”

这也是 CRUSH 名字里 Controlled 的含义之一值得去体会的地方：

- 不是完全随机重洗
- 而是在规则约束下做可控重分布

所以 CRUSH 的强大之处不是“它也能算哈希”，而是：

- 它把拓扑、故障域和放置规则一起变成了可计算输入

## 第 14 层：从运维视角看，rule 和 failure domain 比“副本数”更能决定集群气质

很多初学者配置 Ceph 时，最容易只看：

- size = 3

但这远远不够。

因为真正决定你这 3 份副本是不是“真的有故障隔离意义”的，往往更取决于：

- rule
- failure domain

举个最直观的例子：

### 3 副本 + failure domain = `host`

意味着单机故障一般不会把三份副本一起打掉。

### 3 副本 + failure domain = `rack`

意味着连机架故障都被纳入隔离目标。

所以在设计 Ceph 拓扑时，更成熟的问题不是：

- 我要几副本？

而是：

- 我希望副本跨哪一层故障域？

这正是 CRUSH 真正体现架构价值的地方。

## 把 CRUSH 压缩成一条“最短主链”

如果你想把这一篇全部内容压缩成一条主线，可以记成下面这样：

```text
对象名 / locator key
  -> hash
  -> raw pg_t
  -> pool 归一化 / 生成 placement seed
  -> 取 pool 对应的 crush rule
  -> 在 CRUSH map 的 bucket 树上执行 rule
  -> choose / chooseleaf 按 failure domain 挑节点
  -> 得到 raw osd set
  -> 再叠加 upmap / pg_temp / down 过滤 / primary_affinity
  -> 得到 up / acting set
```

只要这条链你记住了，后面不管看运维配置、源码调用链还是 PG peering，都会更容易理解。

## 初学者最容易混淆的 9 个点

### 1. 认为 CRUSH 是“对象直接映射 OSD”

更准确地说，是对象先到 PG，再由 CRUSH 把 PG 映射到 OSD。

### 2. 认为 CRUSH 只是“随机均匀分布”

不对。它是基于拓扑和规则的可控放置算法。

### 3. 认为 bucket 就是“文件夹”

不对。它是故障拓扑层级里的内部节点。

### 4. 认为 rule 只是副本数配置

不对。rule 本质上是放置策略程序。

### 5. 认为 failure domain 是一个装饰属性

不对。它直接决定副本隔离的故障边界。

### 6. 认为 chooseleaf 和 choose 差不多

不对。chooseleaf 的关键在于跨故障域选节点后继续下钻到叶子 OSD。

### 7. 认为 CRUSH 输出就是最终写入集合

不对。后面还有 `up set`、`acting set` 和 peering 修正。

### 8. 认为 CRUSH 可以脱离 OSDMap 单独理解完

不完全对。真正执行链路必须连着 pool、OSDMap、PG 一起看。

### 9. 认为 CRUSH 的重点只有算法实现细节

不对。更重要的是“拓扑 + 规则 + 故障域”这套设计思想。

## 这一篇最应该留下的 5 个直觉

### 直觉一：CRUSH 解决的是“如何算出位置”，不是“如何存储内容”

这是它和 OSD/BlueStore 的边界。

### 直觉二：Ceph 的定位是两段式：对象到 PG，PG 到 OSD

不要把对象直接映射 OSD 当成精确描述。

### 直觉三：bucket 表示拓扑，rule 表示策略，failure domain 表示隔离目标

这三者合起来才是 CRUSH 的核心。

### 直觉四：chooseleaf 是 failure domain 真正落成算法行为的关键动作

这一步非常值得建立心智模型。

### 直觉五：CRUSH 给的是原始放置结果，真实运行还要叠加 up/acting 语义

这就是它和 OSDMap / Peering 接上的地方。
