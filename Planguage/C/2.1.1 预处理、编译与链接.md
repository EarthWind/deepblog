# 预处理、编译与链接

这一阶段的核心问题不是“怎么写一个程序”，而是“一个程序为什么能被编译器接受并最终跑起来”。理解预处理、编译与链接，会让你后续面对头文件、多文件项目、重复定义、未定义引用等问题时不再只会靠试错。

## 本阶段目标

- 理解预处理器在做什么。
- 区分声明和定义。
- 理解编译、汇编、链接的大致流程。
- 能看懂并处理基础的多文件构建问题。

## 核心知识

### 1. 预处理在做什么

预处理发生在正式编译之前，常见任务包括：

- 展开宏
- 处理 `#include`
- 处理条件编译

预处理器更像一个“文本改写器”，它并不真正理解完整语义。

### 2. 宏的作用与风险

宏定义示例：

```c
#define PI 3.1415926
#define SQUARE(x) ((x) * (x))
```

宏的优点：

- 简洁
- 编译前替换
- 适合常量和简单模板

宏的风险：

- 没有类型检查
- 参数会被直接替换
- 容易因为优先级产生副作用

### 3. 声明与定义

- 声明：告诉编译器“这个名字存在”
- 定义：真正分配存储或给出函数体

例如：

```c
int add(int a, int b);
```

这是声明。

```c
int add(int a, int b) {
    return a + b;
}
```

这是定义。

### 4. 编译流程概览

可以先建立这条主线：

1. 预处理
2. 编译
3. 汇编
4. 链接

在学习阶段，不要求你掌握每个中间文件细节，但必须知道链接负责把多个目标文件和库拼起来。

## 可运行示例：多文件函数组织的最小模型

先看逻辑内容。假设有两个文件：

`main.c`

```c
#include <stdio.h>

int add(int a, int b);

int main(void) {
    printf("sum = %d\n", add(3, 4));
    return 0;
}
```

`add.c`

```c
int add(int a, int b) {
    return a + b;
}
```

编译时可以一起参与构建：

```bash
gcc main.c add.c -o app
```

运行后输出：

```text
sum = 7
```

这个例子非常重要，因为它说明：

- `main.c` 只知道 `add` 的声明
- `add.c` 提供 `add` 的定义
- 链接阶段把它们真正拼在一起

## 可运行示例：条件编译调试宏

```c
#include <stdio.h>

#define DEBUG

#ifdef DEBUG
#define LOG(msg) printf("[DEBUG] %s\n", msg)
#else
#define LOG(msg)
#endif

int main(void) {
    LOG("program started");
    printf("Hello build process!\n");
    LOG("program ended");
    return 0;
}
```

这个程序展示了条件编译最常见的用途之一：在开发环境下打印调试信息，在发布环境下关闭日志。

## 常见误区

### 把 `#include` 理解成“导入代码库”

更准确地说，它会把目标文件内容插入到当前位置，再继续处理。

### 不理解“为什么函数明明写了却找不到”

这往往是：

- 没有参与链接
- 名字不一致
- 只有声明没有定义

### 宏像函数一样用，但忽略副作用

例如：

```c
#define SQUARE(x) x * x
```

当传入复杂表达式时很容易出错，所以一般要写成 `((x) * (x))`。

## 学习建议

- 遇到“重复定义”“未定义引用”错误时，不要慌，先判断是声明问题、定义问题还是链接问题。
- 学习宏时，重点关注替换结果，而不是只看表面写法。
- 开始尝试把小程序拆成多个 `.c` 和 `.h` 文件。
