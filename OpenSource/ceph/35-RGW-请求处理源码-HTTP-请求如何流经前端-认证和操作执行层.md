# RGW 请求处理源码：HTTP 请求如何流经前端、认证和操作执行层

## 这篇文章要解决什么问题

上一篇我们已经把 RGW 的总结构图立起来了：

- `RGW` 不是底层对象存储本体
- 它是建在 `RADOS` 之上的对象网关和对象服务层
- 对外说的是 S3/Swift 协议
- 对内维护的是用户、bucket、object、bucket index 和对象元数据

但总览讲清楚之后，下一个自然的问题马上就来了：

**一个真实的 HTTP 请求，到底是怎样流经 RGW 的？**

也就是：

- 请求最早是在哪一层被接住的？
- `beast` 前端和 RGW 主逻辑之间怎么衔接？
- URI 是怎么被分发到 S3 或 Swift handler 的？
- 认证是在什么时候做的？
- 为什么最后总会落成一个 `RGWOp`？
- `verify_requester()`、`verify_permission()`、`execute()` 这些步骤分别在干什么？

如果只允许用一句话先给结论，那就是：

**RGW 的请求处理主线可以概括成：`HTTP Frontend` 先把连接和请求读进来，再由 `process_request()` 驱动 `REST` 分发、认证和 `RGWOp` 选择，随后进入统一的“认证后执行框架”，在 `verify_permission()`、`verify_params()`、`pre_exec()`、`execute()`、`complete()` 这一套标准流水线上完成真正的对象操作。**

这一篇的目标，就是把这条主链真正讲清楚。

## 先建立第一条边界：RGW 请求处理不是“一上来就 execute”

这是理解整篇文章的第一步。

很多人第一次读 RGW 源码时，最容易带着一种直觉：

- 请求来了
- 选一个操作
- 执行

这个理解太粗了。

更准确地说，一个请求在 RGW 里至少要经历下面几层语义转换：

- HTTP 请求
- 协议 handler
- 认证后的请求上下文
- `RGWOp`
- 后端具体存储动作

### 为什么这个分层特别重要

因为它说明：

- RGW 不是“前端拿到请求后立刻写 RADOS”

而是先把一个外部 HTTP 请求，一层层收敛成内部可执行的对象服务操作。

只有这个心智图先建立起来，后面的源码才不会乱。

## 第 1 层：先从进程启动角度看，请求入口到底在哪

如果从运行时视角去看，请求链最早的入口当然不是：

- `RGWOp`

而是：

- RGW 的前端

从主程序看，RGW 的启动主线大致是：

```text
rgw_main
  ->
AppMain
  ->
init_frontends1()
  ->
init_storage()
  ->
cond_init_apis()
  ->
init_frontends2()
```

这条线最值得先记住的直觉是：

- **RGW 会先把前端和后端都装配起来，然后才真正开始接流量。**

### 这说明什么

说明 RGW 请求处理不是一个单函数故事，而是：

- 先有进程级装配
- 再有前端监听
- 然后才进入每个请求的处理链

这点要先立住。

## 第 2 层：为什么 `beast` 前端是当前主请求入口

从前端实现看，当前最值得优先理解的主入口就是：

- `beast`

也就是基于 `asio` 的 HTTP 前端。

### 为什么这层很关键

因为从真实运行时看，一个请求最早会经历的是：

- accept 连接
- 读 HTTP header/body
- 构造请求对象
- 调用 `process_request(...)`

也就是说，前端层真正解决的是：

- 网络连接和 HTTP 读写

而不是：

- 对象语义处理

### 所以这里最准确的职责边界是什么

- `frontend` 负责把网络请求安全、稳定地交给 RGW 内部处理主链

这点很重要。

## 第 3 层：为什么 `process_request()` 是理解整条主线的最佳中心点

如果只允许在 RGW 请求链里抓一个总入口函数，我会优先抓：

- `process_request()`

### 为什么

因为它几乎正好位于：

- 前端接流量

和

- 内部认证/分发/执行

之间的中心位置。

也就是说：

- 前端层把请求交进来
- `process_request()` 再去驱动后面的所有核心步骤

### 所以它最适合被理解成什么

- RGW 的请求处理总调度函数

这就是为什么它是读请求链源码时最好的切入点。

## 第 4 层：`process_request()` 里第一件最重要的事，不是认证，而是先选对 handler

很多人会自然以为：

- 请求进来后的第一核心动作一定是认证

但在 RGW 里，更靠前的一件事其实是：

- 先选 handler

也就是先回答：

- 这是哪种协议请求？
- 该由哪个 REST manager 管？
- 最终该用哪个 handler 去处理？

### 为什么这一步必须先发生

因为：

- S3 和 Swift 的请求模型不同
- 同一个 HTTP method 在不同协议里含义也不同
- 后续认证策略、bucket/object 解析方式也都依赖 handler

所以真正合理的顺序是：

- 先识别协议与资源形态
- 再进入对应协议的认证与执行逻辑

这点非常关键。

## 第 5 层：为什么 `RGWREST::get_handler()` 是“协议分发层”的核心入口

从源码结构看，REST 分发的核心入口就是：

- `RGWREST::get_handler()`

### 它在干什么

大致可以概括成下面这条主线：

```text
preprocess()
  ->
mgr.get_manager(...)
  ->
manager.get_handler(...)
  ->
handler.init(...)
```

### 这说明什么

说明 RGW 的 REST 分发不是：

- 一个巨大的 if/else 直接分所有协议和资源

而是分成了几层：

- 总入口
- manager 选择
- 具体 handler 选择

这就是典型的：

- “先按协议/资源域分层，再把请求交给具体处理器”

的结构。

## 第 6 层：为什么 `RGWRESTMgr` 可以理解成“资源域级路由器”

继续往下走，`RGWRESTMgr` 这层特别值得先建立一个直觉：

- 它不像最终业务 handler 那样细
- 也不像总入口那样粗

更适合把它理解成：

- 资源域级路由器

### 为什么这样说

因为它主要在回答：

- 当前 URI/resource 更像属于哪个大类处理域

比如：

- S3 默认域
- Swift 域
- admin 域
- STS / IAM 相关路径

### 这意味着什么

说明 RGW 路由的真正结构是：

- 总入口先选 manager
- manager 再选更具体的 handler

这能很好解释为什么 RGW 同时能承载很多协议子域，却没有把路由逻辑完全写成一坨。

## 第 7 层：为什么 S3 / Swift 的 handler 选择本身就已经包含了大量语义判断

到了具体协议层，事情会变得更有意思。

以 S3 为例，请求并不是只按：

- `GET`
- `PUT`

这样简单分流。

更准确地说，handler 选择通常还会考虑：

- 是 service 级请求
- 还是 bucket 级请求
- 还是 object 级请求
- 有没有 subresource
- 有没有特殊 query/action
- 是不是 website / STS / IAM 等特殊路径

### 这说明什么

说明 RGW 的“路由”已经不只是 Web 框架意义上的 URI 路由，而是：

- 协议语义路由

也就是说，它在路由阶段就已经开始理解：

- 这个请求在对象存储语义上到底想做什么

这点特别重要。

## 第 8 层：为什么 `handler->get_op()` 是请求从“协议语义”进入“内部操作模型”的转折点

前面几层都还更像：

- 在理解请求

而到了：

- `handler->get_op()`

请求就真正开始转成 RGW 内部的执行对象了。

### 这一步在回答什么

- 这个请求最终应该落成哪个 `RGWOp` 子类？

也就是说，RGW 在这里把：

- 一个 HTTP/REST/S3/Swift 请求

收敛成了：

- 一个统一的内部操作对象

### 为什么这一步如此关键

因为从这里开始，后面的执行框架就可以更多面向：

- `RGWOp`

而不是直接面向：

- 原始 HTTP 请求

这就是整个架构能稳定扩展的重要原因之一。

## 第 9 层：为什么 `RGWOp` 是 RGW 请求处理架构里的核心抽象

如果只选一个 RGW 请求处理里的核心类，我会选：

- `RGWOp`

### 为什么

因为它把一个对象服务动作所需要的关键阶段统一抽象出来了：

- `verify_requester()`
- `verify_permission()`
- `verify_op_mask()`
- `verify_params()`
- `pre_exec()`
- `execute()`
- `complete()`

### 这意味着什么

意味着 RGW 的请求执行不是：

- 每个 REST 路径各自写一套完整流程

而是：

- 大家共享同一套操作生命周期框架

然后再由具体 `RGWOp` 子类填充差异化逻辑。

这就是它非常值得优先理解的原因。

## 第 10 层：为什么 `verify_requester()` 不是权限检查，而是更早一层的认证入口

到了 `RGWOp` 这层，一个很容易混淆的点就是：

- `verify_requester()`
- `verify_permission()`

看起来都像“验证”

但它们不是同一层问题。

### `verify_requester()`

- 更偏“你是谁”
- 也就是认证

### `verify_permission()`

- 更偏“你能不能干这件事”
- 也就是授权

### 为什么一定要区分这两步

因为一个请求可能：

- 能成功识别身份

但仍然：

- 没有权限操作当前 bucket/object

所以从主链角度看，这里必须建立一个清晰边界：

- **先认证，再授权。**

这在 RGW 里体现得非常明确。

## 第 11 层：为什么 S3 认证入口最终会收束到认证策略栈

如果顺着 S3 请求往里看，你会发现 S3 的认证入口并不是：

- 某个单点函数硬编码做完所有逻辑

而是会收束到一套：

- 认证策略栈

### 为什么这层设计很重要

因为 RGW 要支持的认证来源和变体很多，比如：

- SigV2
- SigV4
- STS
- 外部身份源
- 本地 key

如果全部写死在一个流程里，后面几乎没法维护。

### 所以更准确的理解方式是

- handler 先决定用哪类认证入口
- 认证入口再走 `Strategy::apply()`
- 认证策略栈逐层尝试并把结果写回 `req_state`

这就是 RGW 认证结构比表面看起来更规整的地方。

## 第 12 层：为什么 `req_state` 是理解请求链的关键背景对象

读到这里时，一个很容易被忽略、但其实非常重要的对象就是：

- `req_state`

### 为什么它关键

因为整条请求链很多阶段都在围绕它工作：

- 前端解析后的请求上下文
- handler 初始化的信息
- 认证结果
- 用户/bucket/object 解析结果
- 权限掩码
- 操作阶段中的运行态数据

### 所以最准确的理解方式是

- `req_state` 不是一个零散参数包
- 它更像请求生命周期中的共享上下文对象

这能帮助你理解为什么认证、路由、权限和执行层都不断读写它。

## 第 13 层：为什么 `postauth_init()` 是一个特别值得注意的中间步骤

很多人读请求链时，会只盯：

- 认证前
- 认证后 execute

但中间其实还有一个很重要的过渡步骤：

- `handler->postauth_init()`

### 为什么它值得注意

因为认证完成后，请求上下文通常还要进一步完成：

- bucket/object 归一化
- 某些资源定位修正
- 认证后才能安全完成的上下文初始化

### 这说明什么

说明认证成功并不等于：

- 请求马上可以执行业务操作

中间还要完成：

- 从“已知身份”到“已准备好执行对象服务操作”的状态收敛

这就是 `postauth_init()` 的价值。

## 第 14 层：为什么真正的统一执行框架在 `rgw_process_authenticated()`

如果说 `process_request()` 是总调度中心，那么：

- `rgw_process_authenticated()`

就是认证后的统一执行框架中心。

### 它最值得记住的主线是

```text
init_permissions()
  ->
retarget()
  ->
read_permissions()
  ->
init_processing()
  ->
verify_op_mask()
  ->
verify_permission()
  ->
verify_params()
  ->
pre_exec()
  ->
execute()
  ->
complete()
```

### 为什么这条线特别关键

因为它其实把 RGW 所有操作都压进了一套标准骨架里。

无论你最后是：

- GET Object
- PUT Object
- DELETE Object
- Bucket 操作

都基本要经过这条骨架。

这就是 RGW 请求处理结构真正稳定的地方。

## 第 15 层：为什么 `init_permissions()`、`read_permissions()`、`verify_permission()` 要分成多步

很多人第一次看到这里会觉得：

- 为什么权限相关要拆这么多层？

这是个好问题。

### 最直观的理解方式是

- `init_permissions()` 更像准备阶段
- `read_permissions()` 更像把 ACL / policy / bucket/object 权限材料读进来
- `verify_permission()` 才是最终判断

### 为什么不直接一步做完

因为 RGW 的权限判断本来就不只是：

- 看一个用户是不是 owner

它还可能涉及：

- ACL
- bucket policy
- IAM policy
- OPA
- 匿名访问
- 子资源和操作类型差异

所以权限系统天然需要：

- 先准备上下文
- 再收集策略材料
- 最后统一裁决

这就是它被拆成多步的原因。

## 第 16 层：为什么 `execute()` 前还有 `verify_params()` 和 `pre_exec()`

继续往执行阶段再深一点，也能看到一个典型设计思路：

- 真正干活之前，还要继续过两道门

### `verify_params()`

- 更偏参数合法性和语义前置检查

### `pre_exec()`

- 更偏执行前准备动作

### 这意味着什么

说明 RGW 的执行框架不是：

- 只要认证授权过了就直接干

而是还会确保：

- 参数层面成立
- 前置状态准备完成

这让 `execute()` 本身能更聚焦于：

- 真正的业务写读删逻辑

这是一种很典型也很健康的执行分层。

## 第 17 层：为什么 `complete()` 不是可有可无的收尾函数

再往后看，`complete()` 也很容易被新读者低估。

很多人会本能觉得：

- 核心都在 `execute()` 了
- `complete()` 只是顺手收尾

其实不应该这么轻视它。

### 为什么

因为一个请求真正对外结束之前，还往往要处理：

- 响应状态收尾
- 统计与日志
- 请求生命周期结束动作

### 所以更准确的理解方式是

- `execute()` 负责主要业务动作
- `complete()` 负责把这次操作完整地闭环

这在请求处理框架里很重要。

## 第 18 层：为什么同一个 `GET`，最终也可能变成不同的 `RGWOp`

这是理解协议分发和操作选择关系的关键例子。

很多人会下意识觉得：

- `GET` 就对应一个读操作

其实在 RGW 里完全不够。

因为一个 `GET` 请求到底变成什么 `RGWOp`，还会受到很多条件影响：

- 是 service 级别
- 还是 bucket 级别
- 还是 object 级别
- 是否带某个 query/subresource
- 是否是特殊资源请求

### 这说明什么

说明：

- HTTP method 只是最外层信号
- 真正的内部操作类型要结合资源语义一起判断

这就是为什么 `handler->get_op()` 在 RGW 里如此关键。

## 第 19 层：为什么 GET Object 和 PUT Object 最适合作为理解 `RGWOp` 的典型样本

如果你后面要继续往深处读，我最建议优先抓两个典型操作：

- `RGWGetObj`
- `RGWPutObj`

### 为什么选这两个

因为它们几乎覆盖了对象服务操作里最常见的两类主线：

## `RGWGetObj`

- 读路径
- 权限检查
- 条件请求
- 对象读取与响应输出

## `RGWPutObj`

- 写路径
- 上传相关语义
- 权限和配额
- 对象写入与元数据更新

### 这说明什么

只要这两个典型 Op 读通了，你对：

- 认证后如何走进具体操作
- 不同操作怎样复用统一框架

就会有非常清晰的直觉。

## 第 20 层：把整条 RGW 请求主链压缩成一张图

如果这一篇只记一张图，我建议记下面这张：

```text
beast frontend
  ->
handle_connection()
  ->
process_request()
  ->
RGWREST::get_handler()
  ->
handler->get_op()
  ->
op->verify_requester()
  ->
handler->postauth_init()
  ->
rgw_process_authenticated()
  ->
init_permissions / read_permissions
  ->
verify_permission / verify_params
  ->
pre_exec
  ->
execute
  ->
complete
```

这张图里最关键的一句话是：

- **HTTP 请求并不是直接变成 RADOS 操作，而是先被 RGW 逐层收敛成 `RGWOp`，再走统一执行框架。**

## 第 21 层：为什么这一篇先停在 `RGWOp` 层，而不继续深挖 RADOS 落地

因为如果这一篇再继续往下，会立刻进入另一类复杂度：

- bucket index
- object head/tail
- 元数据对象
- RADOS 读写与事务更新

这些当然非常重要，但它们更适合放到后面的：

- bucket/object 元数据设计
- 多站点同步
- 具体 PUT/GET 深挖

这些主题里讲。

这一篇最重要的目标不是：

- 讲完 RGW 所有后端细节

而是先把：

- 请求怎么进来
- 怎么分发
- 怎么认证
- 怎么选 `RGWOp`
- 怎么进入统一执行骨架

这条主链真正讲清楚。

只有这张图先立住，后面再看具体 bucket/object 落地时才不会乱。

## 用一句话重新概括这篇

如果把这篇全部内容压缩成一句尽量准确的话，我会这样说：

**RGW 的请求处理主线可以概括为：HTTP 请求先由 `beast` 前端接入，再进入 `process_request()` 做 REST 分发与 handler 选择，随后通过 `handler->get_op()` 把协议请求收敛成内部 `RGWOp`，先完成认证与认证后初始化，再进入 `rgw_process_authenticated()` 这条统一执行骨架，依次完成权限准备、授权、参数校验、预执行、真正执行与完成收尾；因此 RGW 真正强大的地方，不是某个单点函数，而是这套把多协议请求稳定压缩成统一操作生命周期的框架。**

## 把整篇压缩成一条最短骨架

如果你只想记一条骨架，可以记成下面这样：

```text
前端接住 HTTP
  ->
process_request 选 handler
  ->
handler 选 RGWOp
  ->
verify_requester 做认证
  ->
postauth_init 收敛上下文
  ->
统一执行框架做授权/校验/预执行
  ->
execute 真正执行业务
  ->
complete 收尾
```

只要这条骨架记住了，RGW 请求处理主线基本就不会乱。

## 初学者最容易混淆的 10 个点

### 1. 认为 RGW 请求一进来就直接执行对象操作

不对。中间还有路由、认证、Op 选择和统一执行框架。

### 2. 认为认证是第一件事

不完全对。通常还要先选对协议 handler。

### 3. 认为 `process_request()` 自己干完了所有事

不对。它更像请求总调度中心。

### 4. 认为 REST 路由只是普通 URI 匹配

不对。它还包含协议语义判断。

### 5. 认为 `RGWOp` 只是一个简单命令对象

不对。它承载了完整操作生命周期接口。

### 6. 认为 `verify_requester()` 和 `verify_permission()` 是一回事

不对。前者偏认证，后者偏授权。

### 7. 认为认证成功就能直接执行

不对。还要做 `postauth_init()`、权限准备、参数校验等步骤。

### 8. 认为 `execute()` 前不需要太多准备

不对。`verify_params()` 和 `pre_exec()` 都很关键。

### 9. 认为 `complete()` 只是可忽略的收尾

不对。它负责请求生命周期闭环。

### 10. 认为同一个 HTTP method 一定只对应一个 `RGWOp`

不对。还要结合资源类型和 subresource 语义。

## 这一篇最应该留下的 5 个直觉

### 直觉一：`process_request()` 是请求链中心，而不是前端或具体 Op

这是第一原则。

### 直觉二：RGW 会先把协议请求收敛成 `RGWOp`

再进入统一执行框架。

### 直觉三：认证和授权在 RGW 里是明确分层的

这点必须立住。

### 直觉四：`rgw_process_authenticated()` 是真正的执行骨架

它比单个业务 Op 更值得先理解。

### 直觉五：RGW 的强大在于统一生命周期框架，而不是零散函数技巧

这是最核心的架构价值。
