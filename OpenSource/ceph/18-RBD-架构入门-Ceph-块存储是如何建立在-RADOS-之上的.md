# RBD 架构入门：Ceph 块存储是如何建立在 RADOS 之上的

## 这篇文章要解决什么问题

前面几篇我们已经把 Ceph 底层数据面拆得比较清楚了：

- `RADOS` 是什么
- 对象怎样映射到 `PG`
- 写请求怎样进 `OSD`
- `BlueStore` 怎样把对象落到本地
- `EC` 怎样通过编码换容量

但对很多真正使用 Ceph 的人来说，最熟悉的入口可能不是：

- 对象存储

而是：

- 块存储

也就是：

- 虚拟机磁盘
- 云盘
- `/dev/rbdX`
- QEMU 里的 `rbd:pool/image`

这时一个非常关键的问题就会出现：

**Ceph 明明底层是 RADOS 对象存储，为什么上层却能把它用成“块设备”？**

也就是：

- `RBD image` 到底是什么？
- 一个 image 和底层 object 是什么关系？
- `striping` 在 RBD 里意味着什么？
- `snapshot` 和 `clone` 为什么能成立？
- `librbd`、`krbd`、QEMU 之间到底是什么关系？

如果只允许用一句话先给结论，那就是：

**RBD 不是 Ceph 底层多出的一种新存储介质，而是建立在 RADOS 之上的块设备语义封装：它把一个逻辑 image 拆成一组有规则命名和布局的对象，再通过 `librbd` 或 `krbd` 向上提供“像磁盘一样”的访问接口。**

这一篇的目标，就是把这句话真正展开。

## 先建立第一条边界：RBD 不是“另一套独立存储系统”，而是 RADOS 之上的块语义层

这是理解 RBD 的第一步。

很多人第一次接触 Ceph 时，很容易把下面几件事想成彼此平行：

- `RADOS`
- `RBD`
- `CephFS`
- `RGW`

这种看法只对了一半。

更准确地说：

- `RADOS` 是底层分布式对象存储
- `RBD`、`CephFS`、`RGW` 都是建立在它之上的上层服务

也就是说：

- `RBD` 不是和 `RADOS` 并列的另一套底座
- 它是建立在 `RADOS` 之上的块存储抽象

### 这意味着什么

意味着当你说“我有一个 RBD image”时，本质上你并不是说：

- Ceph 真的给你造了一块物理磁盘

而是说：

- Ceph 用一组 RADOS 对象，给你模拟出了一块逻辑块设备

这就是理解 RBD 的第一条主线。

## 第 1 层：为什么对象存储之上还能做出块设备语义

这个问题特别值得先讲清楚。

很多人直觉里会觉得：

- 对象存储是对象存储
- 块设备是块设备
- 这两种抽象差很远

这没错，但“差很远”不等于“不能转换”。

### 块设备真正需要的是什么

对上层来说，块设备最重要的语义其实是：

- 有一个线性的逻辑地址空间
- 可以按偏移读写
- 可以像磁盘一样被文件系统、虚拟机、数据库使用

### 对象存储真正提供的是什么

RADOS 提供的是：

- 一组命名对象
- 每个对象可以存二进制数据和元数据
- 可以通过客户端库去读写这些对象

### 那中间差什么

差的不是“能力完全不存在”，而是：

- 一个把线性块地址空间映射到对象集合的中间层

而这个中间层就是：

- `RBD`

所以从架构角度看，RBD 的关键工作不是“自己存数据”，而是：

- 把块设备语义翻译成对象读写语义

## 第 2 层：RBD image 到底是什么

讲 RBD 时，最容易被误解的词之一就是：

- `image`

很多人会下意识把它想成：

- 一个大文件
- 一个块设备文件
- 一块单独存储区域

这些理解都不够准确。

更准确地说：

- `RBD image` 是一个逻辑块设备映像

它向上表现为：

- 一块可以按块访问的磁盘

但向下并不是：

- 一个单独的大对象

而是：

- 一组按照规则组织的对象和元数据

### 这点为什么特别重要

因为只要你意识到：

- 一个 image 不是一个对象

后面这些问题就都顺了：

- 为什么会有 object size
- 为什么会有 striping
- 为什么 snapshot/clone 可以只复制元数据关系
- 为什么 resize 不一定意味着真正复制整块内容

## 第 3 层：image 和 object 到底是什么关系

这是 RBD 的绝对核心问题。

最短答案是：

**一个 RBD image 会被拆成多个 RADOS objects。**

也就是说：

- image 是上层逻辑块设备视角
- object 是底层真实存储视角

### 这条映射为什么成立

因为一块大的逻辑磁盘，不可能总被当成一个巨对象来处理。

Ceph 更自然的做法是：

- 把 image 的线性地址空间按一定规则切分
- 每段映射到一个对象

这样做的好处很多：

- 更易并行
- 更易分布
- 更易扩展
- 更适合底层 PG/CRUSH/OSD 的对象模型

所以 image 到 object 的关系，本质上就是：

- 块语义到对象语义的第一层映射

## 第 4 层：为什么 RBD 一定要有 striping

如果 image 最终会拆成多个对象，那自然就会遇到：

- 按什么规则拆？

这就引出了：

- `striping`

### 最直观的理解

你可以把它先理解成：

- 把一个逻辑 image 的地址空间切成很多条带，再映射到多个对象

这件事的作用不是为了“概念更优雅”，而是非常工程化的：

- 避免一个大 image 压在一个对象上
- 提升并行度
- 让不同对象可以落到不同 PG/OSD

### 为什么这对块设备特别重要

因为块设备天然容易很大。

如果一块 100GB、1TB、甚至更大的 image 只映射成一个对象，很多事情都会变得不合理：

- 单对象过大
- 并行度差
- 迁移和恢复粒度差
- 底层对象模型失去意义

所以 RBD 的 striping 不是附属特性，而是：

- 块设备建立在对象存储之上时的基础机制

## 第 5 层：`object-size`、`stripe-unit`、`stripe-count` 到底分别在表达什么

这是 RBD 最容易把人看晕的一组参数。

可以先不追求所有细节，而是先建立用途感。

### `object-size`

- 一个底层对象大概承载多少数据

### `stripe-unit`

- 条带化时每一段的基本粒度

### `stripe-count`

- 一个 object set 里会有多少并行条带成员

### 为什么这组参数重要

因为它们共同决定了：

- image 的逻辑地址空间怎样被切分
- 切分后对象布局长什么样
- 连续写和并发访问会分散到哪些对象上

所以这组参数不是“创建时顺手填一下的实现细节”，而是：

- RBD 块语义映射到底层对象布局的关键控制参数

## 第 6 层：为什么说 RBD 的底层本质仍然是对象，而不是块

这点特别值得反复强调。

因为很多人一旦看到 `/dev/rbd0` 或 QEMU 里的虚拟磁盘，就会自然忘记底层其实是：

- RADOS objects

但从 Ceph 架构角度看，真正重要的事实一直没变：

- Ceph 底层仍然只存对象

RBD 所做的事情只是：

- 把块设备访问翻译成对象读写

所以当上层发来：

- “读 image 偏移 X，长度 Y”

RBD 内部要做的其实是：

- 算出命中了哪些对象
- 把偏移翻译成对象内偏移
- 再通过 `librados` 去读写对应对象

这就是为什么下一篇 RBD 读写路径源码会自然接在这一篇后面。

## 第 7 层：为什么 snapshot 在 RBD 里可以做得很自然

理解了 image 是一组对象之后，snapshot 为什么成立就更容易理解了。

先给最短直觉：

- snapshot 不是“把整块磁盘复制一遍”

而更接近：

- 对当前 image 状态建立一个可引用的时间点视图

### 为什么对象化布局有利于 snapshot

因为 RBD 的 image 不是一个巨型单体文件，而是：

- 一组数据对象
- 再加上一组描述 image 的元数据对象

这意味着 snapshot 可以更多依赖：

- 元数据冻结
- 对象引用关系
- 写时复制或 copy-up 语义

而不是每次都真的复制整块数据。

所以 snapshot 的自然成立，本质上来自两点：

- image 已经被拆成对象集合
- RBD 维护了足够的 image 元数据关系

## 第 8 层：为什么 clone 也是 RBD 架构里非常自然的能力

如果 snapshot 只是建立一个时间点视图，那么 clone 更进一步：

- 它允许基于一个 snapshot 派生出新的 image

### 为什么这在对象化布局里特别自然

因为 child image 不需要在一开始就把 parent 的所有数据对象都复制一遍。

更高效的方式是：

- 先记录 parent/child 关系
- 读 miss 时回退到 parent
- 当 child 第一次要写某块时，再做 copy-up

这就是很多文档里反复提到的 layering / clone 语义核心。

### 这说明什么

说明 clone 并不是 RBD 后来附加上的一个小技巧，而是：

- 对象化 image + 元数据关系建模

自然能长出来的一种能力。

## 第 9 层：所以 RBD 真正存的不只是数据对象，还有元数据对象

这是很多初学者会忽视的一点。

如果你只盯着：

- image 会拆成多个数据对象

你还是会漏掉 RBD 架构的一半。

因为 RBD 还需要维护很多元数据，比如：

- image 自己的身份与 id
- header 信息
- object 命名前缀
- features
- parent/child 关系
- snapshot 关系
- object-map 等高级特性信息

这也是为什么你在源码里会看到：

- `header_oid`
- `object_prefix`
- `rbd_id.$image_name`
- `rbd_directory`

这些对象或字段。

这说明：

- RBD 不只是“把数据切成对象”
- 它还需要一整套元数据对象来维持块设备语义

## 第 10 层：`librbd` 在这套架构里到底是什么

讲到这里，终于可以讲客户端侧的核心库了：

- `librbd`

### 它的最准确定位是什么

可以先概括成一句话：

- `librbd` 是用户态 RBD 客户端库

### 它解决什么问题

它不是去重写底层 RADOS，而是建立在：

- `librados`

之上，为上层提供：

- image 打开/关闭
- 读写
- snapshot
- clone
- resize
- 锁
- object-map

这些块设备语义和镜像语义。

也就是说：

- `librados` 更接近对象访问库
- `librbd` 更接近块设备语义库

这个分层非常重要。

## 为什么说 `librbd` 是“块语义翻译层”

因为它真正做的事情可以概括成：

- 接住块设备请求
- 把逻辑偏移映射到底层对象
- 维护 image 元数据语义
- 再通过 `librados` 去和 Ceph 集群交互

所以如果你问：

- 在用户态里，是谁把“磁盘读写”翻译成“对象读写”的？

答案通常就是：

- `librbd`

## 第 11 层：`krbd` 又是什么

如果 `librbd` 是用户态路径，那么：

- `krbd`

就是另一条非常重要的路径。

### 它的最准确定位

- `krbd` 是 Linux 内核态 RBD 客户端路径

更直观地说，当你执行：

```bash
rbd device map pool/image
```

背后实际上是在：

- 通过 userspace 工具触发内核 rbd 模块
- 把一个 RBD image 映射成 Linux 块设备

最终你会看到：

- `/dev/rbdX`

### 这意味着什么

意味着在 `krbd` 路径下，Ceph 块设备不只是“应用库直接用”，而是：

- 真正进入 Linux block layer 语义

所以 `krbd` 的意义不是“另一个访问库”，而是：

- 把 RBD 接到内核块设备栈里

## 第 12 层：`librbd` 和 `krbd` 的关系应该怎么理解

很多人容易把两者理解成：

- 谁更新、谁落后
- 谁高级、谁低级

更准确的理解其实是：

- 它们是两条不同的客户端接入路径

### `librbd`

- 用户态
- 常见于 QEMU、qemu-img、OpenStack 某些用户态集成

### `krbd`

- 内核态
- 常见于把 image 映射成 `/dev/rbdX` 供宿主机或内核块层使用

### 所以它们不是简单替代关系

而是：

- 面向不同接入方式的两种客户端实现

这点非常关键，因为很多后续的性能、语义、特性差异，都和“用户态还是内核态”有关。

## 第 13 层：QEMU 为什么通常直接走 `librbd`

这是虚拟化场景里非常重要的一点。

现代 Ceph 虚拟化栈里，一个非常常见的路径是：

```text
QEMU
  ->
librbd
  ->
librados
  ->
Ceph 集群
```

### 为什么这条路径重要

因为它说明：

- QEMU 不一定要先依赖宿主机 `rbd map`
- 它可以直接在用户态使用 RBD

这也是文档里为什么反复强调：

- QEMU 可以直接把 Ceph RBD image 当后端块设备使用

### libvirt 在这里是什么角色

更准确地说：

- libvirt 是编排和配置层
- 真正和 Ceph 交互的仍然是 QEMU 配合 `librbd`

这个区分也非常值得点出来。

## 第 14 层：`ImageCtx` 为什么是理解 `librbd` 的最好入口之一

如果从源码层面看 RBD，最值得先抓住的对象之一就是：

- `ImageCtx`

### 为什么

因为你会在这里直接看到很多 RBD 架构最关键的字段：

- `data_ctx`
- `md_ctx`
- `header_oid`
- `object_prefix`
- `features`
- snapshot 相关状态

### 这说明什么

说明 `ImageCtx` 不是一个随便的上下文壳子，而几乎就是：

- 一个打开中的 RBD image 在用户态的核心表示

也就是说，当你理解了 `ImageCtx` 里的字段时，你对这几个关键事实就会更有感觉：

- image 有数据对象和元数据对象两个维度
- image 不是裸数据块，而是带 features 和元数据状态的复杂实体
- `librbd` 的很多操作，最终都要落在这些上下文信息上

## 第 15 层：`cls_rbd` 为什么也很关键

讲 RBD 架构时，很多人会忽略：

- `cls_rbd`

其实它非常重要。

### 它是什么

可以先简单理解成：

- RBD 元数据在 OSD 侧的类方法入口

### 为什么它重要

因为 RBD 不只是客户端自己记一堆状态，它很多镜像元数据操作会借助：

- OSD class

来维护 image 相关的集群侧元数据对象。

这说明 RBD 的实现并不是：

- “所有智能都在客户端”

而是：

- 客户端库 + 集群内元数据对象/类方法

一起协作完成。

这也是为什么 RBD 架构既像一个客户端库，又像一个建立在 RADOS 之上的完整子系统。

## 第 16 层：把 RBD 架构压缩成一张最重要的图

如果这一篇只记一张图，我建议记下面这张：

```text
上层应用 / 虚拟机 / 文件系统
  |
  +-- 用户态路径: QEMU -> librbd -> librados -> RADOS
  |
  +-- 内核态路径: rbd map -> krbd -> RADOS

RBD image
  |
  +-- 数据面: 一组按 striping 规则组织的 RADOS objects
  |
  +-- 元数据面: header / id / directory / snapshot / clone / features 等对象和状态
```

这张图里最重要的不是某个函数名，而是四层关系：

### 第一层

- RBD 是块语义，不是底层新存储

### 第二层

- image 会拆成对象

### 第三层

- image 还带一整套元数据对象和特性状态

### 第四层

- 上层接入路径分为用户态 `librbd` 和内核态 `krbd`

## 第 17 层：为什么说 RBD 最精彩的地方不是“存成块”，而是“把对象系统变成块系统”

这是整篇最想让读者留下的认知。

因为从表面看，RBD 给你的只是：

- 一块虚拟磁盘

但从系统设计看，它真正厉害的地方在于：

- 底层完全还是对象系统
- 却能通过布局、元数据、快照、克隆、映射关系和客户端库，向上给出块设备语义

所以 RBD 的本质并不是：

- Ceph 在 RADOS 旁边又造了一套磁盘系统

而是：

- Ceph 成功把对象存储能力重新包装成了块存储服务

这也是为什么理解 RBD 的最好办法永远不是从 `/dev/rbd0` 开始，而是从：

- image -> object -> metadata -> client path

这条链开始。

## 用一句话重新概括 RBD

如果把这篇全部内容压缩成一句尽可能准确的话，我会这样说：

**RBD 是 Ceph 建立在 RADOS 之上的块设备抽象层：它把一个逻辑 image 映射为多组数据对象和元数据对象，再通过 `librbd` 或 `krbd` 向上暴露磁盘语义，并在此基础上提供 snapshot、clone、resize 等能力。**

## 把 RBD 架构压缩成一条最短骨架

如果你只想记一条骨架，可以记成下面这样：

```text
RADOS 负责底层对象存储
  ->
RBD 把一个逻辑 image 拆成多个对象
  ->
再配上一组 image 元数据对象
  ->
librbd 在用户态提供块设备语义
  ->
krbd 在内核态提供块设备映射
  ->
QEMU 等上层应用再把它当成普通磁盘使用
```

只要这条链记住了，RBD 的大部分结构就不会乱。

## 初学者最容易混淆的 10 个点

### 1. 认为 RBD 和 RADOS 是两套并列底座

不对。RBD 建立在 RADOS 之上。

### 2. 认为一个 image 就是一个底层对象

不对。一个 image 会拆成多个对象。

### 3. 认为 `/dev/rbdX` 说明底层真的存在一块磁盘

不对。那只是块设备语义暴露方式。

### 4. 认为 striping 只是一个优化参数

不对。它是块语义映射到底层对象布局的基础机制。

### 5. 认为 RBD 只需要数据对象，不需要额外元数据

不对。snapshot、clone、features、image 身份都依赖元数据对象。

### 6. 认为 snapshot 就是整盘复制

不对。它更依赖对象化布局和元数据关系。

### 7. 认为 clone 一开始就复制 parent 全部数据

不对。更典型的是 parent/child 引用和写时 copy-up。

### 8. 认为 `librbd` 和 `krbd` 只是名字不同

不对。一个是用户态路径，一个是内核态路径。

### 9. 认为 libvirt 直接和 Ceph 交互

不对。它更多是编排层，真正交互常发生在 QEMU + `librbd`。

### 10. 认为理解 RBD 只要会用 `rbd map` 就够了

不对。那只能看见内核态映射，不足以理解整体架构。

## 这一篇最应该留下的 5 个直觉

### 直觉一：RBD 是块语义层，不是另一套底层存储

这一点必须先立住。

### 直觉二：image 的本质是一组对象，而不是一个巨型文件

这是理解后续所有能力的基础。

### 直觉三：RBD 既有数据对象面，也有元数据对象面

只盯数据面会理解不完整。

### 直觉四：`librbd` 和 `krbd` 是两条不同客户端接入路径

这会决定后续读写路径和使用场景。

### 直觉五：snapshot 和 clone 能成立，是因为 RBD 在对象布局和元数据关系上做了足够多的设计

这正是 RBD 架构的精彩之处。
