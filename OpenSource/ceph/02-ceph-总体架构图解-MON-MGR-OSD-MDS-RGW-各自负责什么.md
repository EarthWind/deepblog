# Ceph 总体架构图解：MON、MGR、OSD、MDS、RGW 各自负责什么

## 这篇文章要解决什么问题

上一篇我们完成了第一步认知切换：Ceph 不是单纯的文件系统，也不是单纯的云盘，更不是单纯的 S3 服务，而是一个统一的分布式存储底座。在这个底座之上，Ceph 再向外提供块、文件和对象三种主要访问方式。

但只理解“底座和接口”的关系还不够。真正开始部署、排障、读源码时，你很快就会碰到另一类更具体的问题：

- `MON`、`MGR`、`OSD`、`MDS`、`RGW` 到底谁是控制面，谁是数据面？
- 为什么有些组件是所有集群都需要的，有些却只在特定场景下出现？
- 一次客户端请求到底会经过哪些进程，不会经过哪些进程？
- 为什么 Ceph 强调“控制面很重要，但不直接在数据路径中转发普通 IO”？

这一篇的目标就是建立 Ceph 的第一张“进程与职责全景图”。你不需要在这一篇里掌握所有内部细节，但至少要形成一种稳定直觉：

- 某个组件存在的意义是什么
- 它维护什么状态
- 它处在哪条路径上
- 它和其他组件的边界在哪里

只要这张图建立起来，后面你再去看 `PG`、`CRUSH`、`BlueStore`、`MDS` 状态机、`RGW` 请求执行层时，就不会总是迷路。

## 先给出结论：Ceph 的角色不是并列堆砌，而是分层协作

可以先把 Ceph 粗略分成三层：

- 控制面：维护集群地图、状态、配置、编排、监控视图
- 数据面：真正执行对象存储、复制、恢复、持久化
- 接口层：面向不同客户端提供块、文件、对象网关等访问语义

对应到最常见的几个进程，大致可以这样看：

- `MON`：控制面核心，维护集群地图与关键一致状态
- `MGR`：管理平面核心，提供监控、Dashboard、编排插件、扩展模块
- `OSD`：数据面核心，执行对象读写、复制、恢复、回填和后台任务
- `MDS`：CephFS 元数据面核心，只在使用 CephFS 时需要
- `RGW`：对象网关层，只在需要 S3/Swift 接口时需要

先注意两个常见误区：

- 这些角色不是每次 IO 都全部参与
- 不是每个 Ceph 集群都必须同时启用 `MDS` 和 `RGW`

如果你的集群只提供 `RBD`，那 `MDS` 和 `RGW` 都可能完全不在这条业务路径上。如果你的集群只提供 CephFS，那 `MDS` 会很重要，但 `RGW` 可以不存在。

## 第一张总图：把几个核心角色放进一张图里

先建立最重要的一张系统图：

```text
                           +---------------------------+
                           |           客户端           |
                           |---------------------------|
                           | librados / RBD / CephFS / |
                           | S3 client / Swift client  |
                           +-------------+-------------+
                                         |
                  +----------------------+----------------------+
                  |                                             |
                  v                                             v
        +-------------------+                         +-------------------+
        |    直接访问集群    |                         |    协议网关访问    |
        | librados / RBD /  |                         |       RGW         |
        | CephFS client     |                         +---------+---------+
        +---------+---------+                                   |
                  |                                             |
                  +----------------------+----------------------+
                                         |
                                         v
                         +---------------+---------------+
                         |           RADOS / OSD         |
                         | 对象存储、复制、恢复、落盘     |
                         +---------------+---------------+
                                         ^
                                         |
                 +-----------------------+-----------------------+
                 |                                               |
                 v                                               v
       +-------------------+                           +-------------------+
       |       MON         |                           |       MGR         |
       | 集群地图/仲裁/状态 |                           | 管理模块/监控/编排 |
       +-------------------+                           +-------------------+

                         CephFS 场景下，客户端还会与 MDS 交互
                                   +-------------------+
                                   |       MDS         |
                                   | 元数据协调与管理   |
                                   +-------------------+
```

这张图最关键的阅读方式不是看“谁和谁连着”，而是看“谁属于哪种职责”。

你可以先记住一句话：

**`MON` 和 `MGR` 更偏控制面，`OSD` 是数据面核心，`MDS` 和 `RGW` 是面向特定访问语义的专用角色。**

## `MON`：集群控制面的核心之一

第一次接触 Ceph 的人，经常把 `MON` 想成“集群中央控制器”，甚至误以为所有请求都要经过它。这种理解只对了一半。

`MON` 的确非常关键，但它关键的地方在于维护集群的一致视图，而不是承担普通数据 IO 的转发。

你可以先把 `MON` 理解成：**负责维护“整个集群现在长什么样”的权威角色。**

它关心的主要是这些东西：

- monitor 自身的成员关系和仲裁
- OSD 的存活状态与集群地图
- pool、放置规则、PG 相关全局信息
- MDS、MGR 等其他服务的全局状态映射
- 配置、认证、健康状态等控制面信息

从仓库结构上看，理解 `MON` 的第一入口通常是 `src/mon`。里面你会反复看到这些名字：

- `Monitor`
- `MonMap`
- `MonClient`
- `OSDMonitor`
- `MDSMonitor`
- `MgrMonitor`
- `Paxos`

这组名字已经在暗示 `MON` 的职责了：它不是一个“大杂烩服务”，而是由多个 monitor service 共同维护不同类型的集群全局状态。

### `MON` 不做什么

为了建立边界感，理解 `MON` 不做什么和理解它做什么同样重要。

`MON` 通常不负责：

- 代替客户端执行普通对象读写
- 转发每一次块设备读写请求
- 承担 CephFS 文件数据本体的读写
- 直接持久化业务对象数据

换句话说，`MON` 是“告诉大家规则和地图的人”，不是“替大家搬运业务数据的人”。

### 为什么 `MON` 不能在数据路径上当中心转发器

如果所有客户端 IO 都要穿过 `MON`，那它很快会成为：

- 性能瓶颈
- 扩展瓶颈
- 故障放大点

Ceph 的设计目标恰恰是让客户端尽可能依据集群地图和放置规则直接找到数据归属，而不是每次都问中心节点“下一步该找谁”。

所以对 `MON` 最准确的第一印象应该是：

**它是控制面强中心，不是数据面强中转。**

## `MGR`：Ceph 的管理平面与扩展能力中心

如果说 `MON` 解决的是“集群关键状态一致性”和“权威地图分发”，那么 `MGR` 解决的更多是：

- 如何把集群运行状态暴露得更友好
- 如何承载监控、Dashboard、编排和插件能力
- 如何让 Ceph 在不把所有逻辑都塞进 `MON` 的情况下持续扩展管理功能

很多初学者会问：既然已经有 `MON` 了，为什么还要有 `MGR`？

一个很直接的理解方式是：

- `MON` 更偏核心控制面与关键状态维护
- `MGR` 更偏管理平面、可观测性、模块扩展和运维集成

`MGR` 最容易被看到的能力包括：

- Dashboard
- Prometheus 指标导出
- orchestrator 接口
- 自动扩缩容或建议类模块
- 各种面向运维的管理命令和插件功能

从源码结构看，和 `MGR` 相关的核心目录通常包括：

- `src/mgr`
- `src/pybind/mgr`

前者更偏 C++ 侧管理进程框架，后者则承载大量 Python 模块化能力，例如：

- `src/pybind/mgr/prometheus`
- `src/pybind/mgr/orchestrator`
- `src/pybind/mgr/rook`
- `src/pybind/mgr/rgw`
- `src/pybind/mgr/volumes`

这也反过来说明 `MGR` 的角色定位：它本身就是 Ceph 的一个扩展平台。

### `MGR` 和 `MON` 的边界

你可以先用一句很粗但很有用的话来记：

- `MON` 更像集群关键元信息和一致性状态的守门员
- `MGR` 更像管理功能、监控能力和生态扩展的承载平台

实际系统里二者当然会协作，但不要把它们理解成“两个都能做一切的管理进程”。

### `MGR` 会不会进入业务 IO 主路径

通常不会。

`MGR` 对运维很重要，对 Dashboard 很重要，对监控和编排很重要，但一次普通对象写入、RBD 数据块落盘、CephFS 文件数据落入后端时，主路径通常不依赖 `MGR` 做中转。

所以从“请求链路优先级”上看：

- 没有 `MGR`，很多管理能力会退化甚至缺失
- 没有 `MON`，集群控制面无法成立
- 没有 `OSD`，就没有真正的数据存储能力

这三者的重要性都很高，但重要的方式不同。

## `OSD`：Ceph 真正的数据面执行核心

如果只能选一个“最代表 Ceph 数据面本体”的角色，那通常就是 `OSD`。

`OSD` 的职责可以概括成一句话：

**Ceph 最终是靠 `OSD` 来存对象、处理副本、执行恢复，并把数据真正写进后端存储介质。**

它负责的事情包括：

- 接收对象读写请求
- 参与主副本或纠删码相关执行流程
- 执行复制、确认、提交等数据路径动作
- 维护 PG 相关状态机
- 处理 peering、recovery、backfill
- 调度 scrub、snaptrim 等后台任务
- 调用 BlueStore 等后端存储引擎完成持久化

从源码结构看，后续读到最多的目录之一就是 `src/osd`。你会不断看到这些核心名字：

- `OSD`
- `PG`
- `PrimaryLogPG`
- `PeeringState`
- `ReplicatedBackend`
- `ECBackend`
- `OSDMap`

这部分代码之所以庞大，不是因为 Ceph “故意写复杂”，而是因为分布式数据面本来就要同时解决：

- 放置
- 一致性
- 故障恢复
- 后台校验
- 性能隔离
- 持久化语义

### 为什么说 `OSD` 才是数据面核心

因为无论上层接口是：

- `librados`
- `RBD`
- `CephFS`
- `RGW`

只要最后要把用户数据真正落进 Ceph 集群，绝大多数情况下都绕不开 `OSD`。

也就是说，`RBD`、`CephFS`、`RGW` 的差异，主要体现在“上层语义怎么组织”；但一旦进入底层对象存取，最终大多都会汇入 `OSD`。

### `OSD` 和 `MON` 的关系

这两个角色很容易被一起谈，但职责必须分开看：

- `MON` 决定集群视图和规则
- `OSD` 根据规则执行具体数据动作

一个更形象的比喻是：

- `MON` 像地图和交通规则发布者
- `OSD` 像真正跑在路上的运输系统

后面你学习 `CRUSH`、`PG`、peering 时，这个分工会越来越重要。

## `MDS`：只在 CephFS 场景下出现的元数据面核心

`MDS` 是最容易被误解的角色之一。很多人一看到它，就以为 Ceph 所有数据都必须经由 `MDS`。其实完全不是。

`MDS` 的职责非常明确：**它是 CephFS 的元数据服务，不是整个 Ceph 集群的通用元数据中心。**

换句话说：

- 你不用 CephFS，就可以完全没有 `MDS`
- 你在做 `RBD` 或 `RGW` 时，业务数据路径通常和 `MDS` 没关系

那为什么 CephFS 一定要有 `MDS`？

因为文件系统语义里最复杂的部分往往不是数据块本身，而是元数据：

- 路径解析
- inode / dentry 关系
- 目录操作
- rename
- 权限
- session
- capability 协调
- 多客户端缓存一致性

这些都不是单纯“对象存一下”就能解决的。所以 CephFS 在 `RADOS` 之上又引入了专门的元数据面角色，也就是 `MDS`。

从源码结构看，CephFS 的核心实现主要分布在：

- `src/mds`
- `src/client`

其中在 `src/mds` 里你会反复看到这些关键名字：

- `MDSDaemon`
- `MDSRank`
- `MDCache`
- `Locker`
- `MDLog`
- `Server`

这说明 `MDS` 的核心工作并不是“存文件数据”，而是“让文件系统元数据语义成立”。

### `MDS` 在 CephFS 路径里到底做什么

可以先建立一个简单直觉：

- 文件数据本体通常还是落在 `RADOS` / `OSD`
- 文件系统层面的元数据协调由 `MDS` 承担

所以 `MDS` 更像是 CephFS 的“元数据大脑”，不是 Ceph 全系统的“数据搬运工”。

## `RGW`：对象存储协议的出口，而不是 RADOS 本身

`RGW` 即 RADOS Gateway。它的职责是把 Ceph 底层能力封装成对象存储服务接口，对外提供 S3 或 Swift 风格协议。

很多人一听“Ceph 对象存储”，会把 `RGW` 和 `RADOS` 混成一层。这个区分必须尽早建立：

- `RADOS` 是 Ceph 内部的分布式对象存储底座
- `RGW` 是 Ceph 向外暴露对象存储协议时的网关层

`RGW` 处理的是这类问题：

- HTTP 请求解析
- S3/Swift 协议兼容
- 认证鉴权
- bucket 与对象元数据组织
- multipart upload、生命周期、多租户等高层语义

从源码结构看，这部分主要在 `src/rgw`，后面你会经常遇到：

- `rgw_op`
- `rgw_rest`
- 各种认证、bucket、metadata 相关实现

### `RGW` 在系统中的位置

`RGW` 不是所有 Ceph 客户端的必经入口。

只有当你的业务通过 S3/Swift 风格接口访问 Ceph 时，`RGW` 才会出现在请求路径上。如果你走的是 `RBD` 或 `CephFS`，那主路径通常并不经过 `RGW`。

所以对 `RGW` 的正确定位是：

**它是对象网关层，不是 Ceph 底层对象引擎本体。**

## 把几类典型请求路径串起来看

讲完各自职责后，最有用的做法就是把常见访问路径串起来。你会发现，不同组件的差别不是“谁更高级”，而是“谁在当前场景里真正参与请求”。

## 路径一：原生对象访问

如果客户端通过 `librados` 直接访问 Ceph，对应的大致路径可以先理解成：

```text
应用
  -> librados
  -> 获取集群地图/建立连接
  -> 直接与目标 OSD 通信
  -> OSD 执行对象读写
```

在这条路径里：

- `MON` 提供集群地图和控制面信息
- `OSD` 执行真实数据操作
- `MGR` 通常不在主 IO 路径
- `MDS` 不参与
- `RGW` 不参与

这是理解 Ceph“非中心化数据路径”的最直接例子。

## 路径二：RBD 块存储访问

如果业务使用的是 `RBD`，你可以先把路径理解成：

```text
虚拟机 / QEMU / 内核块层
  -> librbd
  -> librados
  -> OSD
  -> 后端持久化
```

在这条路径里：

- `RBD` 负责把块设备语义翻译成对象级操作
- `OSD` 负责底层对象执行与复制恢复
- `MON` 仍然提供控制面视图
- `MDS` 不参与
- `RGW` 不参与

所以你会看到，`RBD` 的主路径离 `OSD` 很近，离 `MDS` 和 `RGW` 都很远。

## 路径三：CephFS 文件系统访问

如果业务使用的是 CephFS，大致路径会变成：

```text
应用
  -> CephFS 客户端
  -> 与 MDS 交互元数据
  -> 与 OSD 交互文件数据
```

这条路径非常重要，因为它清楚体现了 `MDS` 的边界：

- 路径解析、目录项、权限、会话等元数据操作依赖 `MDS`
- 文件数据本体最终仍由 `OSD` 承担

也就是说，`MDS` 不是“替 CephFS 存一切”，而是“让文件系统语义成立”。

## 路径四：S3/Swift 网关访问

如果业务通过对象网关访问，大致路径可以理解成：

```text
S3 / Swift 客户端
  -> RGW
  -> RGW 完成协议解析与鉴权
  -> RGW 访问底层 RADOS
  -> OSD 执行对象存取
```

在这条路径里：

- `RGW` 是外部协议入口
- `OSD` 仍然是数据最终执行者
- `MDS` 不参与
- `MGR` 通常不在主 IO 路径

从这里你也能看出一个规律：**`OSD` 是不同上层接口最终汇流的数据面核心。**

## 哪些角色是“全局必需”，哪些是“按场景出现”

这也是部署和架构理解中非常实用的一个判断。

### 几乎所有 Ceph 集群都离不开

- `MON`
- `OSD`
- 通常还有 `MGR`

其中：

- 没有 `MON`，控制面和集群一致视图无法正常成立
- 没有 `OSD`，就没有真正的数据存储能力
- 没有 `MGR`，现代 Ceph 的管理体验、监控与模块能力会明显缺失

### 只在特定场景下需要

- `MDS`：只在启用 CephFS 时需要
- `RGW`：只在启用对象网关服务时需要

这个判断非常重要，因为它能帮助你避免一种常见误解：以为每个 Ceph 组件都天然属于每条业务链路。实际上，Ceph 是一个按能力组合的系统。

## 读源码时应该先把哪些目录和角色对应起来

如果这一篇的目的，是让你之后看源码不迷路，那么最值得先建立的映射关系就是下面这组：

- `src/mon` 对应 `MON`
- `src/mgr` 与 `src/pybind/mgr` 对应 `MGR`
- `src/osd` 对应 `OSD`
- `src/mds` 对应 `MDS`
- `src/client` 对应 CephFS 客户端
- `src/rgw` 对应 `RGW`

这组映射之所以重要，是因为后面每当你问“这个需求应该去哪层找实现”，其实都离不开“角色 -> 目录”的脑内索引。

例如：

- 想看集群地图和 monitor 相关逻辑，先去 `src/mon`
- 想看对象读写、PG、恢复、复制，先去 `src/osd`
- 想看 CephFS 元数据控制流，先去 `src/mds`
- 想看管理插件和 Dashboard 生态，先去 `src/pybind/mgr`
- 想看对象网关请求处理，先去 `src/rgw`

## 初学者最容易混淆的 6 个边界

### 1. `MON` 和 `MGR` 都是“管理进程”，所以作用差不多

不对。它们都属于控制面或管理面相关角色，但定位不同：

- `MON` 更偏关键一致状态和权威地图
- `MGR` 更偏管理扩展、监控和生态能力

### 2. `MON` 是 Ceph 的请求转发中心

不是。`MON` 很关键，但普通数据 IO 主路径通常不经由 `MON` 做中心转发。

### 3. `MDS` 是 Ceph 的全局元数据中心

不是。`MDS` 只服务于 CephFS 元数据语义，不是 `RBD`、`RGW` 或整个 `RADOS` 的统一元数据入口。

### 4. `RGW` 就等于 Ceph 的对象存储底层

不对。`RGW` 是对象协议网关，底层对象存储本体是 `RADOS`，数据执行核心是 `OSD`。

### 5. `MGR` 不在数据路径，所以不重要

不对。它可能不在普通 IO 主路径，但对集群管理、可观测性、插件生态和运维能力至关重要。

### 6. `OSD` 只是“磁盘守护进程”

这个说法太低估它了。`OSD` 不只是“看着磁盘”，而是承担了 Ceph 最核心的数据面协议和执行逻辑。

## 这一篇最应该留下的 4 个直觉

如果读完这一篇，只保留 4 个判断，我建议你保留下面这些。

### 直觉一：先分控制面、数据面、接口层，再谈组件

一上来就背每个进程的所有功能，很容易乱。先分层，再看角色，理解成本会低很多。

### 直觉二：`MON` 管“集群视图”，`OSD` 管“真实数据动作”

这是 Ceph 最关键的一组边界，也是后续理解 `CRUSH`、`PG`、恢复机制的基础。

### 直觉三：`MDS` 和 `RGW` 都不是全场景必备组件

它们是为特定访问语义服务的：一个服务文件系统元数据，一个服务对象协议网关。

### 直觉四：不同上层接口最终大多汇流到 `OSD`

这能帮你快速判断：某个问题更可能出现在上层语义层，还是底层数据执行层。
