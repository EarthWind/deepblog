## 二、对象与内存模型

### 2.1 对象、存储与生命周期

存储是一段字节区域，对象是在某段存储中具有类型和生命周期的实体。获得内存不等于对象已经开始生命周期，释放前也可能需要结束对象生命周期。

```cpp
struct Point { int x; int y; };
Point point{1, 2};
```

对象具有：类型、大小、对齐要求、值、地址和生命周期。

### 2.2 对象表示与值表示

对象表示是组成对象的 `unsigned char`/`std::byte` 序列；其中参与表示值的位属于值表示，其余可能是 padding。

不能用 `memcmp` 普遍判断两个 C++ 对象是否值相等，因为：

- padding 可能不同。
- 多种位模式可能表示相同值。
- 指针和浮点存在特殊表示。

### 2.3 大小、对齐和填充

```cpp
struct Example
{
    char flag;
    std::uint64_t value;
    char code;
};
```

编译器可能插入 padding 满足成员和数组元素对齐。使用 `sizeof`、`alignof`、`offsetof`（仅适用标准布局类型）观察。

调整成员顺序可能减小尺寸，但公开类型布局涉及 ABI，且不能牺牲语义清晰性。

### 2.4 `alignas`

```cpp
struct alignas(64) Counter
{
    std::atomic<std::uint64_t> value{};
};
```

可用于硬件、SIMD 或减少 false sharing，但 64 不是所有平台统一缓存行大小。过度对齐会浪费内存并改变分配要求。

### 2.5 trivial 与 standard-layout

相关 type traits：

- `is_trivially_copyable`。
- `is_trivially_destructible`。
- `is_standard_layout`。
- `has_unique_object_representations`。

只有满足相应规则时，才能安全使用某些逐字节复制、C 互操作或布局操作。

### 2.6 `std::byte` 与字节观察

```cpp
auto bytes = std::as_bytes(std::span{&object, 1});
```

可观察对象表示，但不能因此任意把字节解释成另一个类型对象。

### 本节练习

1. 比较不同成员顺序的结构体尺寸。
2. 查看数组元素地址验证尾部 padding。
3. 判断哪些类型可平凡复制。
4. 解释值相等为什么不保证对象字节相同。

## 三、生命周期与底层构造

### 3.1 placement new

```cpp
alignas(T) std::byte storage[sizeof(T)];
T* object = ::new (storage) T(arguments...);
std::destroy_at(object);
```

placement new 不分配内存，只在给定存储中构造对象。必须保证大小、对齐和生命周期正确。

### 3.2 原始存储

容器实现需要区分：

- 已分配但未构造的槽位。
- 已构造元素。
- 已销毁但尚未释放的存储。

优先使用 `std::construct_at`、`std::destroy_at` 和 allocator traits，而不是手工拼接不完整协议。

### 3.3 隐式生命周期类型

现代 C++ 对部分隐式生命周期类型允许特定分配或字节操作开始生命周期，但规则精细。若类型非平凡或代码可用正常构造表达，就不要依赖边界规则。

### 3.4 `std::launder`

在少数原地重构、const/基类子对象等情况下，旧指针不能直接用于新对象，`std::launder` 可获得符合模型的指针。它不是修复任意类型转换或悬空指针的工具。

### 3.5 `std::start_lifetime_as`

C++23 提供相关显式生命周期工具，适用于从字节存储开始某些对象生命周期的底层场景。使用前必须检查类型条件和工具链支持。

### 本节练习

1. 用标准构造/销毁工具管理一个对象槽位。
2. 为固定容量容器画出每个槽位的生命周期状态。
3. 找出 placement new 后遗漏析构的问题。

## 四、别名、类型转换与未定义行为

### 4.1 strict aliasing

编译器通常假设不兼容类型的指针不会指向同一对象，以便优化。通过不允许的类型指针读取对象可能产生未定义行为。

错误示例：

```cpp
float value{1.0F};
auto bits = *reinterpret_cast<std::uint32_t*>(&value);
```

正确查看位模式：

```cpp
auto bits = std::bit_cast<std::uint32_t>(value);
```

要求两类型大小相同且满足 `bit_cast` 的约束。

### 4.2 `reinterpret_cast`

它只执行标准允许的低层转换，不会自动解决：

- 对齐。
- 对象生命周期。
- 别名规则。
- 有效对象类型。
- 指针来源和可达性。

能编译不代表能解引用。

### 4.3 指针运算

指针运算只在同一数组对象及其尾后位置范围内有定义。尾后指针可比较和作为结束标记，但不可解引用。

### 4.4 整数溢出

- 有符号整数溢出：未定义行为。
- 无符号整数：按模运算，但仍可能是业务 bug。
- 浮点溢出遵循浮点环境和实现规则，不能与整数类比。

尺寸计算应防止乘法溢出：

```cpp
if (count > max_size / element_size) {
    return error;
}
```

### 4.5 移位和位运算

移位量为负或不小于类型宽度等情况可能未定义；有符号负值的位操作也容易出现不可移植结果。位级协议优先使用固定宽度无符号类型。

### 4.6 指针 provenance

现代优化器不仅把指针视作整数地址，还追踪其与存储和对象来源的关系。指针转整数再任意运算后转回不保证普遍可用。底层序列化不应持久化进程地址。

### 4.7 Sanitizer

UBSan、ASan 能发现大量问题，但并不覆盖所有 UB，也不能让依赖 UB 的代码变正确。配合严格警告、静态分析和代码审查。

## 五、字节序、编码与序列化

### 5.1 endian

```cpp
if constexpr (std::endian::native == std::endian::little) {
    // ...
}
```

网络和文件格式必须规定字节序，不能直接写入宿主对象内存并假设跨平台稳定。

### 5.2 `byteswap`

C++23 `std::byteswap` 可转换整数端序。浮点和复合类型需要先定义明确格式，不能简单对任意对象字节反转。

### 5.3 安全序列化原则

- 明确字段宽度、字节序和编码。
- 不序列化 padding、裸指针或 vtable。
- 检查长度和整数溢出。
- 版本化格式。
- 对不可信数据做边界验证。
- 不用 `reinterpret_cast<T*>` 直接读取可能未对齐字节。

### 5.4 文本编码

UTF-8 字节数不等于字符数；code point 也不等于用户感知字素。性能优化不能破坏编码边界和规范化需求。

## 六、编译与链接

### 6.1 翻译过程

```text
源文件 → 预处理 → 编译 → 汇编 → 目标文件 → 链接 → 程序/库
```

诊断定位：

- 预处理问题：宏、包含顺序。
- 编译问题：语法、类型、模板实例化。
- 链接问题：未定义符号、重复定义、库顺序。
- 加载问题：动态库缺失、版本或搜索路径。

### 6.2 符号与名称修饰

C++ 重载通过 name mangling 编码函数名、命名空间和参数类型。不同编译器 ABI 可能不兼容。使用 `nm`、`objdump`、`dumpbin` 等查看符号。

### 6.3 ODR

同一实体的多份定义必须符合单一定义规则。常见风险：

- 头文件定义非 inline 全局变量。
- 不同宏使 inline/template 定义不同。
- 构建配置混用不兼容类布局。

### 6.4 静态与动态链接

静态库链接进最终产物；动态库在加载时解析。动态链接涉及导出、装载路径、运行库、ABI 和版本管理。

### 6.5 LTO

链接时优化允许跨翻译单元内联和全程序分析，可能改善性能和体积，但增加链接成本，也可能暴露潜伏的 ODR/UB 问题。

### 6.6 PGO

典型流程：

1. 构建带插桩程序。
2. 用代表性负载运行。
3. 收集 profile。
4. 用 profile 重新优化构建。

训练负载不真实可能使性能变差。

### 6.7 Modules 对编译的影响

C++20 Modules 主要改善依赖隔离和编译模型，不保证运行更快。模块仍需构建系统按依赖顺序生成和消费 BMI。

## 七、汇编与优化器

### 7.1 阅读目标

不必成为汇编专家，先识别：

- 函数调用与返回。
- load/store。
- 条件跳转。
- 循环。
- 栈帧。
- SIMD 指令。
- 是否内联和消除计算。

### 7.2 优化级别

- `-O0`：便于源码调试。
- `-O1/-O2`：常规优化。
- `-O3`：更激进，未必总更快。
- `-Os/-Oz`：偏向体积。
- MSVC 常见 `/O2`。

Debug 性能不能代表生产性能。

### 7.3 As-if Rule

编译器可进行任何不改变标准可观察行为的转换。若程序含 UB，优化器不必保留程序员想象的行为。

### 7.4 Compiler Explorer

用于比较：

- 不同编译器与选项。
- 抽象是否被内联消除。
- vectorization report。
- 模板实例化代码。

小片段汇编不能替代完整程序 profiling。

### 7.5 逃逸与内联

对象是否逃逸函数、函数能否内联、别名是否明确都会影响优化。`final`、LTO、restrict 类扩展等可能帮助，但应先让接口和数据流清晰。

## 八、CPU 缓存与数据布局

### 8.1 存储层次

寄存器、L1/L2/L3 cache、内存和存储设备延迟差异巨大。性能常受数据移动而非算术运算限制。

### 8.2 空间与时间局部性

- 空间局部性：访问相邻数据。
- 时间局部性：短时间重复访问同一数据。

连续 `vector` 常比节点链表更快，即使理论复杂度相似。

### 8.3 AoS 与 SoA

```cpp
struct Particle { float x, y, z, mass; };
std::vector<Particle> aos;

struct Particles {
    std::vector<float> x, y, z, mass;
};
```

- AoS 适合经常处理完整单个对象。
- SoA 适合批量处理少数字段和 SIMD。
- 混合布局可能更实际。

应根据访问模式和可维护性基准比较。

### 8.4 false sharing

多个线程写不同变量，但变量位于同一缓存行，会引发缓存行来回转移。分片计数、填充或重新布局可改善，但过度 padding 会浪费内存。

### 8.5 预取

硬件预取器擅长规则连续访问。软件预取是平台相关高级优化，距离错误会增加流量或污染缓存。仅在 profiler 证明需要时使用。

### 8.6 数据导向设计

从“每帧/每请求实际访问哪些数据”出发布局，而不是只按现实对象分类。目标包括：

- 热/冷数据分离。
- 批处理。
- 减少指针追踪。
- 压缩工作集。

## 九、分支、虚调用与间接访问

### 9.1 分支预测

规则分支通常预测良好，随机数据分支代价较高。无分支写法不一定更快，可能增加指令和数据依赖。

### 9.2 数据排序

按状态分组或排序可提高分支可预测性和批处理效率，但排序本身有成本，并可能改变语义。适合反复处理同一批数据。

### 9.3 虚函数成本

主要成本可能来自：

- 间接调用难以内联。
- 对象分散导致 cache miss。
- 分支目标不稳定。

单次 vtable 查找通常不是首要瓶颈。可比较 `variant`、批量分组、静态多态或数据表方案。

### 9.4 `likely/unlikely`

```cpp
if (condition) [[likely]] { }
```

这是优化提示而非语义保证。错误提示可能伤害性能，需使用真实 profile。

## 十、SIMD 与向量化

### 10.1 SIMD

单条指令同时处理多个数据元素。适合：

- 图像和音频。
- 矩阵和数值计算。
- 编解码。
- 批量过滤和变换。

### 10.2 自动向量化

帮助编译器：

- 连续数据。
- 简单循环。
- 明确无别名。
- 减少循环携带依赖。
- 使用优化构建和向量化报告。

不要直接假设编译器已向量化，应检查报告和汇编。

### 10.3 显式 intrinsic

intrinsic 提供架构指令访问，但带来：

- 平台依赖。
- 多版本分派。
- 对齐和尾部处理。
- 可维护性成本。

先使用标准算法、编译器自动向量化或成熟 SIMD 库。

### 10.4 `std::simd`

标准 SIMD 接口目标进入 C++26。在此之前实现与命名空间支持因工具链而异。学习 mask、lane、load/store 和 reduction 概念，同时把可移植生产使用与实验代码分开。

### 10.5 浮点语义

重排和向量化可能改变舍入误差、NaN 和 signed zero 行为。`fast-math` 会放宽语义，不能在需要严格 IEEE 行为的代码中盲目使用。

## 十一、内存分配与 allocator

### 11.1 分配成本

动态分配可能涉及元数据、锁、系统调用、碎片和 cache miss。实际分配器常有线程缓存，不能只凭次数推断成本。

### 11.2 减少分配

- 值语义与栈对象。
- `vector::reserve`。
- 批量分配。
- 小对象优化。
- arena 或 pool。
- 复用缓冲区。

避免为减少分配而制造危险的全局对象池。

### 11.3 allocator 模型

标准容器通过 allocator 获取原始存储，再在其中构造元素。自定义 allocator 必须满足复杂协议；普通项目优先使用 polymorphic allocator。

### 11.4 `std::pmr`

```cpp
std::array<std::byte, 4096> buffer;
std::pmr::monotonic_buffer_resource resource{buffer.data(), buffer.size()};
std::pmr::vector<std::pmr::string> values{&resource};
```

`pmr` 将分配策略从容器类型中部分运行时化，便于批量生命周期和性能实验。

### 11.5 常用 memory resource

- `monotonic_buffer_resource`：只增长，整体销毁，适合请求/帧生命周期。
- `unsynchronized_pool_resource`：单线程小块分配池。
- `synchronized_pool_resource`：带同步的池。
- `null_memory_resource`：拒绝上游分配，可验证固定预算。

### 11.6 arena

arena 适合大量对象具有相同总体生命周期的场景。代价是不能单独回收，且析构、外部引用和峰值内存必须设计清楚。

### 11.7 对象池

对象池适合构造昂贵、频繁复用且数量可控的对象。风险：

- 旧状态未清理。
- 跨线程竞争。
- 悬空句柄。
- 内存长期不归还。
- 比通用分配器更慢。

### 本节练习

1. 记录项目分配次数和大小分布。
2. 使用 monotonic resource 优化一次解析任务。
3. 用 null resource 验证固定内存预算。
4. 比较默认分配器、pool 和 arena。

## 十二、Profiling 与 Benchmark

### 12.1 正确流程

```text
定义指标 → 建立基线 → profile → 提出假设 → 最小修改 → 基准验证 → 回归测试
```

先优化耗时占比最高且确实影响目标指标的部分。

### 12.2 指标

- 吞吐量。
- 平均与 P95/P99 延迟。
- CPU 时间与墙钟时间。
- 分配次数与峰值内存。
- cache miss、分支 miss。
- 上下文切换和锁等待。
- 功耗或指令数。

### 12.3 工具

- Linux：perf、Valgrind/Callgrind、heaptrack。
- Windows：Visual Studio Profiler、WPA、Intel VTune。
- 跨平台：Tracy、Google Benchmark。
- 编译器：优化报告、Compiler Explorer。

### 12.4 微基准陷阱

- 计算被优化删除。
- 只测 Debug。
- 数据始终在 cache。
- 输入过于理想。
- 分支结果恒定。
- 把设置/分配计入目标操作。
- 未报告方差。
- CPU 频率和后台负载不稳定。

### 12.5 Amdahl 定律

若某部分只占总时间 5%，即使无限加速它，整体提升也有限。优化应结合端到端 profile。

## 十三、高性能并发

### 13.1 分片与局部聚合

每线程先在本地累计，再合并，通常比每次更新共享 atomic 或 mutex 更好。

### 13.2 NUMA

多路系统中，内存访问成本取决于内存所在节点。线程固定、first-touch、分区数据和跨节点通信会影响性能。只有目标硬件需要时深入优化。

### 13.3 锁竞争

优化顺序：

1. 减少共享状态。
2. 缩短但保持完整的临界区。
3. 分片锁或数据。
4. 批量操作。
5. 更适合的同步原语。
6. 最后才考虑无锁。

### 13.4 内存序

弱化内存序前应写出同步证明，并在目标架构测试。x86 上“看起来正常”不能证明 ARM 等弱内存模型正确。

- 验证不同尺寸、对齐和尾部元素。
- 比较数值误差与性能。

### 项目三：Arena 与固定容量容器

- 正确处理大小、对齐和生命周期。
- 支持非平凡对象构造销毁。
- 失败行为明确。
- 使用 Sanitizer 和随机操作测试。
- 与标准容器和 pmr 比较。

### 项目四：数据导向粒子系统

- 实现 AoS 和 SoA 两版。
- 批量更新位置与速度。
- 分离热冷数据。
- 比较 cache miss、向量化和帧耗时。
- 加入多线程分片并处理 false sharing。

### 项目五：性能实验报告

- 选择真实项目热点。
- 保存复现数据和构建选项。
- 给出 flame graph 或 profiler 证据。
- 至少提出三种假设并逐一验证。
- 记录失败优化和原因。
- 同时验证正确性与性能回归。

### 项目验收标准

- 有优化前基线和明确指标。
- profiler 证明优化目标是真实热点。
- 不依赖 UB 或不可移植假设。
- Debug、Release、ASan、UBSan 均验证。
- 基准设置和环境可复现。
- 结果报告方差而非只给最好一次。
- 性能提升未破坏 API、正确性和可维护性。

## 十七、易错点汇总

1. 把已分配存储当成已构造对象。
2. placement new 后遗漏销毁。
3. 用 memcmp 比较任意对象。
4. 用 reinterpret_cast 绕过类型系统并直接解引用。
5. 忽略对齐和别名规则。
6. 依赖有符号溢出回绕。
7. 直接将结构体字节写入跨平台文件。
8. 只在 Debug 模式测性能。
9. 从短汇编片段推断完整系统性能。
10. 假设 O3 必定快于 O2。
11. 用链表降低理论插入复杂度却恶化缓存。
12. 没有访问模式证据就改为 SoA。
13. 把所有虚函数替换成复杂模板。
14. 使用 likely/unlikely 却没有 profile。
15. 盲目启用 fast-math。
16. 手写 SIMD 却遗漏尾部和不同 ISA。
17. 自定义 allocator 协议不完整。
18. 对象池保留脏状态或悬空句柄。
19. 微基准计算被优化删除。
20. 优化非热点并牺牲可维护性。
