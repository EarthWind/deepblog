## 一、正确看待 C++26

C++26 功能虽已进入标准工作成果范围，但编译器和标准库支持仍不均衡。学习时把内容分为三层：

1. **模型层**：掌握 Contracts、Reflection、Execution、SIMD 等设计思想。
2. **实验层**：使用最新 GCC/Clang/MSVC 和 feature-test macro 编写隔离示例。
3. **生产层**：只有目标工具链、运行库、构建系统和 CI 全部验证后才采用。

提案论文用于解释设计，不等于可移植实现。准确语法和接口应以最终标准文本、实现文档及功能宏为准。

### C++26 主线

```text
正确性：Contracts、库 hardening
元编程：Reflection、<meta>、pack indexing、结构化绑定包
异步：Execution Control、sender/receiver、task、async scope
容器：inplace_vector、hive、indirect、polymorphic、optional<T&>
数值：simd、linalg、饱和算术、Philox
并发：hazard pointers、RCU、原子 min/max 与 reduction
编译期：constexpr exceptions、placement new、更多 constexpr 容器
数据流：并行 Range 算法、views 扩展、reserve_hint
```

## 二、Contracts

Contracts 用于在接口和实现中表达预期成立的条件，包括函数前置条件、后置条件和断言。

### 2.1 核心思想

- 前置条件：调用者必须保证。
- 后置条件：函数成功完成后实现保证。
- assertion：程序特定位置应成立的性质。
- contract violation：条件未满足时进入标准定义的处理模型。

示意代码（实际语法按所用编译器实现核对）：

```cpp
double divide(double numerator, double denominator)
    pre(denominator != 0.0)
    post(result: std::isfinite(result));
```

### 2.2 `<contracts>`

支持库提供 violation 信息、assertion kind、evaluation semantic、detection mode 和默认处理器调用设施。功能宏可检查 `__cpp_lib_contracts`。

### 2.3 Contracts 不是什么

- 不替代外部输入验证。
- 不替代错误返回或异常。
- 不自动证明程序正确。
- 不应依赖条件表达式副作用。
- 不能假设所有构建配置都以相同方式检查。

### 2.4 迁移方法

先写清 API 前置/后置条件和不变量，再在隔离层采用语法。生产代码仍需为不可信输入保留正常检查。

### 练习

1. 为矩阵索引、排序算法和账户操作列出契约。
2. 区分调用者违约与可预期业务失败。
3. 设计不含副作用的条件表达式。

## 三、静态反射与 `<meta>`

Reflection 允许程序在编译期检查和生成与类型、成员、名称及声明有关的代码，是 C++26 最具结构性影响的能力之一。

### 3.1 学习模型

```text
程序实体 → reflection/meta-info → 编译期查询/变换 → 生成或注入程序结构
```

重点概念：

- 获取实体的反射信息。
- 查询名称、类型、成员和属性。
- 遍历成员反射序列。
- splice/注入被反射实体。
- 编译期生成静态对象、数组或字符串。
- constexpr 异常用于反射诊断。

### 3.2 典型用途

- 自动序列化。
- 枚举名称映射。
- 结构体字段遍历。
- 数据库/协议绑定。
- 测试与调试展示。
- 编译期注册表。

### 3.3 设计边界

- 反射不自动定义序列化格式和版本兼容。
- 私有性、访问控制和注解策略需明确。
- 生成代码会增加编译时间和诊断复杂度。
- ABI、字段顺序和外部协议不能仅依赖当前类布局。
- 反射库支持仍可能部分实现。

### 3.4 实验建议

建立独立 `reflection-lab`，用功能宏包围代码；不要让实验语法扩散到稳定业务层。为反射生成结果编写编译期正反例与运行时黄金测试。

## 四、Pack Indexing 与模板增强

### 4.1 Pack indexing

允许直接按索引选择参数包元素，减少 tuple 辅助、递归模板和 `index_sequence` 样板。

示意：

```cpp
template<class... Types>
using first_type = Types...[0];
```

实际可用语法需按编译器支持核对。

### 4.2 结构化绑定包

结构化绑定可引入参数包，从 tuple-like 数据中分离头部、尾部或中间序列，适合泛型分解和访问者实现。

### 4.3 Fold-expression 约束排序

C++26 改进涉及折叠表达式的约束排序，减少逻辑上“更受约束”但编译器无法正确排序的重载问题。仍应优先复用命名 Concept 的原子约束。

### 4.4 Concept/变量模板作为模板参数

模板可更直接接受 Concept 或变量模板类抽象，适合高阶元编程，但公共接口应控制复杂度。

### 4.5 Placeholder variables

无名占位变量允许明确表示值仅为结构或生命周期需要，而无需人为制造名称。

## 五、编译期能力扩展

### 5.1 constexpr exceptions

常量求值中可使用更自然的异常式错误传播；若异常未在常量求值内处理，表达式不能成为有效常量结果。它改善编译期库内部控制流与诊断，不意味着运行时异常成本消失。

### 5.2 constexpr placement new

帮助容器和底层库在常量求值中管理对象生命周期。普通业务代码仍优先 `construct_at`、容器和标准抽象。

### 5.3 更多 constexpr 容器

队列、优先队列等更多容器和适配器可用于常量求值，但最终能否 constexpr 完成仍取决于分配和对象生命周期是否满足规则。

### 5.4 用户生成的 `static_assert` 消息

断言消息可来自编译期生成的数据，提高泛型诊断质量。消息仍应短、稳定并说明失败要求。

### 5.5 constexpr cast from `void*`

改善类型擦除和分配器设施的常量求值能力，但不能绕过类型、生命周期和别名规则。

### 5.6 `#embed`

在编译时嵌入二进制资源，适合着色器、测试数据和固件资源。需考虑生成物体积、增量编译、资源版本和平台工具链支持。

## 六、Execution Control Library

该库以 sender/receiver 模型表达可组合异步工作。

### 6.1 三个核心概念

- sender：描述最终可能完成的异步操作。
- receiver：接收 value、error 或 stopped 完成信号。
- scheduler：决定工作在哪个执行资源上运行。

```text
sender ──connect──> receiver = operation state
operation state ──start──> execution
```

### 6.2 完成通道

异步操作通常有三类完成：

- value：成功结果。
- error：错误。
- stopped：取消或停止。

它们是类型系统中的显式组成，而不是约定的整数状态码。

### 6.3 组合器

重点理解：

- `just`：创建已知值 sender。
- `then`：转换成功值。
- `let_value`：根据值产生后续 sender。
- `when_all`：组合多个操作。
- error/stopped 恢复。
- execution environment 查询和修改。

准确命名空间和最终接口以实现文档为准。

### 6.4 Structured concurrency

父作用域应知道子任务，完成、错误和取消不会成为无人管理的后台工作。async scope、task 和 parallel scheduler 等设施延伸这一模型。

### 6.5 协程互操作

协程表达暂停/恢复，sender 表达组合和完成协议，scheduler 表达执行位置。三者不是替代关系。

### 6.6 落地难点

- 标准库实现支持较少。
- 模板错误可能复杂。
- 生命周期和 environment 传播需要严格理解。
- 不应把实验 sender 直接替换成熟生产异步库。

### 练习

1. 画出 value/error/stopped 三通道。
2. 用伪代码组合读取、解析、保存操作。
3. 比较 future、协程和 sender 的职责。
4. 为任务树设计取消传播。

## 七、新容器与所有权类型

### 7.1 `std::inplace_vector<T, Capacity>`

固定容量、运行时长度、对象内存储：

- 不发生动态分配。
- 元素连续。
- 容量是类型的一部分。
- 超过容量的操作按接口规定失败或抛出。
- 插入删除仍可能移动元素并使观察位置失效。

适合嵌入式、实时、协议小数组和确定内存预算。

### 7.2 `std::hive`

分块存储并复用擦除槽位，目标是：

- 插入/擦除效率。
- 未擦除元素地址稳定。
- 避免链表每节点分配。

它通常不连续、不支持普通随机访问。适合游戏实体、稳定句柄对象集合等。

### 7.3 `std::indirect` 与 `std::polymorphic`

提供间接存储但面向值语义的包装：复制包装对象可复制底层值，而不是共享身份。`polymorphic` 面向多态值语义。应与 `unique_ptr` 的独占身份语义区分。

### 7.4 `optional<T&>`

表达“可能存在的非拥有引用”。它不延长对象生命周期，仍可能悬空。适合替代部分可空裸指针查询接口，但所有权必须清晰。

### 7.5 选择表

| 需求 | 候选 |
|---|---|
| 动态长度、堆容量 | `vector` |
| 固定上限、对象内存储 | `inplace_vector` |
| 稳定元素地址、频繁擦除 | `hive` |
| 独占身份 | `unique_ptr` |
| 间接值语义 | `indirect` |
| 多态值语义 | `polymorphic` |

## 八、SIMD、线性代数与数值

### 8.1 `<simd>`

标准数据并行类型明确表达 lane 级并行，包括 vector、mask、load/store、选择、数学、位和 permutation 操作。

```cpp
// 示意，具体类型名按实现核对
std::simd::native_simd<float> values;
auto result = values * scale + offset;
```

### 8.2 使用原则

- 保留标量正确基线。
- 处理对齐、尾部和 mask。
- 不假设固定 lane 数。
- 检查向量化汇编和性能。
- 验证浮点误差、NaN 和 signed zero。

### 8.3 `<linalg>`

提供基于 mdspan 等抽象的 BLAS 风格自由函数接口，用于向量/矩阵运算。接口不等于某个特定高性能后端；实现、布局、执行策略和数值语义仍需测量。

### 8.4 饱和算术

结果超出范围时钳制到最小/最大值，适合音视频、图像和信号处理。它与模运算和溢出检查是不同语义。

### 8.5 Philox 随机引擎

计数器型随机引擎适合并行、可复现流分割。统计质量不等于密码安全，不应用于密钥或安全令牌。

### 8.6 mdspan 改进

包括 padded layout、`at()` 和更便利推导等方向。算法不能无条件假设连续布局。

## 九、并发内存回收

### 9.1 Hazard Pointers

线程在访问可能被并发删除的节点前发布 hazard；删除者只有确认没有 hazard 指向节点后才回收。

核心流程：保护、验证、使用、解除保护、退休、扫描回收。

### 9.2 RCU

读者在 read-side critical section 中读取旧版本；更新者发布新版本，并等待 grace period 后回收旧版本。适合读多写少，但更新、回收和线程注册协议必须严格。

### 9.3 边界

- 标准设施降低接口碎片，不会让无锁算法自动正确。
- 仍需证明内存序、节点状态和回收协议。
- TSan 不能证明 lock-free 正确性。
- 优先成熟实现和锁版本基准。

### 9.4 原子扩展

C++26 包括原子 min/max、浮点 min/max、reduction 等方向，并弱化/弃用 `memory_order::consume` 的旧模型。每个弱内存序仍需明确同步证明。

## 十、Ranges 与算法扩展

重点包括：

- 并行 Range 算法。
- `views::concat`、`views::indices`。
- `as_input_view`。
- `approximately_sized_range`。
- `reserve_hint`。
- 随机生成 Range 算法。
- filter view 安全性改进。

`reserve_hint` 允许惰性范围向目标容器提供近似容量信息，但提示不等于精确大小。并行算法要求函数对象、元素访问和异常策略满足并发要求。

## 十一、文本、诊断与库完善

- `std::text_encoding`：识别和描述文本编码，不等于完整转码库。
- `<debugging>`：调试器存在性等支持设施。
- constexpr 文本格式化。
- filesystem path 的显示/原生编码字符串。
- `exception_ptr_cast`。
- `variant::visit` 成员形式。
- 更多 freestanding 标准库。
- 标准库 hardening。
- `<stdbit.h>`、`<stdckdint.h>` 提供 C 兼容位和检查算术设施。

旧设施也有移除或弃用，例如 strstream、wstring_convert 和旧 shared_ptr 原子自由函数。迁移时应先消除弃用警告。

## 十三、阶段项目

### 项目一：契约与 hardening 实验库

- 为固定容量容器定义前置/后置条件。
- 外部输入仍正常验证。
- 比较 Debug 断言、Contracts 和 expected。

- 编译期遍历字段。
- 支持字段过滤和名称映射。
- 格式版本独立于内存布局。
- 私有字段和未知字段策略明确。
- 与手写实现比较编译时间和诊断。

### 项目三：Sender 数据管道

- value/error/stopped 三通道。
- 组合读取、解析和保存。
- scheduler 明确。
- 支持协作取消和 async scope。
- 与 future/协程版本比较。

### 项目四：固定预算实时组件

- 使用 inplace_vector。
- 禁止运行阶段动态分配。
- 容量失败语义明确。
- 基准与最大内存预算验证。

### 项目五：SIMD 图像处理

- 标量基线。
- simd vector/mask 实现。
- 正确处理尾部。
- 比较不同 ISA 和数据布局。
- 验证数值误差和性能。

### 项目六：安全回收实验

- 先实现 mutex 基线。
- 使用 hazard pointer 或 RCU 实验。
- 写出内存序和回收证明。
- 压力测试、TSan 与性能比较。
- 明确实验代码不自动具备生产资格。

### 验收标准

- 每项功能有 feature-test 和支持矩阵。
- 不把提案草案当稳定 API。
- Contracts 不替代外部验证。
- 反射生成格式有版本策略。
- sender 生命周期和完成通道明确。
- SIMD 有标量基线与尾部测试。
- 无锁回收有协议证明。
- fallback 和原生路径均通过 CI。

## 十六、易错点汇总

1. 把论文中的语法当作最终实现语法。
2. 只看 `-std=c++26` 就假设标准库完整。
3. 用 Contracts 检查普通用户输入。
4. 契约条件带副作用。
5. 用反射直接序列化内存布局。
6. 反射生成代码导致编译时间失控。
7. 把 sender 当作已运行任务。
8. 忽略 stopped 通道和取消传播。
9. 把协程、scheduler 和 sender 混为一谈。
10. inplace_vector 超容量策略不明确。
11. 假设 hive 连续或随机访问。
12. optional<T&> 比被引用对象活得更久。
13. SIMD 忽略尾部、mask 或浮点差异。
14. 把 Philox 用作密码学随机源。
15. 无锁节点退休后立即释放。

17. 并行 Range 谓词访问未同步状态。
