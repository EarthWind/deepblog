## 二、显式对象参数（Deducing `this`）

### 2.1 基本语法

```cpp
struct Value
{
    int data{};

    void print(this const Value& self)
    {
        std::println("{}", self.data);
    }
};
```

显式对象成员函数没有隐式 `this`，对象作为第一个参数出现。

### 2.2 合并 cv/ref 重载

旧式接口：

```cpp
T& value() &;
const T& value() const&;
T&& value() &&;
const T&& value() const&&;
```

C++23 可写：

```cpp
template<class Self>
decltype(auto) value(this Self&& self)
{
    return std::forward_like<Self>(self.value_);
}
```

`std::forward_like` 按另一个类型的 cv/ref 限定转发表达式，适合显式对象参数。

### 2.3 CRTP 的部分替代

```cpp
struct Printable
{
    void print(this const auto& self)
    {
        self.print_impl();
    }
};
```

派生类型可继承该接口，`self` 推导为实际对象类型，减少部分 CRTP 样板。它不等于运行时多态，也不会生成虚分派。

### 2.4 递归 Lambda

```cpp
auto factorial = [](this auto self, unsigned int value) -> unsigned long long {
    return value <= 1 ? 1 : value * self(value - 1);
};
```

无需额外 `std::function` 或把 Lambda 自身作为普通参数传入。

### 2.5 限制

- 显式对象成员不能是 `static` 或 `virtual`。
- 函数声明不再在末尾写 cv/ref 限定。
- `self` 可能按值声明，但复制对象语义和成本必须明确。
- 转发返回值时仍需严格处理生命周期。

### 练习

1. 用显式对象参数合并容器访问器重载。
2. 用 `forward_like` 保留成员 cv/ref。
3. 将一个简单 CRTP mixin 改为显式对象参数。
4. 实现递归树遍历 Lambda。

## 三、`if consteval` 与编译期改进

### 3.1 `if consteval`

```cpp
constexpr int compute(int value)
{
    if consteval {
        return compile_time_algorithm(value);
    } else {
        return runtime_algorithm(value);
    }
}
```

它直接表达当前调用是否处在立即函数上下文，比误用 `std::is_constant_evaluated()` 更清晰。

### 3.2 `if !consteval`

```cpp
if !consteval {
    record_runtime_metrics();
}
```

两个分支的外部可观察语义应保持一致，除非接口明确说明差异。

### 3.3 constexpr 扩展

C++23 放宽多项常量求值限制，并使更多标准库组件可 constexpr 使用。工具链支持可能分阶段落地，应通过功能宏检测。

### 3.4 `static operator()` 与 `static operator[]`

无状态函数对象可声明静态调用运算符，避免隐式对象参数：

```cpp
struct Add
{
    static int operator()(int left, int right)
    {
        return left + right;
    }
};
```

### 3.5 多维下标

```cpp
class Matrix
{
public:
    double& operator[](std::size_t row, std::size_t column);
};

matrix[2, 3] = 1.0;
```

需要与旧代码中的逗号表达式警告和兼容性区分。

### 3.6 `auto(x)` 与 `auto{x}`

显式按值复制表达式并进行类似 `auto` 推导：

```cpp
auto copy = auto(reference);
```

在泛型代码中可用于明确获得值，而不是保留引用。

## 四、简化隐式移动与返回

C++23 简化返回局部对象时的隐式移动规则，使一些曾需两阶段重载处理的场景更一致。

```cpp
Widget make_widget()
{
    Widget result;
    return result;
}
```

仍然不要写：

```cpp
return std::move(result);
```

这可能阻碍 NRVO。直接返回局部对象，让语言选择复制消除或移动。

对返回引用、const 对象和特殊转换构造仍需按实际规则分析，不能把“隐式移动”理解为所有返回都自动搬走资源。

## 五、`std::expected`

### 5.1 显式成功或错误

```cpp
#include <expected>

enum class ParseError
{
    empty,
    invalid,
    out_of_range
};

std::expected<int, ParseError> parse_integer(std::string_view text);
```

使用：

```cpp
auto result = parse_integer(input);
if (result) {
    std::println("value = {}", *result);
} else {
    handle(result.error());
}
```

### 5.2 构造错误

```cpp
return std::unexpected{ParseError::invalid};
```

错误类型应能稳定分类，必要时附带位置和上下文，不应迫使调用者解析字符串。

### 5.3 monadic 操作

```cpp
auto result = read_text(path)
    .and_then(parse_config)
    .transform(normalize_config)
    .or_else(report_error);
```

- `and_then`：成功时调用返回 expected 的函数。
- `transform`：成功值映射为另一个值。
- `or_else`：错误时处理或转换。
- `transform_error`：转换错误类型。

操作函数的返回类型和引用生命周期必须符合接口要求。

### 5.4 `expected<void, E>`

表示操作成功但无结果值，或失败并携带错误：

```cpp
std::expected<void, SaveError> save(const Document& document);
```

### 5.5 与异常的选择

expected 适合频繁、可预期且调用者通常处理的失败；异常适合无法建立对象或需要跨多层传播的非常规失败。项目应建立一致策略，而不是逐函数随意选择。

### 练习

1. 使用 expected 实现整数和配置解析器。
2. 建立三步 monadic 数据管道。
3. 使用 `expected<void, E>` 表示保存结果。
4. 比较 expected 与异常版 API 的调用代码。

## 六、`std::print` 与格式化

### 6.1 基本输出

```cpp
#include <print>

std::print("processing {}...", filename);
std::println("done: {} records", count);
```

`println` 自动添加换行，格式语法沿用 `std::format`。

### 6.2 类型安全

编译期格式字符串可在编译期检查参数类型和占位符。动态格式字符串需使用相应运行时格式机制，不能把任意用户输入直接当编译期格式字符串。

### 6.3 输出目标

可向标准输出或文件流相关目标输出，具体重载和 Unicode/终端行为需按实现与平台验证。

### 6.4 自定义 formatter

用户类型可特化 `std::formatter<UserType, CharT>`。应保持解析和格式化职责清晰，避免 formatter 修改对象或依赖全局状态。

### 6.5 错误输出

日志系统仍需级别、结构化字段、同步和敏感信息控制。`print` 是输出设施，不是完整日志框架。

## 七、`std::generator`

### 7.1 惰性序列

```cpp
#include <generator>

std::generator<int> sequence(int count)
{
    for (int value{}; value < count; ++value) {
        co_yield value;
    }
}
```

使用：

```cpp
for (int value : sequence(5)) {
    std::println("{}", value);
}
```

### 7.2 生命周期

generator 对象管理协程 frame。引用参数和 `co_yield` 引用值必须保证底层对象在消费期间有效。

### 7.3 惰性与副作用

函数调用不一定立即执行协程体；遍历时才恢复。提前停止遍历会销毁 frame，局部 RAII 对象应正确清理。

### 7.4 递归展开

标准 generator 支持嵌套序列的相关能力，但写法和工具链支持需查询当前实现。避免为简单容器返回 generator，因为容器本身已是 Range。

### 7.5 generator 不等于异步 task

它主要是同步惰性生成器，不提供线程切换、异步 I/O、调度或取消系统。

## 八、`std::mdspan`

### 8.1 多维非拥有视图

```cpp
#include <mdspan>

std::vector<double> storage(rows * columns);
std::mdspan matrix{storage.data(), rows, columns};
matrix[2, 3] = 4.5;
```

mdspan 不拥有数据，底层存储必须在视图使用期间有效。

### 8.2 核心组成

- element type。
- extents：静态或动态维度。
- layout mapping：索引到偏移的映射。
- accessor：元素访问策略。

### 8.3 布局

- `layout_right`：最后一个索引连续，类似 C 行主序。
- `layout_left`：第一个索引连续，常见于 Fortran/列主序环境。
- `layout_stride`：自定义步长。

算法不能在未约束时假设所有 mdspan 连续或使用同一布局。

### 8.4 静态维度

```cpp
using Matrix4 = std::mdspan<double, std::extents<std::size_t, 4, 4>>;
```

静态维度进入类型，有利于编译期检查和优化；动态维度更灵活。

### 8.5 与 span 的区别

- span：一维连续视图。
- mdspan：多维索引视图，布局不一定连续。

### 练习

1. 用 mdspan 实现矩阵转置读取。
2. 比较 layout_left 和 layout_right 遍历性能。
3. 为静态 4×4 和动态矩阵设计接口。
4. 找出底层 vector 扩容导致 mdspan 失效的问题。

## 九、诊断与底层工具

### 9.1 `std::stacktrace`

```cpp
#include <stacktrace>

auto trace = std::stacktrace::current();
std::println("{}", trace);
```

适合断言、错误边界和诊断报告。符号名称和源码位置依赖调试信息、链接器、平台和部署环境。

不要把堆栈文本当作稳定机器接口，也不要在高频热路径无测量地采集。

### 9.2 `std::unreachable`

```cpp
switch (state) {
case State::ready: return 1;
case State::done: return 2;
}
std::unreachable();
```

执行到 `unreachable` 属于未定义行为。它是给编译器的强承诺，不是断言或错误处理。外部输入可能触达的路径绝不能使用它代替检查。

### 9.3 `std::byteswap`

```cpp
auto network_order = std::byteswap(value);
```

用于整数端序转换，不能直接对任意对象进行跨平台序列化。

### 9.4 `std::to_underlying`

```cpp
auto raw = std::to_underlying(color);
```

安全、清晰地取得枚举底层值。

## 十、容器与可调用对象

### 10.1 `flat_map` 与 `flat_set`

提供有序关联接口，底层通常使用连续容器：

- 查询缓存局部性好。
- 批量构建后查询多的场景有优势。
- 中间插入删除是线性移动。
- 迭代器失效不同于节点式 map/set。

默认容器选择仍需基准和修改模式分析。

### 10.2 `std::move_only_function`

```cpp
std::move_only_function<void()> task =
    [resource = std::make_unique<Resource>()] {
        resource->run();
    };
```

支持 move-only 可调用对象，适合任务队列。它仍有类型擦除和间接调用成本。

### 10.3 容器 range 接口

常见新增：

- `assign_range`。
- `insert_range`。
- `append_range`。
- `prepend_range`。
- `from_range` 构造。

它们直接接受 Range，减少手写 begin/end，但仍需关注单遍 Range、别名、自插入、容量和失效规则。

### 10.4 `string::contains`

```cpp
if (text.contains("error")) { }
```

表达包含意图，比与 `npos` 比较更直观。

## 十一、Ranges 扩展

### 11.1 `ranges::to`

```cpp
auto result = source
    | std::views::filter(predicate)
    | std::views::transform(transformer)
    | std::ranges::to<std::vector>();
```

用于将惰性 Range 物化为容器。

### 11.2 `zip` 与 `enumerate`

```cpp
for (auto [name, score] : std::views::zip(names, scores)) {
    // ...
}

for (auto [index, value] : std::views::enumerate(values)) {
    // ...
}
```

zip 通常由最短输入范围决定结束，不能假设长度相同；若业务要求相同应显式验证。

### 11.3 窗口和分块

- `chunk(n)`：不重叠分块。
- `slide(n)`：滑动窗口。
- `stride(n)`：按步长采样。
- `chunk_by(predicate)`：按相邻关系分组。
- `adjacent<N>`：相邻 N 元组。

### 11.4 组合视图

- `repeat`。
- `cartesian_product`。
- `join_with`。
- `zip_transform`。
- `adjacent_transform`。

笛卡尔积可能组合爆炸，惰性只减少存储，不减少最终遍历的元素数量。

### 11.5 `ranges::fold_left/right`

折叠算法表达从左或从右聚合。对非结合运算和浮点计算，方向会影响结果。

### 11.6 生命周期与代理引用

zip、transform 等 View 的元素可能是 tuple-like 代理或临时值。不要无条件保存其引用；先检查 range reference 类型和底层所有者生命周期。

## 十二、智能指针与 C API 互操作

### 12.1 `out_ptr`

传统 C API：

```cpp
int create_resource(Resource** output);
```

C++23 适配：

```cpp
std::unique_ptr<Resource, Deleter> resource;
create_resource(std::out_ptr(resource));
```

### 12.2 `inout_ptr`

用于 API 可能接收现有指针并替换它的情况。必须理解底层函数在成功和失败时是否释放旧资源，否则可能重复释放或泄漏。

### 12.3 使用边界

- 删除器必须匹配 C API 释放函数。
- 明确 API 是纯输出还是输入输出。
- 检查失败时输出参数的状态。
- 若 API 语义复杂，手写小型 RAII 适配器可能更清晰。

## 十三、其他重要库改进

- `optional` monadic 操作：`and_then`、`transform`、`or_else`。
- `basic_string::resize_and_overwrite`：允许在受控缓冲区中生成字符串。
- `spanstream`：在外部字符缓冲区上进行流操作。
- `std::bind_back` 的相关方向需按具体标准版本核对，不应混入未正式属于 C++23 的功能。
- `std::start_lifetime_as`：显式开始隐式生命周期类型。
- `std::unreachable_sentinel`：辅助某些无限范围/算法接口。
- `std::ranges` 算法结果和 const iterator 设施扩展。

面对长特性列表，应按项目价值排序，而不是逐项强制使用。

## 十四、编译器支持与迁移

### 14.1 功能测试宏

```cpp
#include <version>

#if defined(__cpp_lib_expected)
    // 使用 std::expected
#endif
```

常见宏包括：

- `__cpp_explicit_this_parameter`。
- `__cpp_if_consteval`。
- `__cpp_lib_expected`。
- `__cpp_lib_print`。
- `__cpp_lib_generator`。
- `__cpp_lib_mdspan`。
- `__cpp_lib_stacktrace`。
- `__cpp_lib_ranges_to_container`。

宏的准确阈值应查询当前文档和工具链，不凭记忆填写。

### 14.2 支持层次

需要分别检查：

1. 编译器前端接受语法。
2. 标准库提供头文件和类型。
3. 实现行为完整。
4. 链接或运行时依赖可部署。
5. CMake 和 IDE 能正确识别。

stacktrace、generator、print、时区等设施可能受额外运行库或平台能力影响。

### 14.3 渐进迁移顺序

1. 开启 C++23 模式并保持测试通过。
2. 优先引入 `expected`、`print`、`to_underlying` 等局部设施。
3. 引入 Ranges 新视图改善明确的数据流。
4. 为任务队列采用 move-only callable。
5. 在数值模块试点 mdspan。
6. 显式对象参数和 generator 等按编译器矩阵试点。

### 14.4 fallback

若需支持旧工具链：

- 在适配层集中 polyfill 或第三方替代。
- 不让条件宏散落业务代码。
- CI 同时覆盖原生实现和 fallback。
- 文档明确行为差异。

## 十七、阶段自测

### 概念题

1. 显式对象参数如何减少 cv/ref 重载？

## 十八、易错点汇总

1. 显式对象访问器错误返回悬空引用。
2. 认为显式对象参数提供运行时多态。
3. if consteval 两条路径语义不一致。
4. 返回局部对象时多余使用 std::move。
5. expected 错误只存不可分类字符串。
6. monadic Lambda 返回类型不符合协议。
7. 把用户输入直接作为格式字符串。
8. generator 引用局部或已销毁外部数据。
9. 把 generator 当作异步调度器。
10. vector 扩容后继续使用旧 mdspan。
11. 假设所有 mdspan 都是行主序连续数据。
12. 在外部可达路径调用 unreachable。
13. 认为 flat_map 插入也是对数复杂度。
14. zip 输入长度不一致却未验证。
15. 保存 View 的代理引用。
16. out_ptr 删除器与 C API 不匹配。
17. 只检查编译器版本而忽略标准库实现。
18. fallback 宏散落整个业务代码。
