# PG 是 Ceph 的灵魂：为什么不是直接把对象映射到 OSD

## 这篇文章要解决什么问题

上一篇我们已经把 `CRUSH` 讲清楚了：

- Ceph 不依赖一个每次都要访问的中心元数据定位服务
- 它通过 `CRUSH map + rule + failure domain` 来计算数据应该落在哪些 OSD 上

但如果你继续往前追一步，很快会发现一个更关键的问题：

**既然 CRUSH 已经能算位置，Ceph 为什么不直接把对象映射到 OSD，而是还要在中间插一个 `PG`？**

这是理解 Ceph 的真正分水岭。

很多初学者会把 `PG` 看成：

- 一个实现细节
- 一个对象分组技巧
- 一个为了“均匀一点”而加上的中间层

这些理解都抓到了一部分，但都不够深。

如果只允许用一句话先给结论，那就是：

**PG 不是 Ceph 多出来的一层负担，而是 Ceph 用来同时解决扩展性、数据放置、一致性协商和故障恢复的核心抽象。**

这一篇的目标，就是把这句话真正展开。

## 先建立第一条边界：PG 不是“对象目录”，而是“状态机实例”

第一次理解 `PG` 时，最容易产生一个误区：

- 觉得 PG 只是把一堆对象简单分到同一个桶里

这当然不完全错，但如果只停留在这个层面，你后面会一直理解不透：

- peering 为什么按 PG 做
- recovery 为什么按 PG 做
- primary 为什么是 PG 级别的
- scrub 为什么也是 PG 级别的

所以一开始就要把边界拉高一点：

### PG 确实是对象分桶

也就是：

- 一批对象先归到同一个 PG

### 但 PG 远远不只是对象分桶

它还是：

- placement 单元
- primary 协调单元
- peering 单元
- recovery/backfill 单元
- scrub 单元

也就是说，Ceph 不只是拿 PG 来“归类对象”，而是进一步把：

- 放置
- 状态
- 协调
- 修复

这些系统行为都绑定到 PG 上。

这也是为什么很多人说：

- PG 是 Ceph 的灵魂

## 为什么这个问题这么关键：如果没有 PG，Ceph 也许还能“存”，但很难“稳定地存”

这是理解 PG 的最好视角。

从纯数学定位角度说，Ceph 当然可以尝试：

- 对象名直接 hash
- 直接跑 CRUSH
- 直接得到 OSD 集合

也就是说，单从“算出这份对象放哪”这件事上看，**对象直映 OSD 在数学上不是不可能**。

真正的问题在于：

- Ceph 要解决的不是“能算位置”这么简单

它还要解决：

- 海量对象的管理规模
- 扩容缩容时的数据迁移组织
- 副本一致性收敛
- 故障恢复的边界与顺序
- 后台校验与长期维护

而一旦这些事情也考虑进来，直接对象映射 OSD 的代价就会变得非常高。

所以理解 PG 的正确方式不是：

- 为什么 Ceph 不直接做对象到 OSD 的计算

而是：

- 为什么 Ceph 不愿意把所有运行时状态和修复逻辑都降到对象粒度

答案就是：

- 成本太高，而且几乎不可控

## 第 1 层：为什么不能直接“对象 -> OSD”

这一层是全文最重要的问题。

先直接把结论摆出来：

**如果 Ceph 直接按对象管理 placement、一致性和恢复，系统会在四件事上同时失控：状态规模、重平衡粒度、故障恢复组织、写入顺序协商。**

下面逐条展开。

## 原因一：对象数量太大，不能按对象维护完整运行时状态

Ceph 官方文档里对这一点说得非常直接：

- 按单对象管理 placement 和元数据，面对数百万甚至更多对象时，计算和维护成本会太高

这句话非常关键，因为它点出了 PG 的第一性价值：

- **压缩状态规模**

### 为什么对象粒度会爆炸

因为如果你真的按对象管理，你就可能要为每个对象维护或频繁推导：

- 当前放置位置
- 副本状态
- 版本边界
- 缺失信息
- 恢复进度
- 日志边界

当对象规模上升到百万、千万甚至更多时，这种模型几乎必然会把：

- 内存
- CPU
- 网络状态交换

一起拉爆。

而 PG 的作用就是把海量对象压缩成有限数量的“状态机实例”。

所以从最根本上说：

- **PG 首先是一个扩展性抽象**

## 原因二：如果直接对象映射 OSD，重平衡会碎成海量微迁移

Ceph 的设计目标之一，是在拓扑变化时：

- 只迁移必要的数据
- 而不是全量重排

如果你直接按对象粒度管理，那扩容、缩容、设备变化带来的重平衡，会变成：

- 海量对象级碎片迁移

这会带来两个问题：

### 1. 控制面和数据面都很难组织

你需要面对太多独立迁移动作。

### 2. 恢复和重平衡的“边界”非常模糊

因为几乎每个对象都可能是一个独立小任务。

而有了 PG 之后，Ceph 可以把迁移和重平衡组织成：

- 一组 PG 的移动与修复

这并不意味着对象层面不存在迁移，而是意味着：

- 系统的调度边界和状态边界被稳定在 PG 上

所以 PG 不是为了省一个 hash 步骤，而是为了给系统一个：

- 可组织的重平衡粒度

## 原因三：一致性和恢复不能按海量对象级小事务来收敛

这是 PG 真正“灵魂级”的地方。

Ceph 的写入不是单机 write，而是分布式副本写。也就是说，系统必须回答：

- 哪个副本是当前 primary
- 哪些写已经被认为在当前 authoritative history 中
- 哪些对象版本缺失
- 哪些副本需要补数据

如果这些事情都按对象独立处理，理论上可以，但工程上会变成：

- 海量小状态机
- 海量小日志边界
- 海量小恢复事务

这会让系统极其难以收敛。

而 Ceph 的选择是：

- 把一批对象放进一个 PG
- 然后对这个 PG 维护统一的 `PGInfo`、`PGLog`、Missing Set、past intervals、acting/up 信息

也就是说，PG 的第二层核心价值是：

- **为一致性和恢复提供统一边界**

## 原因四：primary 协调如果按对象粒度做，调度和顺序控制会太昂贵

Ceph 的普通副本写路径本质上是：

- 客户端写 primary
- primary 再协调 replicas

这意味着系统必须有一个比较稳定的协调单位。

如果 primary 是按对象级别频繁散落协调，那么你需要面对：

- 更多粒度的排序
- 更多粒度的状态切换
- 更多粒度的 leader-like 语义

而 Ceph 的工程取舍很明确：

- 把 primary 协调收敛到 PG 级别

于是系统就能用更少的状态描述：

- 这一批对象当前由谁负责
- 这一轮历史以谁为准
- 这一组副本如何恢复

所以 PG 的第三层价值，是：

- **把主副本协调从对象粒度提升到可管理粒度**

## 第 2 层：PG 到底是什么

讲完“为什么必须有 PG”，现在再回头回答：

- PG 到底是什么？

最准确的理解不是单选题，而是一个四合一答案：

## 1. PG 是对象分桶单元

对象先通过 hash 和 pool 参数落到 PG。

这是最基础的一层。

## 2. PG 是放置单元

CRUSH 不是直接吃裸对象，而是先以 PG 为输入，再算 `up set` / `acting set`。

## 3. PG 是一致性协商单元

peering、authoritative history、`up_thru`、日志边界，都是围绕 PG 展开。

## 4. PG 是恢复与后台维护单元

recovery、backfill、scrub 都是以 PG 为基本边界推进。

如果要把这四层压缩成一句话：

**PG 是 Ceph 把“数据分布”和“数据治理”绑定到一起的核心中间层。**

## 第 3 层：对象、PG、CRUSH、OSD 到底怎么分工

这一层特别适合和上一篇 CRUSH 做衔接。

先给出一张最简分工图：

```text
对象
  -> 先归到 PG
PG
  -> 再交给 CRUSH 做放置计算
CRUSH
  -> 计算这个 PG 理论上应该落到哪些 OSD
OSD
  -> 对这个 PG 执行真正的读写、复制、恢复和校验
```

这张图最重要的结论是：

- 对象不是直接面向 OSD 管理的
- OSD 也不是直接以“单对象集合”作为最核心治理粒度

Ceph 的主逻辑真正围绕的是：

- PG

所以你可以把上一篇和这一篇连起来记：

- `CRUSH` 负责算位置
- `PG` 负责把这份位置变成可治理的系统边界

## 第 4 层：对象如何进入 PG

从源码主线看，对象到 PG 的关键路径大致会经过：

- `hash_key()`
- `ceph_str_hash()`
- `OSDMap::map_to_pg()`
- `OSDMap::object_locator_to_pg()`

### 这条路径在做什么

本质上就是：

- 取对象名或 locator key
- 做对象 hash
- 得到 raw `pg_t`

再由 pool 的 PG 相关参数，把它归一化到当前实际 PG 空间里。

### 为什么这一步很重要

因为它说明：

- PG 不是运行时随便分组出来的
- 它是对象定位链的第一段正式结果

也就是说，在 Ceph 里，对象一进入系统定位主线，几乎立刻就被提升到了：

- PG 语义

这本身就能说明 PG 的中心地位。

## 第 5 层：为什么 primary 是 PG 级别的，而不是对象级别的

Ceph 文档里对 primary 的定义非常清楚：

- primary 是某个 PG 的 acting set 中的第一成员
- 客户端只把请求发给 primary

这点非常关键，因为它说明：

- primary 不属于某个单独对象
- primary 属于一个 PG

### 这意味着什么

意味着对于这个 PG 内的一批对象，系统把：

- 写入顺序入口
- 副本协调入口
- 恢复协调入口

都收敛到了一个稳定角色上。

这对分布式存储非常重要，因为你必须有一个足够稳定的协调单位，才能让：

- 客户端写
- 副本写
- 故障切换
- 历史收敛

这些事情不至于完全碎片化。

所以 primary 按 PG 建立，本质上也是 PG 价值的一部分。

## 第 6 层：Peering 为什么必须按 PG 做

如果说理解 PG 最关键的入口之一，那一定是：

- peering

因为 peering 会直接告诉你，PG 为什么绝对不是“一个方便分类对象的桶”。

文档对 peering 的定义非常值得反复咀嚼：

- peering 是让存放某个 PG 的 OSD 们，就这个 PG 里的对象和元数据状态达成一致

### 这句话里最重要的词是什么

不是“对象”，而是：

- “某个 PG”

这说明 Ceph 在一致性协商上采取的不是：

- 每个对象自己跑一套小型协商

而是：

- 以 PG 为状态协商边界

### 为什么这很重要

因为 peering 要解决的问题不是简单“谁在线”，而是：

- 当前 authoritative history 是谁
- 哪些日志是可信的
- 哪些副本缺什么
- 当前 acting 该怎么形成

如果这些事情降到对象级别，状态量和消息量都会非常难以控制。

而 PG 刚好提供了一个足够小、又足够可管理的边界。

所以：

- **PG 是 peering 能成立的前提之一**

## 第 7 层：Peering 先统一历史，再决定能不能 active

理解 PG 时，还有一个非常重要的直觉：

- peering 不等于 recovery

这两件事虽然紧密相关，但不是一回事。

### peering 主要解决什么

- 先把这轮 PG 的历史和状态边界统一起来
- 确认谁是 primary
- 确认 `up set` / `acting set`
- 决定是否已经具备 active 条件

### recovery 主要解决什么

- 在 authoritative history 已经选定之后
- 再去补齐缺失对象和副本内容

这是一条非常重要的分层：

```text
peering: 先统一“该听谁的”
recovery: 再补齐“缺了什么”
```

而这两者能被自然接起来，正是因为它们共享同一个基本治理边界：

- PG

## 第 8 层：`up set` 和 `acting set` 让 PG 成为“理想位置”和“真实位置”的连接点

上一篇我们已经讲过：

- `CRUSH` 更像是给出理想放置结果

但集群真实运行时，还会有：

- 故障
- down OSD
- `pg_temp`
- peering 调整

这就带来了：

- `up set`
- `acting set`

两个不同但密切相关的概念。

### 为什么它们和 PG 深度绑定

因为 Ceph 不是在对象级维护：

- 这个对象现在理论位置和实际位置怎么偏离

而是把这个问题收敛到了：

- 这个 PG 现在理论上应去哪
- 实际上暂时由谁来承载

这再次说明 PG 的核心地位：

- 它不是只把对象装进桶里
- 它还把“理想放置”和“当前实际承载”连接起来

## 第 9 层：`PeeringState` 为什么是理解 PG 的源码核心

如果你真要从源码层面抓 PG 的灵魂，最值得看的对象之一就是：

- `PeeringState`

### 为什么不是只看 `PG` 类

因为 `PG` 更像宿主和执行体，而：

- `PeeringState`

把 PG 的状态推进逻辑真正拆成了一棵状态树。

这棵树里你会看到：

- `Initial`
- `Reset`
- `Started`
- `Primary`
- `Peering`
- `Active`
- `Recovering`
- `Backfilling`
- `Clean`
- `Recovered`

### 这意味着什么

意味着 PG 不是静态容器，而是一个真正运行中的状态机实例。

这也正是本文最想让读者留下的认知之一：

- **Ceph 真正运行中的最小治理单元，不是对象，而是 PG 状态机。**

## 第 10 层：为什么 recovery 也一定要按 PG 做

如果 peering 已经说明 PG 是一致性协商边界，那么 recovery 则说明：

- PG 还是故障修复边界

从代码主线上看，恢复会走类似：

- `OSD::do_recovery()`
- `pg->start_recovery_ops()`

然后在 `PrimaryLogPG` 里展开：

- `recover_replicas()`
- `recover_primary()`
- `recover_backfill()`

### 为什么不能按对象自己各补各的

因为恢复并不只是“拷贝对象内容”这么简单，它还涉及：

- 当前 PG 的历史边界
- missing 集合
- 哪些副本该参与
- 哪些对象需要日志恢复
- 哪些对象需要 backfill

这些问题都天然更适合围绕 PG 统一决策。

所以 recovery 之所以可控，也是因为：

- 系统不是在操作海量无组织对象
- 而是在操作有限数量的 PG 状态机

## 第 11 层：Backfill 和 Scrub 继续证明 PG 不是“多余中间层”

如果你对 PG 的价值还不够有感觉，那再看两件事就会更明显：

- backfill
- scrub

### Backfill 说明什么

说明当日志恢复不够时，Ceph 还需要按 PG 组织更大粒度的数据集合对齐。

### Scrub 说明什么

说明长期数据一致性校验，也不是按海量对象随缘跑，而是按 PG 作为管理边界推进。

这进一步证明：

- PG 不是只服务于定位
- 它服务于整个数据生命周期治理

从写入、peering，到恢复、回填、校验，PG 都在中间。

## 第 12 层：为什么 PG 数量不是越多越好

讲到这里，很容易产生另一个误解：

- 既然 PG 这么重要，那是不是越多越好？

答案当然不是。

因为 PG 带来的好处和代价是同时存在的。

### PG 变多的好处

- 数据分布更细
- 负载更均匀
- 并行度更高
- 重平衡粒度更细

### PG 变多的代价

- 更多内存消耗
- 更多 peering 状态机实例
- 更多恢复与统计开销
- 更多 monitor / OSD 管理压力

这也是为什么 Ceph 才需要：

- `pg_num`
- `pgp_num`
- autoscaler

也就是说，PG 是一个非常成功的抽象，但它仍然需要：

- 工程上的数量权衡

这也是 PG 之所以成熟的地方：

- 它不是“越多越完美”
- 而是在可管理数量级内，把系统复杂度压缩到可以承受的程度

## 用一句话重新定义 PG

如果把这篇全部内容压缩成一句尽可能准确的话，我会这样定义：

**PG 是 Ceph 为海量对象系统设计的“可管理状态机单元”：它把对象聚合成有限数量的分布、协调、恢复和校验边界，再通过 CRUSH 和 OSD 把这些边界落到真实存储集群中。**

这句话虽然长，但几乎把 PG 的全部价值都包含进去了。

## 把 PG 压缩成一条最短主线

如果你想把这一篇记成最短结构，可以记成下面这样：

```text
对象很多，不能按对象管理一切
  ->
先把对象归到 PG
  ->
PG 再交给 CRUSH 算 OSD
  ->
每个 PG 建立 primary / replica 关系
  ->
peering 先统一这个 PG 的历史和状态
  ->
recovery / backfill 补齐这个 PG 的副本内容
  ->
scrub 按 PG 校验长期一致性
```

只要这条链你记住了，就很难再把 PG 误解成“可有可无的中间层”。

## 初学者最容易混淆的 9 个点

### 1. 认为 PG 只是对象分桶

太浅了。它还是 placement、一致性、恢复和校验单元。

### 2. 认为对象完全可以直接映射 OSD，PG 只是历史包袱

不对。PG 是 Ceph 解决扩展性和治理复杂度的核心抽象。

### 3. 认为 CRUSH 直接吃裸对象

更准确地说，是对象先变成 PG，再由 CRUSH 做放置。

### 4. 认为 primary 是对象级别的

不对。primary 是 PG 级别的。

### 5. 认为 peering 就是 recovery

不对。peering 先统一历史，recovery 再补数据。

### 6. 认为 `up set` 和 `acting set` 与 PG 关系不大

不对。它们正是 PG 把理想放置和实际运行连接起来的关键概念。

### 7. 认为 PG 的价值只在数据分布

不对。它同样体现在 peering、backfill、scrub 和 recovery 上。

### 8. 认为 PG 越多越好

不对。PG 是在均衡性和管理开销之间的工程权衡。

### 9. 认为看懂 `CRUSH` 就等于看懂了 Ceph 的放置与一致性

不对。没有 PG，CRUSH 只是理想定位算法；真正把它变成可运行系统的是 PG。

## 这一篇最应该留下的 5 个直觉

### 直觉一：PG 首先是扩展性抽象

它让 Ceph 不必按海量对象维护完整运行时状态。

### 直觉二：PG 同时也是一致性和恢复抽象

它给 peering、recovery、backfill、scrub 提供统一边界。

### 直觉三：Ceph 的核心定位链不是“对象 -> OSD”，而是“对象 -> PG -> OSD”

这是理解整个数据面的基础。

### 直觉四：primary、up set、acting set 都是 PG 级语义

这说明 PG 是真正的运行时治理单元。

### 直觉五：没有 PG，Ceph 也许还能算位置，但很难在大规模环境中稳定收敛

这就是为什么说 PG 是 Ceph 的灵魂。
