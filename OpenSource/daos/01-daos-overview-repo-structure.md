# 01. DAOS 是什么：从产品定位到仓库结构

## 为什么先写这篇

如果你是第一次接触 DAOS，最容易遇到的困难不是某个函数看不懂，而是刚打开仓库就被一大堆目录和多种语言吓住了：

- 根目录里既有 `README.md`、`mkdocs.yml`，也有 `SConstruct`。
- `src/` 下面同时出现 Go、C、Protocol Buffers 生成代码和大量 `README.md`。
- 仓库里还有 `docs/`、`ci/`、`utils/` 这些看起来同样重要，但职责并不相同的目录。

所以这一篇不急着钻实现细节，而是先回答四个基础问题：

1. DAOS 到底是什么产品，解决什么问题？
2. 为什么它反复强调分布式、异步、对象存储、SCM、NVMe？
3. 代码库为什么同时有 Go、C、Python、Proto 和 MkDocs？
4. 第一次读源码，应该先从哪些目录和文件建立全局地图？

## 先用一句话理解 DAOS

根目录 `README.md` 对 DAOS 的定义非常直接：DAOS 是一个为大规模分布式非易失性内存而设计的开源软件定义对象存储系统。

把这句话拆开看，基本就能得到 DAOS 的产品定位：

- 它是 **Distributed** 的，不是单机存储引擎，而是要把多台节点上的本地介质组织成一个系统。
- 它是 **Asynchronous** 的，强调非阻塞 I/O 和并发执行，不是传统“一个请求一个阻塞调用”的设计思路。
- 它是 **Object Storage** 的，核心暴露的是对象与 key-value 风格接口，而不是把 POSIX 文件系统当成唯一中心。
- 它是为 **NVM/SCM/NVMe** 时代设计的，不是把旧架构简单搬到新硬件上，而是从一开始就围绕低延迟和高并发介质来组织软件栈。

`README.md` 里还点出了几个关键词：事务化非阻塞 I/O、自愈式数据保护、端到端数据完整性、细粒度数据控制、弹性存储。这说明 DAOS 不只是“能存数据”，而是要在高性能和高可靠之间做系统级折中。

换句话说，DAOS 试图解决的不是“有没有一个分布式存储可用”，而是下面这类更 HPC、更数据密集型的问题：

- 节点很多，设备很多，单机文件系统思路不够用了。
- 介质层次复杂，既有 SCM，也有 NVMe，需要软件栈主动利用各自特点。
- 请求并发高，元数据和数据路径都不能靠重阻塞模型硬扛。
- 系统必须在故障、校验、重建、一致性这些问题上有一整套内建机制。

## DAOS 为什么强调这几个关键词

### 1. 为什么是分布式

`src/README.md` 把 DAOS 描述成一个由多个组件组成的系统，并明确指出一个 DAOS system 是一组通过 fabric 互联的服务器集合。每台服务器管理本地挂载的 SCM 和 NVM 存储，但池不能跨多个系统。

这意味着 DAOS 的基本单位不是“某块盘”，而是“一个由多台服务器协同组成的存储系统”。因此，后面的很多设计都会自然围绕分布式展开：

- 需要有控制面管理节点、配置、设备和生命周期。
- 需要有数据面专门处理高性能 I/O。
- 需要网络通信中间件来完成客户端到服务端、服务端到服务端的通信。
- 需要元数据复制和故障恢复框架，保证集群不是单点。

### 2. 为什么是异步

根目录 `README.md` 直接把 transactional non-blocking I/O 写进了产品能力。这个表述很关键，因为它说明异步不是 API 上的点缀，而是设计中心之一。

从架构上看，DAOS 的数据面 `daos_engine` 是一个多线程 C 进程，依赖 Argobots 做事件驱动并发处理；客户端侧 `libdaos` 也不是只暴露简单同步封装，而是围绕任务和事件组织调用。异步化的价值至少有三层：

- 更容易把高并发 I/O 压到多 target、多线程执行模型上。
- 更容易把网络、存储、校验、重建等不同阶段拆成流水化任务。
- 更适合高性能存储场景下“延迟敏感 + 吞吐敏感”同时存在的负载。

### 3. 为什么是对象存储

DAOS 并不是以 POSIX 文件系统为最底层抽象。`README.md` 明确说它提供 key-value storage interface，而 `src/README.md` 又补充了 `libdaos`、`libdfs`、数组接口、KV 接口等不同层次的客户端能力。

这背后有两个重要含义：

- 真正的原生抽象是对象、key、container、pool 这些概念。
- POSIX 兼容能力不是没有，而是通过 `libdfs` 等上层封装来提供迁移路径。

这也是为什么在 `src/` 里你会看到 `pool/`、`container/`、`object/` 这些目录处于核心位置。DAOS 的业务模型不是先有文件，再硬拆成对象；而是先有对象化和事务化的数据模型，再向上适配文件语义。

### 4. 为什么是 SCM 和 NVMe

根目录 `README.md` 对这一点说得很明确：DAOS 从一开始就是为了 next generation NVM technology 设计的，尤其是 SCM 和 NVMe。

结合 `src/README.md` 可以看到这种介质意识如何体现在代码结构里：

- 数据面通过 PMDK 访问 SCM。
- 数据面通过 SPDK 访问 NVMe SSD。
- `src/vos` 负责本地版本化对象存储。
- `src/bio` 负责 blob I/O。
- `src/vea` 负责 NVMe 空间分配。

也就是说，SCM 和 NVMe 对 DAOS 不是“底层硬件细节”，而是决定软件分层的重要前提。你甚至可以把 DAOS 理解成一套围绕新型存储介质重构过的数据服务栈。

## 从产品能力到内部架构

理解 DAOS 的最好方式，不是死记模块名，而是把产品能力映射到真实分层。

`src/README.md` 给出了一条很清晰的主线：DAOS 由控制面、数据面、客户端组件和若干基础设施库共同组成。

### 控制面：`daos_server`

控制面二进制叫 `daos_server`，代码主要在 `src/control/`，语言是 Go。它负责：

- 解析配置文件。
- 准备和配置本地存储资源。
- 启动并监控一个或多个数据面实例。
- 通过 gRPC 暴露管理接口。

这说明控制面的核心职责不是跑数据 I/O，而是做编排、运维、管理和生命周期控制。

### 数据面：`daos_engine`

数据面二进制叫 `daos_engine`，代码主要在 `src/engine/`，语言是 C。它负责：

- 接收和处理元数据与 I/O 请求。
- 通过 CART 参与高性能数据通信。
- 访问本地 SCM 与 NVMe。
- 装载 pool、container、object、rebuild 等服务模块。

这说明 DAOS 的“高性能部分”主要在数据面，控制面和数据面是刻意分开的。

### 客户端与生态接口

`src/README.md` 还把客户端层讲得很清楚：

- `libdaos` 是原生客户端库。
- `libdfs` 在其上模拟 POSIX 风格文件和目录抽象。
- `dmg` 是管理工具，服务于系统管理员。
- `daos_agent` 运行在客户端节点上，参与认证和系统成员信息获取。

这一步很重要，因为它解释了 DAOS 为什么不是“只有一个 server 目录”的工程。它要同时服务：

- 存储系统管理员。
- 应用与中间件开发者。
- 需要 POSIX 迁移路径的用户。
- 更上层的生态集成，如 HDF5、MPI-IO、Spark、TensorFlow-IO。

### 基础设施库

在业务服务之外，DAOS 还维护了一组基础设施层：

- `src/gurt` 和 `src/common` 提供日志、调试、通用数据结构与公共能力。
- `src/cart` 提供高性能通信中间件。
- `src/vos`、`src/bio`、`src/vea` 组成存储内核能力。
- `src/rdb`、`src/rsvc` 提供复制元数据与高可用基础框架。
- `src/placement` 负责对象布局算法。

因此，DAOS 不是单体程序，而更像“控制面应用 + 数据面引擎 + 客户端库 + 一组服务模块 + 一组基础设施库”的组合体。

## 为什么仓库里会同时出现 Go、C、Python、Proto、MkDocs

第一次看 DAOS 仓库，最常见的疑问就是：为什么一个项目要用这么多语言？

答案其实正来自它的分层和工程目标。

### Go：控制面与运维编排

`src/control/` 基本是 Go 代码。原因很直接：

- 控制面偏管理编排、配置、进程控制、gRPC 服务。
- Go 对并发、网络服务、CLI 和工程组织都比较友好。
- `dmg`、`daos_agent`、`daos_server` 这类组件天然适合用 Go 来实现。

所以 Go 主要承担“好维护、好编排、好暴露管理接口”的角色。

### C：数据面、核心服务与存储路径

`src/engine/`、`src/pool/`、`src/container/`、`src/object/`、`src/vos/` 等关键模块大量使用 C。原因也很明显：

- 数据面更贴近高性能网络、内存管理、线程模型和存储介质访问。
- 需要对延迟、资源、布局和执行路径做更细粒度控制。
- 需要与 PMDK、SPDK、Mercury、libfabric 等底层库紧密协作。

因此 C 主要承担“性能关键路径和底层能力”的角色。

### Protocol Buffers：跨进程与跨语言协议

`src/proto/` 存放 `.proto` 文件，`src/README.md` 也明确指出 gRPC 和 dRPC 都依赖 protocol buffers 做序列化。

这解释了两个现象：

- 为什么控制面 Go 代码和数据面 C 代码能共享一部分协议定义。
- 为什么仓库中会看到大量 `*.pb.go`、`*.pb-c.c`、`*.pb-c.h` 之类的生成代码。

Proto 在这里承担的是“统一协议描述”的角色，而不是业务逻辑本身。

### Python：测试、脚本与工程辅助

Python 虽然不是 DAOS 核心数据面的主语言，但在工程体系里很重要。仓库里可以直接看到：

- `utils/run_utest.py`
- `src/client/setup.py`
- `ci/` 和 `utils/` 下的一系列脚本

这些内容说明 Python 主要用在：

- 测试组织与辅助工具。
- 部分客户端包装与开发支持。
- CI、自动化、环境准备等工程任务。

### MkDocs：文档站点组织

`mkdocs.yml` 说明官方文档站点采用 MkDocs 组织，`docs/` 目录则是文档内容本体。导航里可以看到 Overview、Administration Guide、User Guide、Developer Zone 等完整结构。

这意味着仓库里的文档不是随手写的附属品，而是被当成正式产品资产来维护的：

- `docs/overview/` 讲产品模型。
- `docs/admin/` 面向部署和运维。
- `docs/user/` 面向使用者。
- `docs/dev/` 面向开发者。

对源码阅读者来说，这一点很重要，因为很多实现细节都先在文档层给出概念框架。

## 仓库顶层目录应该怎么理解

如果把仓库当成一张地图，最值得先认识的是下面几个位置。

### `src/`：真正的实现主战场

`src/README.md` 就是理解源码分层的总入口。这个目录下同时容纳：

- 控制面：`src/control`
- 数据面：`src/engine`
- 客户端：`src/client`
- 核心服务：`src/pool`、`src/container`、`src/object`、`src/rebuild`、`src/mgmt`
- 存储内核：`src/vos`、`src/bio`、`src/vea`
- 通信与公共库：`src/cart`、`src/common`、`src/gurt`
- 高可用基础设施：`src/rdb`、`src/rsvc`

如果只选一个目录作为“源码总入口”，那就是 `src/`。

### `src/include/`：API 和头文件边界

`src/README.md` 对这里的职责解释得很清楚：

- 面向最终用户的正式 DAOS API 头文件放在这里。
- 公共基础库暴露的头文件也在这里。
- 客户端接口通常在 `src/include/daos`。
- 服务端接口通常在 `src/include/daos_srv`。

这意味着 `src/include/` 不只是头文件堆放区，而是观察“哪些能力是公开边界、哪些能力是内部协作接口”的重要窗口。

### `docs/`：产品、部署、用户与开发文档

`docs/` 不是补充材料，而是理解产品语义的第一站。尤其在第一轮阅读里，文档常常比直接看 C 代码更高效，因为它先把名词、模型和边界讲清楚。

比如：

- 想看整体架构，先看 `docs/overview/architecture.md`
- 想看存储模型，先看 `docs/overview/storage.md`
- 想看事务模型，先看 `docs/overview/transaction.md`

### `ci/`：持续集成与验证流程

`ci/` 下能看到 unit、functional、storage、rpm、provisioning 等子目录，说明 DAOS 的测试和发布不是“跑一个单元测试命令”那么简单，而是覆盖：

- 单元测试
- 功能测试
- 存储场景验证
- 依赖安装和环境准备
- RPM 构建与发布链路

这能帮助我们理解：DAOS 是一个面向真实集群部署的软件系统，工程复杂度本身就是产品复杂度的一部分。

### `utils/`：工程工具箱

`utils/` 下既有配置模板，也有打包脚本、docker 文件、systemd 配置、证书生成、测试入口和开发辅助工具。它更像仓库的“工程后勤中心”。

比如：

- `utils/config/` 提供 `daos_server.yml`、`daos_agent.yml` 等配置样例。
- `utils/rpms/` 负责 RPM 打包相关逻辑。
- `utils/run_utest.py` 是单元测试入口之一。
- `utils/docker/` 和 `utils/systemd/` 反映部署与运行方式。

### `SConstruct`：整个构建系统的总入口

根目录 `SConstruct` 说明 DAOS 的构建体系以 SCons 为主。这个文件做的事情非常多：

- 处理构建选项和环境变量。
- 检查依赖。
- 驱动 `src/` 下各模块的构建。
- 处理安装、测试、RPM 等构建目标。

如果说 `src/README.md` 是“实现地图”，那 `SConstruct` 更像“装配地图”。

### `mkdocs.yml`：文档系统总入口

`mkdocs.yml` 定义了文档站点名称、导航树、主题和插件。它的意义不只是“告诉 MkDocs 怎么生成网页”，更在于它展示了项目如何组织对外知识：

- 发布信息
- 产品概览
- 安装与部署
- 管理指南
- 用户指南
- 开发者文档

对源码读者来说，这份导航树几乎就是一份“先读哪些概念文档”的目录索引。

## 一张适合第一次阅读的源码地图

如果你现在准备真正进入 DAOS 源码，我建议不要按目录名从 A 读到 Z，而是按下面顺序建立理解。

### 第一步：先看产品和整体架构

先读这些文件：

- `README.md`
- `docs/overview/architecture.md`
- `docs/overview/storage.md`
- `src/README.md`

目标不是记细节，而是先建立“系统里有哪些角色、哪些进程、哪些关键概念”。

### 第二步：再看三个核心进程

接着看这些入口：

- `src/control/cmd/daos_server/main.go`
- `src/control/cmd/dmg/main.go`
- `src/control/cmd/daos_agent/main.go`
- `src/engine/init.c`
- `src/engine/module.c`

这一阶段重点是搞清楚：

- 管理命令从哪里进来。
- 控制面如何启动和编排数据面。
- 数据面如何初始化并装载服务模块。

### 第三步：进入服务层

当你已经理解“系统是怎么跑起来的”，再去看：

- `src/pool/README.md`
- `src/container/README.md`
- `src/object/README.md`
- `src/rebuild/README.md`

这时就能把 pool、container、object 理解成真正的服务模块，而不是几个抽象名词。

### 第四步：最后下潜到底层存储内核

再往下读：

- `src/vos/README.md`
- `src/bio/README.md`
- `src/vea/README.md`
- `src/rdb/README.md`
- `src/rsvc/README.md`

到了这一步，你才会真正看见 DAOS 如何把对象模型、版本控制、元数据复制和介质管理落到本地实现上。

## 对新读者最重要的一个认识

第一次读 DAOS，最容易犯的错误是把它当成“一个大服务”来看。其实它更接近一个分层清晰、职责拆分明确的系统：

- Go 控制面负责管理、编排和对外管理接口。
- C 数据面负责真正的高性能 I/O 和服务执行。
- 客户端库负责把应用请求组织成 DAOS 原生语义。
- 基础设施库负责通信、存储、复制、布局和公共能力。
- 文档、构建、测试和运维脚本共同构成完整工程体系。

所以，仓库里语言多、目录多，不是因为工程失控，而是因为产品本身就覆盖了从控制面到数据面、从客户端到运维、从协议到构建测试的完整链路。

## 小结

回到开头的问题，现在可以给 DAOS 一个更务实的定义：

DAOS 不是“某个高性能 KV 引擎”，也不是“某个单纯的分布式文件系统替代品”。它是一套围绕分布式新型存储介质而设计的对象存储软件栈，核心目标是在高性能、高并发、高可靠和可扩展之间取得系统级平衡。

从源码角度看，理解 DAOS 最有效的方法不是直接扎进某个 C 文件，而是先建立这条映射关系：

**产品能力 -> 控制面/数据面分层 -> 服务模块 -> 基础设施库 -> 仓库目录与构建体系**

只要这条主线建立起来，后面无论你是去看 `daos_server` 的启动流程，还是去看 `pool`、`object`、`vos` 的具体实现，都会更容易判断自己现在处在整张地图的哪一层。
