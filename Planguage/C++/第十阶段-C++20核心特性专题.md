> 前置要求：完成现代 C++、STL、模板、并发和工程化阶段。  
> 建议周期：6～8 周  
> 阶段目标：从标准版本视角系统掌握 C++20 的语言和标准库主线，并能够将旧式 C++17 项目有计划地升级。

## 一、C++20 的整体变化

C++20 的核心主线可以概括为：

```text
泛型约束：Concepts
数据处理：Ranges
异步抽象：Coroutines
编译组织：Modules
比较模型：<=>
编译期能力：constexpr / consteval / constinit
并发设施：jthread / semaphore / latch / barrier / atomic wait
基础库：format / span / source_location / chrono / bit
```

这些功能并非彼此孤立：Concepts 支撑 Ranges 的接口约束；协程提供暂停与恢复机制，但需要调度器和异步 I/O；Modules 改变依赖组织，却不取代构建系统和包管理器。

### 学习成果

完成后应能够：

1. 使用 Concepts 约束模板接口。
2. 使用 Ranges 算法、投影与 View 管道。
3. 理解协程 promise/awaiter/frame 协议。
4. 组织基础 module interface、implementation 和 partition。
5. 正确设计三路比较与相等语义。
6. 使用 C++20 编译期、并发和标准库设施。
7. 使用 feature-test macro 处理实现差异。
8. 制定 C++17 到 C++20 的迁移方案。

## 二、Concepts

### 2.1 约束模板

```cpp
#include <concepts>

template<std::integral T>
T gcd(T left, T right)
{
    while (right != 0) {
        const T remainder = left % right;
        left = right;
        right = remainder;
    }
    return left;
}
```

Concepts 的收益：

- 在接口处表达能力要求。
- 更早淘汰不适用候选。
- 改善错误诊断。
- 参与重载排序。

### 2.2 requires-expression

```cpp
template<class T>
concept StreamWritable = requires(std::ostream& output, const T& value) {
    { output << value } -> std::same_as<std::ostream&>;
};
```

四类要求：简单要求、类型要求、复合要求、嵌套要求。

### 2.3 简写函数模板

```cpp
void print(const StreamWritable auto& value)
{
    std::cout << value;
}
```

每个 `auto` 参数对应独立模板参数。若两个参数必须同类型，应显式写模板参数与 `same_as` 关系。

### 2.4 约束设计

约束应最低但充分：实现只遍历一次就要求 `input_range`；需要重复遍历则要求 `forward_range`。过强会排除有效类型，过弱会让错误深入实现。

Concept 主要验证语法和类型关系，不能自动证明比较器满足严格弱序等运行时语义。

### 2.5 迁移 SFINAE

```cpp
// 旧式
template<class T, std::enable_if_t<std::is_integral_v<T>, int> = 0>
T twice(T value);

// C++20
template<std::integral T>
T twice(T value);
```

迁移时不要只机械翻译，要重新检查原约束是否与实现一致。

### 练习

1. 为序列求和函数设计 Range 约束。
2. 定义 `Hashable` 和 `ReservableRange` Concept。
3. 将三个 `enable_if` 接口改写为 Concepts。
4. 编写一般 Range 与 sized Range 的约束重载。

## 三、Ranges

### 3.1 Range 算法

```cpp
std::ranges::sort(users, {}, &User::name);
auto position = std::ranges::find(users, id, &User::id);
```

投影把“取哪个字段”与“如何比较”分开。

### 3.2 Views

```cpp
auto result = values
    | std::views::filter([](int value) { return value % 2 == 0; })
    | std::views::transform([](int value) { return value * value; })
    | std::views::take(10);
```

Views 通常轻量、惰性且非拥有。创建管道不等于已经执行计算。

### 3.3 常用 View

- `filter`、`transform`。
- `take`、`drop`。
- `take_while`、`drop_while`。
- `reverse`、`split`、`join`。
- `keys`、`values`、`elements`。
- `iota`、`counted`、`common`。

### 3.4 生命周期

```cpp
auto bad()
{
    std::vector<int> local{1, 2, 3};
    return local | std::views::filter([](int value) { return value > 1; });
}
```

返回的 View 可能引用已销毁容器。需要返回拥有结果、延长数据所有者生命周期，或使用明确 owning 设计。

### 3.5 borrowed range

Ranges 算法会根据范围是否允许临时对象销毁后继续使用迭代器，返回迭代器或 `std::ranges::dangling`。泛型代码不能假设所有范围都像 `vector` 一样可多次遍历、随机访问或拥有元素。

### 3.6 经典算法互操作

C++20 尚没有所有算法的完整 Ranges 版本，也没有标准 `ranges::to`。需要物化时可用经典算法和插入迭代器，或直接构造目标容器；`ranges::to` 属于 C++23。

### 练习

1. 使用 View 处理筛选、转换和截断。
2. 使用投影按结构体字段排序。
3. 找出返回 View 的悬空问题。
4. 比较惰性重复计算与物化结果。

## 四、Coroutines

### 4.1 三个关键字

- `co_await`：等待 awaitable。
- `co_yield`：产生值并暂停。
- `co_return`：完成协程。

协程本身不创建线程，也不自动提供调度器。

### 4.2 coroutine frame

frame 保存跨暂停点存活的参数、局部变量、promise 和恢复位置。返回对象必须明确谁拥有和销毁 frame。

### 4.3 promise 协议

常见成员：

```text
get_return_object
initial_suspend
final_suspend
return_value / return_void
yield_value
unhandled_exception
```

这里的 promise 与 `std::promise` 无关。

### 4.4 awaiter 协议

```cpp
bool await_ready();
auto await_suspend(std::coroutine_handle<> handle);
Result await_resume();
```

`await_suspend` 决定如何保存、转移或恢复协程，是调度连接点。

### 4.5 generator 示例轮廓

```cpp
Generator<int> sequence(int count)
{
    for (int value{}; value < count; ++value) {
        co_yield value;
    }
}
```

C++20 没有标准 `std::generator`；需要自行实现教学类型或使用成熟库。`std::generator` 属于 C++23。

### 4.6 生命周期风险

- 引用参数跨暂停点可能悬空。
- 捕获式 Lambda 协程的闭包可能提前销毁。
- coroutine handle 默认只是非拥有句柄。
- final suspend 与销毁协议错误会泄漏或重复销毁。
- 异常应由 `unhandled_exception` 保存或传播。

### 4.7 适用场景

- 惰性序列。
- 异步 I/O。
- 状态机。
- 事件流。

生产 task 还需取消、调度、异常、continuation 和执行上下文设计，教学实现不可直接投入生产。

## 五、Modules

### 5.1 基本模块

```cpp
// math.cppm
export module deep.math;

export int add(int left, int right)
{
    return left + right;
}
```

使用：

```cpp
import deep.math;

int main()
{
    return add(1, 2);
}
```

### 5.2 模块单元

- primary module interface unit。
- module implementation unit。
- interface partition。
- implementation partition。
- header unit。

```cpp
export module deep.math:vector;
```

### 5.3 global module fragment

```cpp
module;
#include <legacy_header.h>

export module deep.wrapper;
```

用于在模块声明前处理传统头文件和宏环境。

### 5.4 export

只有导出的声明对导入者可见。模块内部实现细节可避免传统文本包含传播，但导出接口涉及的类型仍需可达。

### 5.5 Modules 不是什么

- 不是包管理器。
- 不自动下载依赖。
- 不取代 CMake/Ninja/MSBuild。
- 不自动解决 ABI 稳定性。
- 不保证运行性能提高。

### 5.6 构建挑战

编译器需先生成模块接口产物，再编译导入者。BMI/IFC/PCM 格式通常编译器特定，不应当作跨编译器发布制品。

### 5.7 迁移策略

1. 先清理头文件自包含和循环依赖。
2. 选择稳定、低依赖模块试点。
3. 保留传统接口兼容层。
4. 确认编译器与 CMake 版本矩阵。
5. 测量全量与增量编译时间。

## 六、三路比较 `<=>`

### 6.1 默认比较

```cpp
#include <compare>

struct Version
{
    int major{};
    int minor{};
    int patch{};

    auto operator<=>(const Version&) const = default;
};
```

默认比较按基类和成员声明顺序进行，只有顺序符合类型语义时才应默认生成。

### 6.2 比较类别

- `strong_ordering`：强全序。
- `weak_ordering`：等价但不一定可替换。
- `partial_ordering`：可能不可比较，如浮点 NaN。

返回类别可由成员比较综合推导。

### 6.3 `==`

默认 `<=>` 可帮助生成关系运算，但相等和排序语义仍需认真设计。大小写不敏感字符串、版本预发布标签和浮点字段不能机械按成员比较。

### 6.4 rewritten candidates

编译器可根据 `==` 和 `<=>` 改写多种比较表达式。若同时保留大量旧式重载，要检查歧义和语义一致性。

## 七、编译期能力

### 7.1 `consteval`

```cpp
consteval int square(int value)
{
    return value * value;
}
```

每次潜在求值调用必须在编译期完成，适合格式验证、编译期标识和代码生成辅助。

### 7.2 `constinit`

```cpp
constinit int counter = 0;
```

确保静态或线程存储期对象静态初始化，但对象之后仍可修改；它不等于 `const`。

### 7.3 constexpr 扩展

C++20 允许更多操作进入常量求值，包括更灵活的函数体、动态分配的受限使用和更多标准库支持。常量求值中的动态分配不能让存储泄漏到运行期。

### 7.4 `std::is_constant_evaluated`

```cpp
constexpr int compute(int value)
{
    if (std::is_constant_evaluated()) {
        return compile_time_path(value);
    }
    return runtime_path(value);
}
```

用于区分常量求值与运行时路径。不要用它制造两个语义不同的算法结果。

### 7.5 `constinit` 解决初始化顺序风险

静态初始化顺序问题常通过 constexpr/constinit、函数局部静态或减少全局状态解决。`constinit` 只能保证初始化阶段，不自动解决所有跨翻译单元依赖。

## 八、并发与同步设施

### 8.1 `jthread` 和 stop token

```cpp
std::jthread worker{[](std::stop_token token) {
    while (!token.stop_requested()) {
        work_once();
    }
}};
```

析构请求停止并 join，停止是协作式的，不能强制终止阻塞代码。

### 8.2 semaphore、latch、barrier

- semaphore：计数许可。
- latch：一次性倒计数同步。
- barrier：可重复阶段同步。

### 8.3 atomic wait/notify

```cpp
state.wait(old_value);
state.store(new_value, std::memory_order_release);
state.notify_all();
```

减少忙等，但仍必须使用正确状态协议和内存序。

### 8.4 `atomic_ref`

```cpp
int value{};
std::atomic_ref<int> reference{value};
reference.fetch_add(1);
```

允许以原子方式访问已有对象，但对象必须满足对齐和生命周期要求，并且同一期间的冲突访问必须全部遵循原子协议。

## 九、核心标准库特性

### 9.1 `std::span`

非拥有连续视图：

```cpp
void process(std::span<const int> values);
```

不负责底层生命周期，适合统一接受数组、array 和 vector。

### 9.2 `std::format`

```cpp
auto text = std::format("{} scored {:.2f}", name, score);
```

提供类型安全格式化。C++20 尚无 `std::print`，后者属于 C++23。

### 9.3 `std::source_location`

```cpp
void log(std::string_view message,
    std::source_location location = std::source_location::current());
```

替代部分文件、行号宏用途。

### 9.4 bit 操作

`<bit>` 提供：

- `bit_cast`。
- `endian`。
- `popcount`。
- `countl_zero`、`countr_zero`。
- `rotl`、`rotr`。
- `has_single_bit`、`bit_ceil`、`bit_floor`、`bit_width`。

位操作通常面向无符号整数。

### 9.5 日历和时区

`<chrono>` 加入：

- `year_month_day`。
- `weekday`。
- 日历日期运算。
- 时区数据库与 zoned time。

时区支持的标准库落地曾长期不一致，使用前检查工具链和时区数据库部署。

### 9.6 其他重要设施

- `starts_with`、`ends_with`。
- `contains`（关联容器）。
- `erase`、`erase_if`。
- `to_array`。
- `midpoint`、`lerp`。
- `make_shared` 数组支持。
- `syncstream`，用于同步输出片段。
- `bind_front`。
- 数学常量 `std::numbers`。

## 十、聚合、Lambda 与语言改进

### 10.1 designated initializer

```cpp
Config config{
    .host = "localhost",
    .port = 8080
};
```

C++ 指定初始化必须遵循成员声明顺序，且只用于聚合类型；规则不同于 C。

### 10.2 Lambda 改进

- 模板参数列表：`[]<class T>(T value)`。
- 无捕获 Lambda 可默认构造和赋值。
- `[=]` 隐式捕获 `this` 被弃用，应明确写 `[this]` 或 `[*this]`。
- Lambda 可用于更多未求值和编译期上下文。

### 10.3 aggregate initialization 改进

括号形式聚合初始化和聚合定义规则有所调整。迁移旧代码时应由编译器警告和测试确认，不要假设所有“像结构体”的类仍是聚合。

### 10.4 `using enum`

```cpp
using enum Color;
auto color = red;
```

适合局部减少限定，但大范围使用可能造成名称冲突。

## 十一、Feature-test macros

### 11.1 为什么使用

`__cplusplus` 只能粗略表示语言模式，不能证明某项标准库功能完整可用。优先使用功能测试宏：

```cpp
#include <version>

#if defined(__cpp_lib_format) && __cpp_lib_format >= 201907L
    // 使用 std::format
#else
    // fallback
#endif
```

### 11.2 常见宏

- `__cpp_concepts`。
- `__cpp_impl_coroutine`。
- `__cpp_modules`。
- `__cpp_lib_ranges`。
- `__cpp_lib_format`。
- `__cpp_lib_jthread`。
- `__cpp_lib_source_location`。
- `__cpp_lib_span`。

准确数值应查询当前编译器文档或 cppreference，不要凭记忆硬编码未经验证的阈值。

### 11.3 CMake 检测

使用 `target_compile_features(target PRIVATE cxx_std_20)` 请求标准级别；对关键库设施可结合小型 `try_compile` 或配置测试。编译通过的配置应进入 CI 矩阵。

## 十二、迁移 C++17 项目

### 12.1 迁移顺序

1. 建立测试、警告和 Sanitizer 基线。
2. 将构建标准切换为 C++20。
3. 修复弃用、行为变化和编译器诊断。
4. 用 Concepts 改善公共泛型接口。
5. 用 Ranges 改写真正提升可读性的处理流程。
6. 用 jthread 改善线程生命周期。
7. 引入 span、format、source_location 等局部设施。
8. Modules 与协程单独试点，不与基础迁移一次完成。

### 12.2 不要机械现代化

- 简单循环不必全部改为长 View 管道。
- SFINAE 若稳定且不公开，可分阶段迁移。
- Modules 的工具链成本可能超过短期收益。
- 协程不应替换普通同步函数。
- `<=>` 只在比较语义清晰时使用。

### 12.3 兼容性

库若仍需支持 C++17，可：

- 保持核心 API 为 C++17，提供 C++20 适配层。
- 通过宏隔离可选实现。
- 分支或版本化发布。
- 明确最低编译器和标准库版本。

避免在公共头文件无意传播 C++20 要求。

## 十六、易错点汇总

1. Concept 约束与实现不一致。
2. 把所有循环机械改成 View 管道。
3. 返回引用局部容器的 View。
4. 假设所有 Range 可多次遍历。
5. 认为协程自动创建线程。
6. 协程 frame 所有权不明确。
7. 引用跨暂停点悬空。
8. 把 Modules 当作依赖下载工具。
9. 提交编译器特定 BMI 作为可移植库。
10. 默认 `<=>` 与业务排序不一致。
11. 对浮点使用错误的强序假设。
12. 混淆 consteval、constinit 和 const。
13. 认为 stop_token 能强制终止线程。
14. atomic_ref 与普通冲突访问混用。
15. span 比底层数据活得更久。
16. 把 std::print 或 std::generator 错列为 C++20。
17. 只检查 __cplusplus 判断库支持。
18. 一次迁移同时引入 Modules、协程和大量重构。
