# 部署一个最小可用 Ceph 集群：从节点角色到 cephadm 编排

## 这篇文章要解决什么问题

前两篇我们已经完成了两层认知：

- Ceph 的本体是统一的分布式对象存储底座
- `MON`、`MGR`、`OSD`、`MDS`、`RGW` 分别属于控制面、管理面、数据面和特定接口层

但如果你还没有真正把一个集群拉起来，这些认知很容易停留在“看起来懂了”的阶段。真正一部署，问题马上就会变得非常具体：

- 一个最小可用 Ceph 集群，到底至少要几台机器、几个角色？
- `cephadm` 到底帮你自动化了什么，哪些仍然要你自己规划？
- 为什么 bootstrap 完只有 `MON` 和 `MGR`，数据还不能真正写进去？
- `MDS` 和 `RGW` 为什么不是一开始就必须部署？
- Prometheus、Grafana、Dashboard 到底是默认就有，还是要手工加？

这一篇的目标不是写一份“所有环境都能直接复制粘贴”的运维手册，而是建立一条最重要的工程主线：

**如何从 0 到 1，把一个最小可用的 Ceph 集群部署出来，并且知道每一步背后的角色边界。**

换句话说，这篇既要回答“怎么做”，也要回答“为什么 Ceph 要这样做”。

## 先讲结论：最小可用，不等于最小演示

很多文章一上来就告诉你“一台机器也能跑 Ceph”。这句话从实验角度没有错，但从工程角度会误导初学者。

因为“一台机器跑起来”和“一个最小可用的集群”不是一回事。

你可以先把部署目标分成三档：

### 1. 单机实验环境

适合：

- 学命令
- 看 Dashboard
- 体验 `cephadm`
- 验证某个功能能不能跑起来

不适合：

- 理解真实故障域
- 验证仲裁与高可用
- 做有代表性的性能结论

### 2. 三节点最小可用集群

适合：

- 建立最基本的 Ceph 工程直觉
- 体验 `MON` 仲裁、`OSD` 分布、主机标签与服务编排
- 作为测试、PoC、小规模验证环境

这是本文重点讨论的目标形态。

### 3. 面向生产的最小起步集群

通常还会继续考虑：

- 网络隔离
- 盘型分层
- 故障域设计
- 更合理的 `MON` 数量
- 独立网关节点
- 监控、安全、备份、容量冗余

这部分会在后续文章逐步展开，本文不试图一次讲完。

所以，本文里的“最小可用”，更准确地说是：

**一个以三节点为基本单位、用** **`cephadm`** **管起来、具备控制面、数据面和基本可观测能力、并可按需扩展 CephFS 与 RGW 的入门级工程集群。**

## 为什么现在 Ceph 更强调 `cephadm`

如果你翻 Ceph 历史资料，会看到很多旧式部署方法，例如手工部署、旧版本工具链、散落在各节点上的配置文件式管理。它们并不是错，只是对于今天的大多数 Ceph 用户来说，已经不再是最优入口。

`cephadm` 的核心价值不是“把几条命令变短”，而是：

- 把 Ceph 服务交给统一的编排入口管理
- 让守护进程以容器方式部署和更新
- 让新增主机、部署服务、下发配置、维护监控栈这类事情有一致的操作模型
- 让集群从“每台机器手工养护”转向“以 service spec 和 orchestrator 视角管理”

官方文档里，对它的定义其实很直接：`cephadm` 通过先 bootstrap 一台主机，再把更多主机纳入集群，然后部署所需服务，来创建和扩展 Ceph 集群。

这句话很重要，因为它已经给出完整主线：

1. 先在第一台主机上完成 bootstrap
2. 再把其他主机加入集群
3. 再部署真正需要的服务

这也是你理解本篇结构的最好方式。

## 部署前先建立一张“最小可用拓扑图”

先不要急着看命令，先看角色落点。

一个三节点最小可用 Ceph 集群，可以先用下面这张图建立直觉：

```text
                  +-------------------+
                  |   管理终端 / 运维机  |
                  | ceph / cephadm CLI |
                  +---------+---------+
                            |
                            v
     +--------------------------------------------------------------+
     |                        Ceph 集群                              |
     |--------------------------------------------------------------|
     |  host1                 host2                  host3          |
     |  -----                 -----                  -----          |
     |  mon                   mon                    mon            |
     |  mgr(active/standby)   osd                    osd            |
     |  osd                   [可选] mds/rgw         [可选] mds/rgw |
     |  _admin                                                   ... |
     +--------------------------------------------------------------+
```

这张图里有几个关键点：

- 第一个节点上通常先 bootstrap 出第一个 `MON` 和 `MGR`
- 三个节点都应该逐步承载 `OSD`，否则你没有真实的数据面
- `MDS` 只有在你启用 CephFS 时才需要
- `RGW` 只有在你需要 S3/Swift 接口时才需要
- 至少一个带 `_admin` 标签的主机要保留 CLI 和 keyring，方便运维

这其实已经隐含了 Ceph 的部署哲学：

- 先有控制面
- 再有数据面
- 最后按业务场景启用接口层

## 先明确主机和网络前提

根据仓库里的 `cephadm` 部署文档，bootstrap 前至少要先满足这些基础条件：

- Python 3
- systemd
- Podman 或 Docker
- 时间同步，例如 Chrony
- LVM2
- SSH 可用

这几项看起来普通，但在 Ceph 里都不是装饰品。

### 为什么时间同步是硬要求

因为 Ceph 里大量服务状态、日志时间线、监控告警、认证和分布式协调都依赖时间一致性。时间漂移会把排障难度直接抬高一个量级。

### 为什么 SSH 是硬要求

因为 `cephadm` 的一个核心前提就是：**它不是只管本机，而是要从 bootstrap 节点出发，把其他主机纳入统一编排。**

所以它需要通过 SSH 去扩展管理边界。

### 为什么 LVM2 很关键

因为 `cephadm` / `ceph-volume` 在部署 OSD 时，需要识别、准备和消费块设备。即使你暂时只是在虚拟机里试验，底层设备编排思路仍然绕不开它。

## 网络规划：部署前最容易低估的一步

很多第一次部署 Ceph 的人，把最多精力放在“命令怎么写”，但真正决定后面是不是顺利的，往往是网络规划。

在最小可用集群里，先记住两种网络概念：

- `public network`：客户端访问、控制面、外部通信主要依赖它
- `cluster network`：OSD 之间复制、恢复、心跳等内部流量主要依赖它

对于三节点实验或入门环境，你完全可以只用一张网络先跑起来。但要知道，这是“简化部署”，不是“最佳实践”。

官方文档也明确提到，更大的集群通常在性能上更受益于把 public 与 cluster traffic 分离，因为 OSD 之间的复制、恢复、心跳会产生大量内部流量。

所以网络规划可以先分成两档理解：

### 入门/PoC 环境

- 单网段先跑通
- 确保所有节点互通
- bootstrap 时指定第一台 `MON` 可达 IP

### 稍认真一点的集群

- 提前区分 public network 和 cluster network
- bootstrap 时就考虑 `--cluster-network`
- 避免后面扩容时再补做大规模网络迁移

## 第一步：安装 `cephadm`，但把它看成“引导器”而不是“全部”

在官方文档里，安装 `cephadm` 的方式可以是发行版安装，也可以是 curl 拉取对应版本的可执行文件。

对工程实践来说，你真正该记住的不是安装命令细节，而是这两个事实：

- 初始拿到 `cephadm` 可执行文件只是第一步
- 真正的目标是让它成为后续 bootstrap 和运维的统一入口

也就是说，`cephadm` 不是你部署完以后就可以忘掉的“安装脚本”，而是后续整套 Ceph 编排模型的入口。

## 第二步：在第一台主机上 bootstrap

这一步是整个 Ceph 集群的“出生动作”。

官方文档里最核心的命令形式是：

```bash
cephadm bootstrap --mon-ip <mon-ip>
```

这里的 `<mon-ip>` 不是随便填一个地址，而是：

- 第一台主机上可达的 IP
- 后续集群主机和客户端都能访问到的地址
- 如果有多网卡、多网段，必须选对外部访问路径明确的那个

### bootstrap 到底做了什么

这是很多初学者最容易忽略的地方。bootstrap 不是“起一个服务”这么简单，它实际上会完成一串非常关键的初始化动作。

根据仓库文档，至少会包括：

- 在本机创建第一个 `MON`
- 在本机创建第一个 `MGR`
- 生成集群 SSH key
- 把公钥写到相应位置，供后续纳管更多主机
- 生成最小可用的 `ceph.conf`
- 生成 `client.admin` keyring
- 给 bootstrap 主机打上 `_admin` 标签

这一步完成后，你拿到的是：

- 一个真正存在的 Ceph 集群控制面起点
- 一个可以继续向外扩展的编排入口
- 一套本地可直接访问集群的 CLI 凭据

### 为什么 bootstrap 完还不能算“可用存储”

因为 bootstrap 默认先帮你立起来的是控制面起点，而不是完整数据面。

也就是说，这时你通常已经有：

- `MON`
- `MGR`
- Dashboard / 监控相关基础能力的入口

但你还没有真正把数据盘纳入 `OSD`，也没有把三节点扩成一个具备真实冗余语义的集群。所以这时更准确的描述应该是：

**你得到了一个“可继续编排的 Ceph 集群起点”，但还没有得到一个成熟的数据存储集群。**

## 第三步：把其他主机纳入集群

bootstrap 完成后，下一件事不是立刻加业务服务，而是先把节点边界建立起来。

根据 `cephadm` 的主机管理文档，纳管一台新主机一般包含两步：

1. 把集群公钥装进新主机的 `authorized_keys`
2. 告诉 Ceph 这台主机属于当前集群

典型操作形态是：

```bash
ssh-copy-id -f -i /etc/ceph/ceph.pub root@host2
ssh-copy-id -f -i /etc/ceph/ceph.pub root@host3

ceph orch host add host2 10.10.0.102
ceph orch host add host3 10.10.0.103
```

### 为什么推荐显式写 IP

文档里明确建议最好显式提供主机 IP，因为如果不提供，Ceph 会立刻尝试通过 DNS 解析主机名并使用解析结果。对于实验环境还好，一旦 DNS、hosts、容器网桥或多网卡稍微复杂一点，后面排查通信问题会非常痛苦。

### 主机标签为什么值得尽早用

`cephadm` 的主机标签不是装饰，它是后面编排 placement 的基础。

最常见的两个用法是：

- 用 `_admin` 保留 CLI 与 keyring 下发
- 用自定义标签区分 `mds`、`rgw`、网关节点、监控节点等角色

例如：

```bash
ceph orch host label add host2 _admin
ceph orch host label add host2 mds
ceph orch host label add host3 rgw
```

对初学者来说，越早接受“主机是按标签和 placement 管理”的思路，越容易理解后面 `cephadm` 的编排模型。

## 第四步：先让 `MON` 仲裁形态变得合理

bootstrap 只会创建第一个 `MON`。但真正健康的 Ceph 集群，不应该长期停留在单 monitor 形态。

仓库文档对 monitor 的建议非常明确：

- 典型集群用三个或五个 `MON`
- 如果集群节点数更多，通常建议五个

对最小可用三节点集群来说，最自然的目标是：

- 三台主机分布三个 `MON`

### 为什么 `MON` 不能只留一个

因为 `MON` 负责的是集群关键状态与仲裁视图。只留一个 monitor，意味着：

- 没有真正仲裁冗余
- 控制面故障域太集中
- 一旦这台机器失联，集群的控制面可用性就会非常脆弱

### `cephadm` 对 monitor 的自动化思路

官方文档指出，`cephadm` 会随着集群规模增长自动补充 monitor，默认最多部署到 5 个，但这依赖正确的子网配置。

这句话背后有两个实践含义：

- monitor 自动扩展不是“凭空发生”，它依赖你前面网络规划没出错
- 即使你依赖自动化，也必须知道它为什么会落到那些主机和网段上

如果你想显式指定 monitor 所在网络或 placement，也可以通过 `ceph config set mon public_network ...` 以及 `ceph orch apply mon` / `ceph orch daemon add mon` 这种方式控制。

本文不展开复杂 mon 迁移，只要求你先建立一个结论：

**最小可用集群不应该长期停留在单** **`MON`** **状态。**

## 第五步：部署 `OSD`，这一步才让集群真正开始“能存数据”

很多新手在这里会有一个强烈落差：前面 bootstrap、加主机、看 Dashboard 都很顺，但为什么还是没有真实存储能力？

答案很简单：因为数据面核心是 `OSD`，而不是 `MON` 或 `MGR`。

### 先看设备，再谈部署

仓库文档里首先强调的是查看设备清单：

```bash
ceph orch device ls
```

这一点非常合理，因为 `cephadm` 不是盲目“见盘就上”，它会判断设备是否可用。

官方文档中，一个设备被视为 available，至少要满足这些条件：

- 没有分区
- 没有 LVM 状态
- 没有挂载
- 没有文件系统
- 不是已有 BlueStore OSD
- 大于 5GB

这组条件值得你认真记住，因为很多“为什么 OSD 起不来”的问题，本质上都不是 Ceph 命令错了，而是设备根本不符合条件。

### 最简单的 OSD 编排方式

对最小可用集群，最常见的方式是：

```bash
ceph orch apply osd --all-available-devices
```

它的优点是快，适合：

- 实验环境
- 盘型单一
- 你确认所有空盘都应该吃进去

但它的风险也很明显：

- 粒度粗
- 不适合复杂的混合介质设计
- 不适合你还没规划好哪些盘做 data、哪些盘做 DB/WAL 的场景

### 更谨慎的方式

文档里也给了指定主机和设备的做法，例如：

```bash
ceph orch daemon add osd host1:/dev/sdb
```

这类方式更适合你在文章、教学和真实环境中保持边界清晰：

- 你知道哪块盘给哪个节点
- 你不会因为“全可用设备”误吃掉不该吃的盘
- 你更容易把角色规划和硬件规划联系起来

### 为什么说部署 OSD 才是真正跨入“可用存储”

因为到了这一步：

- 集群才有真实数据副本承载体
- pool 才有落地的对象存储位置
- 后面不管是 `RBD`、CephFS 还是 `RGW`，底层才有真正的对象存储执行面

一句话概括：

**没有** **`OSD`，前面的控制面再漂亮，也只是一个“知道自己是谁”的集群，不是一个真正能存用户数据的集群。**

## 第六步：确认 `MGR`、Dashboard 与监控栈状态

到这里很多人会问：既然 bootstrap 时已经创建了 `MGR`，那 Dashboard、Prometheus、Grafana 是不是也已经有了？

答案是：**在** **`cephadm`** **管理的集群里，监控栈默认会部署，除非你在 bootstrap 时显式使用了** **`--skip-monitoring-stack`。**

这点非常关键，因为它直接影响你对“最小可用”的理解。

在现代 Ceph 里，所谓“最小可用”，通常不只是：

- 能写入
- 能读出

还应该至少包含：

- 能看健康状态
- 能看容量与性能指标
- 能观察告警

### 监控栈默认包含什么

根据仓库文档，cephadm 监控栈主要包括：

- Prometheus
- Node Exporter
- Alertmanager
- Grafana

而 Dashboard 则是 `ceph-mgr` 的一个管理与可视化模块，它本身不是独立的存储守护进程，但会与 Prometheus/Grafana 深度配合。

### 如果 bootstrap 时没带监控栈怎么办

文档里给了补装方式：

```bash
ceph orch apply node-exporter
ceph orch apply alertmanager
ceph orch apply prometheus
ceph orch apply grafana
```

这组命令很值得记，因为它清楚说明：

- 监控也是通过 orchestrator 管理的服务
- 它们和 `OSD`、`MDS`、`RGW` 一样，都属于可编排对象
- Ceph 管理面和可观测性是集成思路，不是部署后再“外接一个世界”

### Dashboard 在整个部署故事里的位置

根据 `doc/mgr/dashboard.rst`，Ceph Dashboard 是一个 `ceph-mgr` 模块，承载 Web 化管理与监控视图。

这意味着：

- Dashboard 建立在 `MGR` 之上
- 它不是单独的核心数据面组件bbb 
- 它对运维极其重要，但不参与普通 IO 中转

这一点对前两篇的架构认知是一次非常好的落地验证。

## 第七步：按需启用 CephFS，也就是引入 `MDS`

初学者在第 3 篇最容易犯的错误，就是试图“一次性把所有服务都起全”。这在概念上看起来完整，在工程上却经常会让人失去重点。

CephFS 就是一个典型例子。

### 不需要 CephFS 时，根本不用着急起 `MDS`

如果你的当前目标只是：

- 先把对象底座跑通
- 后面给 `RBD` 用
- 先建立 OSD/monitoring/cephadm 基础操作模型

那完全可以先不部署 `MDS`。

### 需要 CephFS 时怎么理解它的部署

仓库里的 `cephadm` 文档明确提到：

- 使用 `ceph fs volume create <fs_name>` 创建 CephFS 时
- 会自动创建所需的 MDS 守护进程

例如：

```bash
ceph fs volume create cephfs --placement="label:mds"
```

这条命令的意义不只是“创建文件系统”，而是同时说明：

- CephFS 不是 bootstrap 就天然存在的
- `MDS` 是按文件系统需求引入的
- placement 可以通过标签控制

这正好验证我们前一篇讲过的角色边界：

**`MDS`** **不是 Ceph 全局必选组件，而是 CephFS 这条语义链路的专用元数据面角色。**

## 第八步：按需启用 `RGW`，也就是对象网关层

和 `MDS` 一样，`RGW` 也不是 bootstrap 时就必须部署的角色。

如果你当前目标是：

- 跑一个 `RBD` 后端
- 或先做 CephFS 验证

那完全不需要马上引入 `RGW`。

但如果你想给外部应用提供 S3/Swift 风格接口，就要把 `RGW` 加进来。

### 最简单的 RGW 部署方式

仓库文档给出的最简单形式是：

```bash
ceph orch apply rgw foo
```

它代表：

- 部署一个 service id 为 `foo` 的 RGW 服务
- 对单集群、非 multisite 的基础场景足够友好

### 为什么实际环境推荐用标签和 placement

文档里也给出了更典型的网关主机做法，例如先给网关主机打 `rgw` 标签，再用 placement 部署：

```bash
ceph orch host label add gwhost1 rgw
ceph orch host label add gwhost2 rgw
ceph orch apply rgw foo '--placement=label:rgw count-per-host:2' --port=8000
```

这背后的工程思想非常重要：

- 不要把所有服务随意混在所有节点上
- 用标签把角色边界显式化
- 用 placement 把“服务应该落哪”表达成可持续管理的配置

对初学者来说，就算当前只有 3 台机器，也最好尽早养成这种思维。

## `cephadm` 的核心价值，不是“自动”，而是“可编排”

写到这里，你会发现 `cephadm` 的真正价值已经很清楚了。

它不是简单帮你执行下面这些散乱动作：

- 起容器
- 拷文件
- 复制密钥
- 安装监控

它真正提供的是一种统一的运维模型：

- 主机被纳入 orchestrator 管理
- 服务通过 service spec 和 placement 进行描述
- 监控、网关、MDS、OSD 都纳入同一套编排视角
- 集群从“按机器手工维护”转向“按角色和服务持续管理”

这也是为什么我建议你在学 Ceph 部署时，不要死记单条命令，而要反复问自己：

- 这是在加主机，还是在加服务？
- 这是在扩控制面，还是在扩数据面？
- 这是通用底座能力，还是某种上层接口能力？

只要你一直这样问，`cephadm` 的命令体系就会越来越清晰。

## 一个最小可用部署顺序的推荐主线

如果把前面的内容压缩成一条真正可执行的学习顺序，我会建议你按下面这个顺序去理解和操作。

### 阶段一：立起控制面入口

- 准备第一台主机基础环境
- 安装 `cephadm`
- 运行 `cephadm bootstrap --mon-ip <ip>`
- 确认第一个 `MON` / `MGR` 正常起来

### 阶段二：把主机边界纳管进来

- 通过 SSH key 纳管其他主机
- `ceph orch host add` 加入节点
- 按需加 `_admin` 和自定义标签

### 阶段三：立起真实数据面

- `ceph orch device ls` 看可用设备
- 明确哪些盘可被消费
- 用 `ceph orch apply osd ...` 或显式设备方式部署 OSD

### 阶段四：确认可观测性入口

- 检查 Dashboard
- 检查 Prometheus / Grafana / Alertmanager / Node Exporter
- 如果 bootstrap 时跳过监控栈，则补部署

### 阶段五：按业务需要扩接口层

- 需要文件系统时创建 CephFS，并引入 `MDS`
- 需要对象网关时部署 `RGW`

这条主线看似普通，但它非常符合 Ceph 的架构本质：

- 先有控制面
- 再有数据面
- 最后再有具体业务语义

## 三节点最小可用集群里，角色怎么放更合理

对于入门级三节点集群，你可以先采用一种“尽量均衡、但不装复杂”的布局思路：

### 基础建议

- 三个节点都纳入集群
- 三个节点都尽量承载 `MON`
- 三个节点都尽量承载 `OSD`
- 至少一个节点保留 `_admin`

### `MGR` 放法

- bootstrap 节点先承载 active `MGR`
- 后续可以再有 standby `MGR`

### `MDS` 放法

- 只有在需要 CephFS 时启用
- 优先放到你打了 `mds` 标签的节点

### `RGW` 放法

- 只有在需要对象网关时启用
- 如果条件允许，尽量用标签把网关节点和纯存储节点区分开

当然，在只有三台机器的情况下，很多角色最终还是会混部。这没有问题。关键是你要清楚：

- 混部是资源条件下的折中
- 不是角色边界不存在

## 初学者在第一个 Ceph 集群里最常踩的 8 个坑

### 1. 把单机实验当成高可用集群

单机只是演示环境，不代表真正理解了 Ceph 的仲裁、故障域和恢复逻辑。

### 2. bootstrap 完就以为“集群已经能存业务数据”

bootstrap 立的是起点，不是完整数据面。真正决定数据能力的是 `OSD`。

### 3. 没看设备状态就直接上 `--all-available-devices`

这在实验环境里方便，在真实环境里很容易误吃设备。

### 4. 把 `MDS` 和 `RGW` 当成开局必备组件

不是。它们只在对应业务语义需要时出现。

### 5. 忽略主机标签

早期偷懒不用标签，后期服务 placement 会越来越乱。

### 6. 网络地址选错

bootstrap 用错 `--mon-ip` 或主机纳管 IP 不明确，会让后续所有问题都更难查。

### 7. 忽略监控栈

Ceph 不是“能写就够了”的系统，没有指标、告警和图表，后续诊断几乎必然痛苦。

### 8. 一开始就追求“所有服务全部起齐”

这会分散注意力。先立起控制面和数据面，再按需引入 CephFS 或 RGW，学习曲线更稳。

## 这一篇最应该留下的 5 个结论

### 结论一：`cephadm` 先 bootstrap 一台主机，再逐步纳管整个集群

这就是今天理解 Ceph 部署的主入口。

### 结论二：最小可用集群应该以三节点为基本单位来理解

单机能演示，但不能代表真正的集群工程逻辑。

### 结论三：bootstrap 立的是控制面起点，`OSD` 才让集群真正开始“能存数据”

这是从“看起来部署成功”到“真的可用”的关键分界。

### 结论四：`MDS` 和 `RGW` 都是按业务需求追加的能力层

它们不是所有 Ceph 集群都必须从 Day 1 启用。

### 结论五：Prometheus、Grafana、Dashboard 不是锦上添花，而是最小可用体验的重要组成

现代 Ceph 的“可用”，不仅是可读写，也应包含基本可观测性。
