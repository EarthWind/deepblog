# RBD 性能测试实战：基准方法、压测工具与指标拆解

## 这篇文章要解决什么问题

前面几篇我们已经把 `RBD` 的架构、IO 路径和高级特性讲得比较完整了。

但一旦进入真实使用阶段，几乎每个人都会立刻碰到另一个问题：

**RBD 的性能到底该怎么测？**

这件事看起来像“跑个 benchmark 就行”，实际上是整个 Ceph 使用过程中最容易踩坑的部分之一。

因为你很快就会遇到一连串让人混乱的问题：

- 用 `rbd bench` 还是 `fio`？
- 应该测 `librbd` 路径、`krbd` 路径，还是 `rbd-nbd` 路径？
- 在宿主机上测 `/dev/rbdX`，和在 QEMU 里测虚拟机磁盘，是不是一回事？
- IOPS、吞吐、时延、队列深度分别该怎么看？
- 为什么同一个集群，换个 `bs` 或 `iodepth` 结果会差这么多？
- 为什么有时测出来的不是 Ceph，而是客户端缓存、宿主机缓存、磁盘缓存甚至网络瓶颈？

如果只允许用一句话先给结论，那就是：

**RBD 性能测试的核心不是“跑哪个命令”，而是先明确你到底在测哪一条 IO 路径、哪一种负载形态，以及最终结果受哪一层瓶颈主导；只有把路径、负载、缓存、并发和指标口径统一起来，测试结果才有可比性。**

这一篇的目标，就是把这件事讲清楚。

## 先建立第一条边界：RBD 性能测试不是一个命令，而是一组“测试设计”

这是理解整篇文章的第一步。

很多人刚开始做性能测试时，会把关注点放在：

- 该执行 `rbd bench`
- 还是该执行 `fio`

这个问题当然重要，但还不够根本。

更根本的问题其实是：

- 你到底在测什么？

比如你可能是在测：

- `librbd` 用户态路径
- `krbd` 内核态路径
- `rbd-nbd` 用户态块映射路径
- QEMU 虚拟机里的磁盘路径

而这些路径哪怕最终都访问同一个 image，测出来的结果也未必能直接横向比较。

所以最重要的第一原则是：

- **先定义测试对象，再定义测试工具。**

## 第 1 层：RBD 性能测试到底在测哪几条路径

如果把 RBD 常见测试路径压缩成最有代表性的几类，大致可以分成下面四种。

## 1. `rbd bench`

- 适合做 RBD 快速基线摸底
- 更像“RBD 自带的快速基准工具”

## 2. `fio + librbd`

- 直接走用户态 RBD engine
- 更接近应用或用户态直接接 `librbd` 的路径

## 3. `fio + krbd`

- 先把 image map 成 `/dev/rbdX`
- 再通过 Linux block layer 压这个块设备

## 4. `fio + rbd-nbd`

- 先把 image 通过 `rbd-nbd` 映射成块设备
- 再在用户态块设备路径上做压测

## 5. QEMU / 虚拟机场景

- 通过 QEMU 直接走 `librbd`
- 或者通过宿主机映射后的块设备再给虚机使用

这几条路径最大的区别，不是命令行长得不一样，而是：

- IO 在哪一层被提交
- 经过哪些缓存
- 经过用户态还是内核态
- 会不会经过额外块层和虚拟化层

所以如果不先把路径讲清楚，后面的测试结论基本都会混。

## 第 2 层：为什么 `rbd bench` 适合做“快速摸底”，但不适合替代系统化压测

官方文档里对 `rbd bench` 的定位非常直接：

- 它用于测 IO 吞吐和时延

这意味着它非常适合做一件事：

- **快速摸底**

比如：

- 这个 image 当前顺序写大概有多快
- 改一下 `--io-size` 或 `--io-threads` 后有什么变化
- 集群大体有没有明显异常

### 为什么它很好用

因为：

- 上手快
- 参数少
- 能快速给出结果

### 但为什么它不够

因为它不够灵活，也不够全面。

比如在很多真实场景里，你关心的是：

- 4K randread
- 8K randwrite
- 70/30 混合读写
- 不同 `iodepth`
- 不同 `numjobs`
- 不同 IO engine

这些更细粒度的测试控制，`fio` 会更合适。

所以最稳妥的定位应该是：

- `rbd bench` 适合快速摸底和回归验证
- `fio` 更适合正式基准测试和实验设计

## 第 3 层：为什么 `fio` 才是 RBD 性能测试的主力工具

如果要做严肃一点的压测，主力几乎一定是：

- `fio`

### 为什么

因为你真正需要控制的维度太多了，比如：

- `rw`
- `bs`
- `iodepth`
- `numjobs`
- `runtime`
- `ioengine`
- `direct`
- `verify`

这些参数共同决定：

- 你到底是在测顺序吞吐
- 还是随机 IOPS
- 是在测单线程低延迟
- 还是测高并发下极限能力

### 这意味着什么

意味着：

- 没有脱离负载参数的“RBD 性能”

你测到的永远是：

- 某种路径
- 在某种负载模型下
- 呈现出的某种性能结果

这也是为什么写性能测试文章时，必须把方法学放在工具前面。

## 第 4 层：为什么一定要分清 `fio + librbd`、`fio + krbd`、`fio + rbd-nbd`

这是整篇最重要的一个实践边界。

很多人会以为：

- 反正都是压同一个 image
- 最终结果应该差不多

这个想法非常危险。

更准确地说，这三条路径测到的东西明显不同。

## `fio + librbd`

- 直接走用户态 `librbd`
- 更接近用户态应用直接接 Ceph 的路径
- 少了 Linux block layer 这一层真实设备抽象

## `fio + krbd`

- 先映射成 `/dev/rbdX`
- 再走 Linux block layer
- 更像宿主机把 RBD 当普通块设备来用

## `fio + rbd-nbd`

- 映射路径不同于 `krbd`
- 用户态桥接更明显
- 一些参数和特性行为也会有所不同

### 所以它们最大的区别是什么

不是“哪个更先进”，而是：

- 你到底想模拟哪个真实生产路径

如果业务最终是：

- QEMU 直接通过 `librbd` 访问 RBD

那你只测 `/dev/rbdX`，很多时候并不能真实代表生产。

反过来，如果业务最终是：

- 宿主机或裸机文件系统直接挂 `/dev/rbdX`

那直接测 `fio rbd` 也不完全对路。

## 第 5 层：QEMU 场景为什么不能简单等同于“测宿主机块设备”

这是虚拟化场景里特别容易犯的错误。

很多人为了图方便，会这样做：

- 先把 image map 成 `/dev/rbdX`
- 然后再把这个块设备给虚拟机
- 最后认为自己测到了“Ceph 虚拟机盘性能”

这不一定错，但它不是唯一也不是最常见的真实路径。

现代 Ceph 虚拟化场景里，经常更接近真实生产的路径其实是：

- QEMU 直接通过 `librbd` 使用 RBD image

### 这意味着什么

意味着：

- QEMU 场景下的性能测试，最好优先考虑你真实上线时的接入方式

不要把：

- “宿主机块设备压测结果”

直接等同于：

- “虚拟机实际块设备体验”

## 第 6 层：为什么 IOPS、吞吐、时延不是一个指标换个单位那么简单

很多初学者看 benchmark 结果时，第一反应是：

- IOPS 高就快
- 吞吐高就强
- 时延低就好

这些话都不算错，但太粗糙。

更准确地说，这三个指标反映的是不同维度。

## IOPS

- 更适合描述小 IO、高频随机访问能力

### 典型场景

- 4K randread
- 4K randwrite

## 吞吐

- 更适合描述大块顺序 IO 的总带宽能力

### 典型场景

- 1M 或 4M 顺序读写

## 时延

- 更适合描述单次 IO 从提交到完成的等待成本

### 典型场景

- 延迟敏感业务
- 小 IO 数据库负载
- 用户体验型业务

### 所以最重要的结论是什么

- **不要拿顺序大块吞吐去证明小随机 IOPS 能力**
- **也不要拿极限 IOPS 去掩盖时延已经很差的现实**

这三者必须结合负载模型一起看。

## 第 7 层：为什么队列深度 `iodepth` 是最容易被低估的参数

如果在所有 `fio` 参数里只挑一个最容易把结果彻底改写的，我会选：

- `iodepth`

### 为什么

因为它决定了：

- 允许多少未完成 IO 同时在途

这会直接影响：

- 并发度
- 吞吐
- IOPS
- 时延形态

### 一个最关键的直觉

低 `iodepth` 更像：

- 单请求、低并发、时延更真实的路径

高 `iodepth` 更像：

- 尽量把设备和集群打满、看极限吞吐或 IOPS

所以你必须先问自己：

- 你是在测真实业务延迟体验
- 还是在测系统打满后的能力上限

这两个问题的答案，可能对应完全不同的 `iodepth` 选择。

## 第 8 层：为什么 `iodepth` 和 `queue_depth` 不能分开看

一旦进入 `krbd` 或块设备路径，还有另一个常被忽视的参数：

- `queue_depth`

文档里对 `krbd` 映射就明确提到：

- `queue_depth` 默认是 128

### 这说明什么

说明你的 `fio` 里即使把 `iodepth` 配得很高，实际链路上还会受到：

- 块设备队列
- 驱动层队列

等限制。

Ceph QA 里的 `rbd_fio` 任务之所以还会同步去调整：

- `/sys/block/.../nr_requests`

本质上也是在承认：

- 应用层并发和块层并发不是一回事

所以当你测块设备路径时，必须同时关心：

- `fio` 的 `iodepth`
- 设备映射层的 `queue_depth`
- 块层请求队列限制

## 第 9 层：为什么 `bs` 会决定你测到的是“哪一个世界”

这是另一个非常容易误判的点。

很多人只改了一个 `bs`，结果差了一大截，就会觉得：

- Ceph 性能不稳定

其实更可能是：

- 你已经在测完全不同的路径特征了

### 小 `bs` 比如 4K、8K

更容易测出：

- IOPS
- 随机读写
- 元数据与调度开销占比
- 时延敏感性

### 大 `bs` 比如 1M、4M

更容易测出：

- 顺序吞吐
- 大块传输效率
- 网络和磁盘总带宽瓶颈

所以 `bs` 并不是一个微调参数，而是：

- 你到底在模拟哪种应用 workload

的核心定义之一。

## 第 10 层：为什么很多“测出来很快”的结果，其实测到的是缓存

这是性能测试里最大的坑之一。

RBD 路径上可能影响结果的缓存非常多，比如：

- `librbd` 客户端缓存
- QEMU cache
- Linux page cache
- 设备写缓存
- 控制器缓存

### 这意味着什么

意味着如果你不先讲清缓存策略，就很可能出现：

- 吞吐看起来很好
- 但你测到的不是 Ceph 集群真实持久化能力

而是：

- 某层缓存的吸收能力

这也是为什么 Ceph 文档会非常强调：

- `librbd` cache 的工作模式
- QEMU cache 与 `rbd_cache` 的对应关系
- 磁盘写缓存对 IOPS 和 commit latency 的影响

## 第 11 层：为什么测试前要先声明缓存策略

这一点特别值得写成实战原则。

比如至少应该明确：

- 是否开启 `librbd` cache
- QEMU 侧是什么 cache 模式
- fio 是否用 `direct=1`
- 宿主机 page cache 是否被绕过
- 底层设备写缓存状态是什么

### 为什么必须这样

因为如果你今天测：

- `cache=writeback`

明天又测：

- `cache=none`

却把结果放在一张图里横向比较，那结论几乎一定不靠谱。

所以性能测试文章里一定要有一句很实用的话：

- **不声明缓存策略的 RBD 性能结果，基本不可复现，也不可比较。**

## 第 12 层：为什么 `rbd bench`、`fio rbd engine`、`fio /dev/rbdX` 应该分别服务不同目标

现在可以把几个常见工具重新放回统一框架里了。

## 场景一：快速摸底

优先：

- `rbd bench`

### 适合回答的问题

- 当前 image 顺序写大概多少
- 改线程数、块大小后有无明显异常

## 场景二：测用户态 `librbd` 路径

优先：

- `fio` 的 `rbd` engine

### 适合回答的问题

- 用户态直连 RBD 的应用行为如何
- 不经过块设备映射时的性能如何

## 场景三：测宿主机块设备路径

优先：

- `fio --filename=/dev/rbdX`

### 适合回答的问题

- Linux block layer + `krbd` 路径表现如何

## 场景四：测 `rbd-nbd`

优先：

- `fio --filename=/dev/nbdX`

### 适合回答的问题

- 用户态 NBD 映射路径表现如何

## 场景五：测虚拟机场景

优先：

- 按真实上线方式测试 QEMU / libvirt / `librbd`

### 适合回答的问题

- VM 真正看到的磁盘体验如何

### 这说明什么

说明你不该问：

- “哪个工具最好？”

而应该问：

- “哪条路径才是我要模拟的真实生产路径？”

## 第 13 层：为什么 Ceph 官方会反复强调基准测试前先测硬件

这一点非常现实。

Ceph 文档里对底层盘性能的建议非常明确：

- 用 `fio` 去评估磁盘 4K 随机写
- 关注磁盘缓存、持续性能、PLP、耐久度

### 为什么这和 RBD 测试直接相关

因为很多时候，你以为自己在测：

- Ceph 很慢

实际测到的可能是：

- 单盘随机写能力不够
- SSD cliffing
- 磁盘写缓存策略不合理
- 网络带宽已经成瓶颈

所以一条非常实用的测试原则是：

- **先知道底层盘和网络的理论边界，再解释 Ceph 集群结果。**

否则很容易把存储集群问题和硬件上限问题混为一谈。

## 第 14 层：为什么“网络带宽”经常在大块吞吐测试里先撞墙

很多人做 1M、4M 顺序写时，看到吞吐不再上涨，就会先怀疑：

- OSD
- BlueStore
- PG 分布

这些当然可能有关，但文档提醒得很清楚：

- 单机磁盘总吞吐不要超过网络能力，否则最终会被网络限制

### 这意味着什么

如果你做的是大块顺序吞吐测试，那么：

- CPU
- 磁盘
- 网络

三者里谁先到顶，谁就会主导结果。

所以吞吐测试不能只盯存储端，还必须问：

- 你的 NIC 多大？
- 客户端和 OSD 间有没有 oversubscription？
- 单节点汇聚带宽和集群网络能否承住？

## 第 15 层：为什么“测试设计”比“命令模板”更重要

讲到这里，其实能看出一个非常重要的结论：

- 真正决定结果是否可信的，不是你会不会写 fio 命令

而是：

- 你有没有先定义 workload
- 你有没有明确路径
- 你有没有固定缓存策略
- 你有没有固定 `bs`、`iodepth`、`numjobs`
- 你有没有区分热身和正式采样
- 你有没有知道瓶颈大概在哪一层

也就是说，性能测试更像：

- 一次实验设计

而不是：

- 一条 shell 命令

这也是为什么一篇真正有价值的性能测试文章，必须把方法论写在前面。

## 第 16 层：一套实用的 RBD 测试顺序应该长什么样

如果把实践方法压缩成最实用的一套流程，我建议这样：

## 第一步：先定义目标路径

比如：

- 测 `librbd`
- 测 `krbd`
- 测 QEMU

## 第二步：先定义 workload

比如：

- 4K randread
- 4K randwrite
- 70/30 randrw
- 1M seqread
- 1M seqwrite

## 第三步：固定缓存策略

比如：

- 是否开 `librbd` cache
- 是否 `direct=1`
- QEMU cache 模式
- 底层设备写缓存状态

## 第四步：固定并发参数

比如：

- `iodepth`
- `numjobs`
- `io-threads`
- `queue_depth`

## 第五步：先做快速摸底，再做正式压测

比如：

- 先 `rbd bench`
- 再 `fio`

## 第六步：按指标分类看结果

比如：

- 小随机看 IOPS + 时延
- 大顺序看吞吐 + 时延
- 高并发看尾延迟和抖动

这套流程不花哨，但非常实用。

## 第 17 层：一张最实用的“工具与场景匹配表”

如果这篇只想留一张最实用的表，可以记成下面这样：

```text
快速摸底:
  rbd bench

用户态 RBD 路径:
  fio + rbd engine

内核块设备路径:
  fio + /dev/rbdX

用户态 NBD 映射路径:
  fio + /dev/nbdX

虚拟机真实场景:
  QEMU / libvirt 按真实上线方式测试
```

这张表最重要的作用不是记命令，而是提醒你：

- 工具要服务于路径

## 用一句话重新概括 RBD 性能测试

如果把这篇全部内容压缩成一句尽量准确的话，我会这样说：

**RBD 性能测试的核心不是“跑一个 benchmark”，而是先明确测试路径、负载模型、缓存策略和并发口径，再根据目标选择 `rbd bench`、`fio + librbd`、`fio + krbd`、`fio + rbd-nbd` 或 QEMU 实测；只有这样，IOPS、吞吐、时延和队列深度这些指标才真正有解释力。**

## 把整篇压缩成一条最短骨架

如果你只想记一条骨架，可以记成下面这样：

```text
先确定测哪条路径
  ->
再确定 workload（bs/rw/iodepth/numjobs）
  ->
再固定缓存策略
  ->
先用 rbd bench 快速摸底
  ->
再用 fio 做正式压测
  ->
按 IOPS/吞吐/时延/队列深度 分维度解读结果
  ->
同时检查底层盘和网络是不是先撞瓶颈
```

只要这条链不乱，你做出来的 RBD 性能测试通常就已经比多数“随手跑一条 benchmark”靠谱很多。

## 初学者最容易混淆的 10 个点

### 1. 认为 RBD 性能测试就是挑一个工具跑一下

不对。首先是实验设计问题。

### 2. 认为 `rbd bench` 能替代系统化压测

不对。它更适合快速摸底。

### 3. 认为 `fio + librbd` 和 `fio + /dev/rbdX` 测的是同一条路径

不对。一个偏用户态，一个偏内核块设备路径。

### 4. 认为宿主机块设备测试就等于虚拟机场景测试

不对。QEMU 常直接走 `librbd`。

### 5. 认为 IOPS 高就说明一切都好

不对。还要看时延、吞吐和 workload 类型。

### 6. 认为 `iodepth` 只是小调参

不对。它几乎会重写整个测试结果的形态。

### 7. 认为不声明缓存策略也能比较结果

不对。这会让结果失去可复现性。

### 8. 认为大块吞吐测不起来一定是 Ceph 本身问题

不对。网络带宽经常先撞墙。

### 9. 认为集群性能一定是集群软件主导

不对。单盘能力、写缓存、SSD 持续性能都可能先成瓶颈。

### 10. 认为同一个 image 在不同接入路径上的结果可以直接横向比较

不对。路径不同，结果天然就可能不同。

## 这一篇最应该留下的 5 个直觉

### 直觉一：先定义路径，再选择工具

这是第一原则。

### 直觉二：`fio` 是正式压测主力，`rbd bench` 是快速摸底工具

两者别混用角色。

### 直觉三：IOPS、吞吐、时延分别回答不同问题

不要把它们混成一个“性能”概念。

### 直觉四：`iodepth`、`queue_depth`、缓存策略会深刻改变测试结果

这些参数必须显式声明。

### 直觉五：很多所谓 Ceph 性能问题，其实先撞的是底层盘或网络

这点特别容易被忽视。
