# CephFS 性能问题排查案例：元数据热点、目录倾斜与 MDS 饱和如何定位

## 这篇文章要解决什么问题

前一篇我们已经把 CephFS 的调优主线讲清楚了：

- 先分元数据链和数据链
- 元数据链重点看客户端 caps/session、MDS cache、目录分布和多 MDS
- 数据链重点看网络、RADOS、OSD、BlueStore 和硬件布局

但即使这些原理都懂了，真正到了线上排障时，很多人还是会立刻陷进另一个更现实的问题：

**CephFS 变慢时，到底该先看哪里？**

也就是：

- 看到 `MDS_SLOW_REQUEST` 以后，下一步到底该干什么？
- 怎么判断是热点目录把某个 MDS 打爆了，还是客户端没及时释放 caps？
- 目录倾斜和 MDS cache 压力，会在指标和症状上长成什么样？
- 明明是 CephFS 慢，为什么有时根因其实在 OSD 或网络？
- `dump_ops_in_flight`、`ceph fs perf stats`、`client ls`、debugfs 这些工具，到底应该按什么顺序看？

如果只允许用一句话先给结论，那就是：

**CephFS 性能排障最怕“看到慢就到处看指标”，最有效的方法是沿着“症状 -> MDS 当前在等什么 -> 热点目录还是 cache pressure -> 客户端会话是否配合 -> 后端 OSD/网络是否拖累”这条链逐层收缩；只有把慢请求、热点目录、caps recall、session 状态和后端 RADOS 观测真正串起来，问题才能稳定定位。**

这一篇的目标，就是把这条排障路径真正讲清楚。

## 先建立第一条边界：CephFS 的“慢”不是一个问题，而是一组不同形态的症状

这是理解整篇文章的第一步。

很多时候大家会笼统地说：

- CephFS 慢

但这个描述其实太粗了。

CephFS 常见的“慢”，至少有下面几种完全不同的症状形态：

## 症状一：元数据操作慢

- create 慢
- stat 慢
- readdir 慢
- rename/unlink 卡

## 症状二：大文件数据 IO 慢

- 顺序吞吐上不去
- 随机读写抖动
- fsync 很慢

## 症状三：偶发卡顿

- 平时还行，偶尔突然整段请求都慢
- 目录热点时明显卡住
- 某一波客户端起来后延迟飙升

### 为什么必须先分这些症状

因为它们背后的根因层次完全不同：

- 元数据症状更偏 `MDS`
- 数据症状更偏 `OSD/网络/BlueStore`
- 偶发卡顿经常和 cache pressure、recall、恢复流量、热点目录迁移叠在一起

所以排障的第一原则不是看参数，而是：

- **先把“慢”翻译成更具体的症状类型。**

## 第 1 层：看到 `MDS_SLOW_REQUEST`，第一反应不该是调参数，而是先看 MDS 在等什么

这是整篇最实用的第一步。

很多人一看到：

- `MDS_SLOW_REQUEST`

第一反应就是：

- 调 `max_mds`
- 调 cache
- 调 recall

这些动作太早了。

更成熟的第一反应应该是：

- **先看当前慢请求到底卡在什么阶段。**

最直接的入口通常就是：

- `ceph daemon mds.<name> dump_ops_in_flight`

### 为什么这个入口如此关键

因为它不是在告诉你“系统慢了”，而是在告诉你：

- 具体有哪些请求还没完成
- 每个请求最后停在什么 event 上

这等于直接把“慢”从模糊现象，压缩成：

- 某一批具体请求正在等某个具体东西

这一步非常值钱。

## 第 2 层：为什么 `dump_ops_in_flight` 应该先看“最后一个 event”

很多人第一次看 `dump_ops_in_flight` 输出，会被大段信息吓住。

其实最有价值的切入点非常简单：

- 先看最后一个 event

### 为什么

因为最后一个 event 通常最接近：

- 当前请求真正卡住的位置

它可能暗示：

- 在等锁
- 在等 mdlog
- 在等 caps
- 在等 OSD
- 在等某个 peer MDS

### 这意味着什么

意味着 `dump_ops_in_flight` 的真正价值不是“列出一堆请求”，而是：

- **告诉你当前瓶颈更像落在哪一层。**

这会直接决定后面你该往：

- 热点目录
- cache pressure
- 客户端会话
- 后端 OSD

哪条线继续收缩。

## 第 3 层：怎么判断你遇到的是“热点目录”而不是“整个 MDS 都不行”

这是 CephFS 排障里最常见、也最容易误判的一类问题。

很多时候你看到：

- 某些 create/readdir 很慢
- 某些客户端特别卡
- 某个 MDS rank 特别忙

就会自然以为：

- 整个 MDS 不够了

其实很可能不是“整体不够”，而是：

- **某个目录或某段子树太热，已经形成元数据热点。**

### 热点目录的典型表现

- 单个目录下堆了大量文件
- 多客户端同时打同一目录
- create / unlink / readdir 都集中在同一棵子树
- 多 MDS 开着，但负载并没有真正打散

### 所以这里最重要的判断问题是

- 慢的是“整个名字空间”，还是“某个局部热点目录”？

这两个问题的调优路径完全不同。

## 第 4 层：为什么目录倾斜比“热点目录”更危险

热点目录和目录倾斜相关，但不完全一样。

### 热点目录

- 更偏某个目录特别热

### 目录倾斜

- 更偏整个目录树的负载分布严重不均

比如：

- 所有租户都往同一层级落
- 虽然目录很多，但真正活跃的都集中在极少数子树
- 多 MDS 环境下只有一两个 rank 特别忙

### 这说明什么

说明问题不只是：

- “一个目录太大”

而更可能是：

- **整个命名空间的热度分布方式不利于多 MDS 扩展。**

这时就不能只盯 dirfrag，而要一起看：

- subtree 分布
- pin 策略
- balancer 是否真的在工作

## 第 5 层：为什么 dirfrag 和多 MDS 不是“开了就自动救你”

这也是排障时特别容易误解的点。

很多人会想：

- 目录热，那我不是已经开 dirfrag 了吗？
- 多 MDS 也开了，为什么还是慢？

这个问题的核心在于：

- dirfrag 和多 MDS 都不是魔法，它们只在 workload 允许被拆分时才会发挥作用

### 什么时候它们可能没明显效果

- 热点仍然集中在一个很窄的目录访问模式里
- 目录虽大，但访问模式不利于拆分
- subtree 迁移跟不上热度变化
- 多 MDS 打开了，但目录树没有真正被打散

### 所以排障时应该问什么

- 当前热点有没有被拆到多个 dirfrag / subtree？
- 多 MDS rank 之间负载是不是明显失衡？
- `pin` / `ephemeral pin` / balancer 是否真的在起作用？

这一步比盲目加 `max_mds` 更重要。

## 第 6 层：为什么 MDS cache 压力往往会把“局部热点”放大成“整体卡顿”

这也是 CephFS 线上很典型的一种现象。

你原本可能只是有：

- 一个热点目录

但如果叠加上：

- MDS cache 已经吃紧

问题就会迅速恶化成：

- 更高的 recall
- 更多的 trim
- 命中率下降
- 更明显的 reply latency 抖动

### 所以什么症状很值得警惕

- `MDS_CACHE_OVERSIZED`
- `MDS_TRIM`
- 明显的 recall 压力
- 活跃文件集远超 MDS cache 预算

### 这说明什么

说明排障时要特别警惕一种错觉：

- 看起来像“某几个请求偶尔慢”

其实背后可能已经是：

- **MDS cache 正在整体失稳。**

## 第 7 层：怎么判断慢是因为客户端不响应 cache pressure

这是另一类特别常见的问题。

有时你会看到：

- MDS 持续 recall
- cache 压力高
- 告警里出现 `MDS_CLIENT_RECALL`
- 某些请求越来越慢

这时问题不一定是：

- MDS 自己 cache 太小

还可能是：

- 某些客户端一直不及时释放 caps

### 典型线索

- `MDS_HEALTH_CLIENT_LATE_RELEASE`
- `MDS_CLIENT_RECALL`
- 某几个 client 的会话状态很异常
- slow request 看起来像卡在 cap 收敛上

### 所以最重要的问题是

- MDS 是不是已经在回收，但客户端没有及时配合？

这是很多 CephFS 排障里极容易漏掉的一层。

## 第 8 层：为什么 `client ls` / `session ls` 在排障时特别值钱

一旦怀疑问题和客户端会话、caps 释放不及时有关，就必须把注意力切到：

- client/session 视角

最典型的入口就是：

- `ceph tell mds.<name> client ls`
- 或 `session ls`

### 为什么它们有价值

因为它们能告诉你：

- 当前到底有哪些客户端挂在这个 MDS 上
- 哪些 session 状态异常
- 哪些客户端可能长期占着大量状态
- 哪些 oldest tid、release 行为不正常

### 这说明什么

说明 CephFS 排障里，客户端不是“黑盒流量源”，而是：

- 一个必须被观测和管理的性能参与者

这和很多传统文件系统排障思路很不一样。

## 第 9 层：为什么有时必须用 eviction 思维解决排障问题

很多人一听到：

- evict client

会本能觉得这太激进。

但在某些 CephFS 故障里，它其实是一个非常现实的排障动作。

### 什么时候会考虑它

- 某个客户端会话已经明显异常
- 长时间不响应 recall
- 继续拖着会让 MDS 一直收不拢状态
- 整个文件系统服务持续被少数坏客户端拖住

### 为什么这时 eviction 是合理的

因为问题已经不只是：

- “这个客户端会不会受影响”

而是：

- “整个系统能不能恢复到可控状态”

所以 eviction 在 CephFS 排障里更像：

- 一种收敛手段

而不是单纯的粗暴操作。

## 第 10 层：为什么 kernel client 排障常常必须看 debugfs 而不是只看 Ceph 命令

一旦你面对的是：

- kernel client

就必须立刻切换一个排障习惯：

- 不能只靠 Ceph 用户态命令看问题

### 为什么

因为很多关键线索会直接出现在：

- `dmesg`
- `/sys/kernel/debug/ceph/...`

比如：

- `mdsc`
- `mds_sessions`
- `osdc`

这些状态文件能帮助你判断：

- 当前是卡在 MDS 协调
- 还是卡在 OSD 数据路径
- caps 是否 stale
- session 是否 reset

### 所以这里最重要的结论是什么

- **kernel client 出问题时，debugfs 往往是最靠近现场的真相入口。**

## 第 11 层：为什么 CephFS 慢有时其实是 OSD 或网络问题伪装出来的

这是整篇必须反复强调的一点。

因为从用户视角看，所有问题都表现为：

- 文件系统慢了

但根因层可以完全不同。

### 什么时候该高度怀疑后端

- 大文件数据 IO 也一起慢
- `dump_ops_in_flight` 看起来像在等 OSD
- kernel debugfs 里的 `osdc` 明显异常
- 集群有 `NEARFULL`
- OSD 有恢复、回填、慢盘、网络抖动

### 这说明什么

说明 CephFS 排障如果只在：

- MDS
- cache
- 目录树

这条线上死盯，很容易只看对一半。

你还必须问：

- 后端对象存储现在是不是也在抖？

这条线特别重要。

## 第 12 层：为什么 `NEARFULL` 这类后端状态会直接改写 CephFS 前端体验

这一点很值得单独点出来。

很多人会觉得：

- `NEARFULL` 是 OSD/容量问题
- 和 CephFS 性能关系不大

其实不是。

在 CephFS 场景里，后端接近满状态会直接改变一些前端行为，比如：

- 写路径可能退化得更同步
- 延迟明显变大

### 这意味着什么

说明某些看起来像“客户端 suddenly 变慢”的问题，真实原因可能只是：

- 后端集群已经不再处于健康写入区间

这就是为什么 CephFS 排障必须随时联动：

- CephFS 视角
- RADOS 健康视角

一起看。

## 第 13 层：一条最实用的 CephFS 排障顺序应该长什么样

讲到这里，可以把最实用的方法压缩成一套固定顺序。

## 第一步：先定义症状类型

比如：

- 元数据慢
- 数据慢
- 偶发卡顿
- 单目录热点

## 第二步：先看 CephFS health messages

比如：

- `MDS_SLOW_REQUEST`
- `MDS_SLOW_METADATA_IO`
- `MDS_TRIM`
- `MDS_CLIENT_RECALL`
- `MDS_CACHE_OVERSIZED`

## 第三步：用 `dump_ops_in_flight` 看慢请求最后卡在哪

这是最关键的分流点。

## 第四步：判断是不是目录热点 / 目录倾斜

比如：

- 单目录过热
- dirfrag 不够
- subtree 没打散
- 多 MDS 负载不均

## 第五步：判断是不是 cache pressure / 客户端不配合 recall

比如：

- recall 很多
- 某些 client/session 明显异常
- 需要看 `client ls` / `session ls`

## 第六步：如果像数据面问题，就切到 OSD / 网络

比如：

- OSD 慢
- recovery/backfill
- `NEARFULL`
- debugfs `osdc`
- 网络抖动

这套顺序是我认为最稳、也最不容易跑偏的一条主线。

## 第 14 层：一张最实用的“症状 -> 下一步看什么”图

如果这一篇只记一张图，我建议记下面这张：

```text
症状: create/stat/readdir 慢
  -> 先看 MDS_SLOW_REQUEST + dump_ops_in_flight + MDS cache/recall

症状: 单目录或单租户特别慢
  -> 先看热点目录 / dirfrag / subtree 分布 / 多 MDS 负载是否倾斜

症状: MDS 告警很多，但客户端也明显卡
  -> 先看 client ls / session ls / recall 是否迟迟收不回来

症状: 大文件读写也一起慢
  -> 先看后端 OSD / 网络 / osdc / NEARFULL

症状: 只有 kernel client 报问题
  -> 先看 dmesg + /sys/kernel/debug/ceph/*
```

这张图的价值不是绝对正确，而是提醒你：

- CephFS 排障最重要的是先做分流，而不是一上来全看。

## 第 15 层：为什么说 CephFS 真正的排障能力，不是会背指标，而是会“收缩问题空间”

这是整篇最想让读者留下的认知。

很多人排障时最大的问题不是：

- 不知道有哪些命令

而是：

- 不知道该先看哪个

CephFS 的复杂度决定了你不可能每次都：

- 从客户端、MDS、多 MDS、cache、OSD、网络、BlueStore 全部同时展开

真正成熟的排障方法，一定是：

- 先按症状分类型
- 再看慢请求卡点
- 再决定是走元数据链还是数据链
- 再继续收缩到热点目录、客户端会话或后端集群

也就是说，CephFS 排障真正难的不是：

- “工具不够多”

而是：

- **能不能快速缩小问题空间。**

这就是这篇文章最想交付的东西。

## 用一句话重新概括这篇

如果把这篇全部内容压缩成一句尽量准确的话，我会这样说：

**CephFS 性能排障最有效的方法，不是看到告警就调参数，而是先把“慢”分解成元数据慢、热点目录、cache pressure、客户端会话异常或后端 OSD/网络拖累，再通过 `dump_ops_in_flight`、health messages、`client/session ls`、debugfs 和后端观测逐层收缩问题空间；只有这样，元数据热点、目录倾斜与 MDS 饱和才会从模糊现象变成可定位的根因。**

## 把整篇压缩成一条最短骨架

如果你只想记一条骨架，可以记成下面这样：

```text
先定义症状类型
  ->
看 CephFS health messages
  ->
用 dump_ops_in_flight 看 MDS 在等什么
  ->
判断是热点目录 / cache pressure / 客户端不配合 recall
  ->
如果不像纯元数据问题，就切到 OSD / 网络 / debugfs
  ->
逐层收缩到真正根因
```

只要这条骨架记住了，CephFS 排障基本就不会一上来乱掉。

## 初学者最容易混淆的 10 个点

### 1. 认为 CephFS 慢就是一个问题

不对。症状类型必须先分开。

### 2. 认为看到 `MDS_SLOW_REQUEST` 就该先调参数

不对。先看请求到底卡在哪。

### 3. 认为热点目录等于整个 MDS 不够

不对。局部热点和整体饱和不是一回事。

### 4. 认为开了多 MDS 就自动不会有目录倾斜

不对。目录树分布方式仍然决定热点是否能被打散。

### 5. 认为 MDS cache 压力只是内存问题

不对。它会直接放大时延和 recall。

### 6. 认为客户端会话只是背景信息

不对。客户端不释放 caps 常常是问题放大的关键因素。

### 7. 认为 eviction 只是粗暴运维动作

不对。它有时是必要的收敛手段。

### 8. 认为 kernel client 排障只看 Ceph 命令就够了

不对。debugfs 和 `dmesg` 经常更关键。

### 9. 认为 CephFS 慢和 OSD/网络无关

不对。数据面问题经常根因在后端。

### 10. 认为排障能力等于知道很多命令

不对。核心是会不会快速收缩问题空间。

## 这一篇最应该留下的 5 个直觉

### 直觉一：排障先分症状，不要一上来混着看

这是第一原则。

### 直觉二：`dump_ops_in_flight` 是最值钱的分流入口之一

这点必须立住。

### 直觉三：热点目录和目录倾斜要单独判断

不要直接等同于“多加 MDS”。

### 直觉四：客户端会话和 recall 行为经常是性能问题的放大器

这一层特别容易漏。

### 直觉五：CephFS 的很多“慢”，真正根因会落到后端 RADOS/OSD

不能只盯 MDS。
