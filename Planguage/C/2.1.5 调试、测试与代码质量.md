# 调试、测试与代码质量

很多初学者以为“写出能跑的代码”就结束了，但真正的编程能力，很大一部分来自排错、验证和重构。这个阶段的重点是让你从“写代码的人”进一步成长为“能维护代码的人”。

## 本阶段目标

- 理解常见编译错误、运行时错误和逻辑错误的区别。
- 学会使用断言、日志和边界检查。
- 建立基础的代码质量意识。

## 核心知识

### 1. 三类常见问题

#### 编译错误

代码连编译都过不去，例如：

- 少分号
- 类型不匹配
- 函数未声明

#### 运行时错误

程序编译通过，但运行时崩溃，例如：

- 空指针访问
- 数组越界
- 非法内存读写

#### 逻辑错误

程序能运行，但结果不对，例如：

- 条件写错
- 循环边界错误
- 算法思路错误

### 2. `assert` 的价值

断言用于表达“程序运行到这里时，这个条件必须为真”。

它非常适合：

- 检查关键前提
- 发现不应该发生的状态
- 在开发阶段尽早暴露问题

### 3. 日志与调试输出

当问题复杂到无法直接肉眼看出时，日志是最直接的观察手段。

例如打印：

- 输入参数
- 关键变量变化
- 分支走向

### 4. 代码质量的基础标准

一段代码至少应做到：

- 命名清晰
- 函数职责明确
- 边界处理清楚
- 错误路径可读

## 可运行示例：用 `assert` 检查前提

```c
#include <stdio.h>
#include <assert.h>

int divide(int a, int b) {
    assert(b != 0);
    return a / b;
}

int main(void) {
    printf("%d\n", divide(10, 2));
    return 0;
}
```

这个程序说明：如果一个条件是函数正常工作的前提，那么可以用断言明确表达。

## 可运行示例：边界检查 + 日志

```c
#include <stdio.h>

int get_element(int arr[], int size, int index, int *out) {
    printf("get_element: size=%d, index=%d\n", size, index);

    if (index < 0 || index >= size) {
        printf("index out of range\n");
        return 0;
    }

    *out = arr[index];
    return 1;
}

int main(void) {
    int arr[3] = {10, 20, 30};
    int value = 0;

    if (get_element(arr, 3, 1, &value)) {
        printf("value = %d\n", value);
    }

    return 0;
}
```

这个例子体现了两个很实用的习惯：

- 进入函数时打印关键参数
- 在访问数组前先做边界检查

## 代码质量建议

### 1. 函数短一点，职责单一点

一个函数最好只做一件相对清晰的事。后续定位问题会容易很多。

### 2. 名字表达意图

例如：

- `calc_total_score`
- `read_student_file`
- `is_valid_index`

这样的命名比 `foo`、`temp1` 更有维护价值。

### 3. 错误路径也要写清楚

很多程序主流程写得很清楚，但失败分支一团糟。实际上高质量代码往往在错误处理上更见功力。

## 常见误区

### 出错了只改代码，不先定位原因

这会让问题反复出现。更好的顺序是：

1. 复现问题
2. 缩小范围
3. 观察变量
4. 修改并验证

### 忽略编译器警告

很多严重问题在警告阶段其实已经暴露了。

### 测试只测“正常输入”

更有价值的测试往往是：

- 空输入
- 极端值
- 非法值
- 边界值

## 学习建议

- 每写完一个函数，想一遍它的边界输入是什么。
- 遇到 bug 时，先构造最小复现样例。
- 在练习阶段就培养“先验证再修改”的习惯。
