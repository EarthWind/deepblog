# RGW 架构总览：S3/Swift 网关如何建立在 RADOS 之上

## 这篇文章要解决什么问题

到前面为止，我们已经把 Ceph 的块、文件、MON、MGR、OSD、BlueStore、CephFS 等主线讲了很多。

这时如果再从产品形态往回看，你会发现 Ceph 对外其实提供了三类很不同的访问方式：

- 块存储：`RBD`
- 文件存储：`CephFS`
- 对象存储：`RGW`

前两者比较好理解：

- `RBD` 提供块设备语义
- `CephFS` 提供文件系统语义

但一到 `RGW`，很多人的直觉就开始混乱。

因为它看起来像：

- 一个 S3 服务
- 一个 Swift 服务
- 一个对象存储网关
- 甚至有时被误以为“就是 Ceph 的对象存储本体”

这些理解都有一部分对，但都不够准确。

真正值得先回答的问题其实是：

- `RGW` 到底是 Ceph 里的哪一层？
- 它和底层 `RADOS` 的关系到底是什么？
- S3 bucket / object 和底层 RADOS object 是不是一回事？
- 为什么 RGW 既要处理 HTTP/S3/Swift，又要管理用户、桶、对象、ACL、策略和多租户？
- 它为什么不是简单“HTTP -> RADOS put/get”的薄代理？

如果只允许用一句话先给结论，那就是：

**RGW 不是 Ceph 的对象存储底座本身，而是建立在 `RADOS` 之上的对象网关层：它对外提供 S3/Swift 协议、认证鉴权、桶/对象命名空间和对象服务语义，对内把这些高层语义拆成用户元数据、bucket index、对象数据和控制逻辑，再落到底层 RADOS 对象、xattrs 和 omap 上。**

这一篇的目标，就是先把这张总图真正立起来。

## 先建立第一条边界：Ceph 的“对象存储”底座其实是 `RADOS`，不是 `RGW`

这是理解整篇文章的第一步。

很多人第一次接触 Ceph 对象存储时，会下意识觉得：

- RGW 就是 Ceph 的对象存储

这个说法不够准确。

更准确地说：

- `RADOS` 才是 Ceph 的通用对象存储底座
- `RGW` 是建立在 `RADOS` 之上的对象访问网关

### 为什么这个区分特别重要

因为这意味着：

- `RADOS` 只关心 pool、object、omap、xattr、复制、恢复、CRUSH 放置这些底层问题
- `RGW` 关心的则是 S3/Swift 语义：用户、bucket、ACL、认证、对象命名、版本化、多租户、生命周期这些高层问题

也就是说：

- **RGW 不是对象底座，而是对象服务层。**

这条边界一定要先立住。

## 第 1 层：为什么 `RGW` 最好理解成“对象协议与对象服务网关”

如果要给 `RGW` 下一个比较准确又不太绕的定义，我会这样说：

- 它是 Ceph 的对象协议网关与对象服务层

### 为什么要把这两个词放在一起

因为 `RGW` 同时做了两件事：

## 对象协议网关

- 对外说 HTTP
- 理解 S3
- 也理解 Swift

## 对象服务层

- 管用户
- 管 bucket
- 管 object 逻辑元数据
- 管 ACL / IAM / policy / 多租户
- 管 bucket index、版本化、生命周期、多站点同步等对象服务能力

### 这说明什么

说明 RGW 不是：

- 简单的“HTTP 转 RADOS”代理

而是：

- 在协议层和存储层之间，插了一整层对象服务语义

这点非常关键。

## 第 2 层：先把整张结构图立住

如果整篇只记一张图，我建议先记下面这张：

```text
S3 / Swift Client
  ->
HTTP Frontend
  ->
RGW 协议分发与认证
  ->
桶 / 对象 / 用户 / ACL / policy 等逻辑服务
  ->
bucket index / metadata / data 映射
  ->
RADOS (objects + xattrs + omap)
  ->
OSD / BlueStore
```

这张图最重要的意义在于：

- 它把 RGW 放到了“应用协议”和“底层对象存储”之间

也就是说，RGW 真正解决的不是：

- 数据怎么可靠落盘

而是：

- **S3/Swift 这些对象服务语义怎样被稳定地建在 RADOS 之上。**

## 第 3 层：为什么 S3 object 和底层 RADOS object 绝对不能直接画等号

这是理解 RGW 最容易踩坑的一点。

很多人第一次看到“Ceph 是对象存储”，自然就会觉得：

- S3 的 object，不就是 RADOS 的 object 吗？

这个理解很危险。

更准确地说：

- S3/Swift 层的 object 是 RGW 提供给用户的逻辑对象
- 底层 RADOS object 是 RGW 用来实现这些逻辑对象的物理对象

这两者既有关联，又绝不是一一等价的概念。

### 为什么

因为一个用户看到的对象，在 RGW 内部通常还会牵涉到：

- bucket index 记录
- 对象 head
- 可能的 tail 对象
- manifest
- xattrs 中的元数据
- 版本化或多段上传相关状态

### 所以最准确的理解方式是

- RGW object 是逻辑对象
- RADOS object 是底层实现材料

这条边界越早立住，后面越不会乱。

## 第 4 层：为什么 RGW 同时支持 S3 和 Swift，并不意味着底层有两套完全不同的存储

这也是一个很值得先讲的点。

从客户端视角看：

- S3 和 Swift 是两套不同协议

但从 Ceph 视角看，这两套协议最终都收束到：

- 同一套 RGW 对象服务逻辑
- 同一套底层 RADOS 存储材料

### 这意味着什么

意味着 RGW 真正厉害的地方不是：

- 同时把两个协议都“转发一下”

而是：

- 把两套外部协议映射到相对统一的内部对象服务模型

所以理解 RGW 时，更重要的不是背协议差异，而是建立下面这个认知：

- **协议层可以多样，但对象服务层和底层存储层要尽量统一。**

这就是 RGW 架构的价值之一。

## 第 5 层：为什么 RGW 的前端首先是一个 HTTP 服务进程

到了运行时，RGW 最直观的形态其实非常像一个 Web 服务。

因为对外它要处理的是：

- HTTP 请求
- URI
- headers
- 签名
- body

### 所以从进程视角看，RGW 首先做什么

- 启动前端
- 监听 HTTP
- 接受连接
- 把请求路由进自己的协议处理逻辑

### 这说明什么

说明 RGW 不是像 `CephFS` 那样以“客户端库协议”为中心，也不是像 `RBD` 那样以块 IO 为中心。

它首先更像：

- 一个对象存储 Web 服务进程

这也是为什么源码主入口会先围绕：

- frontend
- AppMain
- storage driver 初始化

来组织。

## 第 6 层：为什么理解 RGW 架构，一定要先分“前端协议层”和“后端存储层”

这是整篇最重要的结构分层之一。

如果你不先分层，很容易把 RGW 看成：

- 一个复杂但混成一团的大程序

更好的理解方式是先把它分成两大块：

## 前端协议层

- HTTP 前端
- REST 路由
- S3 / Swift / STS / IAM 等协议分发
- 认证与鉴权

## 后端存储层

- 用户/桶/对象元数据模型
- bucket index
- RADOS driver
- 底层对象、omap、xattrs 的读写

### 这意味着什么

意味着 RGW 的复杂度其实不是一种复杂度，而是两类复杂度叠在一起：

- 协议复杂度
- 存储映射复杂度

这点必须先分开。

## 第 7 层：为什么认证在 RGW 里不是边角逻辑，而是主结构的一部分

很多存储系统里，认证像是附加模块。

但在 RGW 里，它绝对不是边角逻辑。

### 为什么

因为 RGW 面向的是：

- S3 Access Key / Secret Key
- AWS Signature v2/v4
- Swift token
- Keystone
- LDAP / 外部身份源
- IAM / STS / 临时凭证

也就是说，认证不是“请求到了以后顺手验一下”，而是：

- 整个协议入口的核心组成部分

### 这说明什么

说明 RGW 从架构上就必须把：

- 协议处理
- 认证策略
- 身份映射
- 权限与策略应用

紧密地绑在一起。

所以理解 RGW，必须把认证当主线看，而不是附录看。

## 第 8 层：为什么 RGW 不能只靠 ACL，还必须有 IAM、STS、外部身份源

进一步讲认证/授权层时，很值得先立住一个认知：

- 对象网关面对的用户模型，比块和文件复杂得多

### 为什么

因为它不仅要支持：

- 单用户 bucket 访问

还要支持：

- 多租户
- 子用户
- access key
- bucket policy
- IAM policy
- 角色扮演
- 临时凭证
- 企业统一身份接入

### 这意味着什么

说明 RGW 从一开始就不是只做：

- “验个 S3 密钥”

而是在逐步长成：

- 一个完整的对象访问身份与策略系统

所以它比很多人想象得更像：

- 一个对象服务平台

而不只是网关。

## 第 9 层：为什么理解 RGW 元数据，必须先把它拆成三类

如果要理解 RGW 如何落到底层存储，我最建议先记住官方那条分类：

- metadata
- bucket index
- data

### 这三类分别在解决什么问题

## metadata

- 用户、bucket、实例信息等逻辑元数据

## bucket index

- bucket 里有哪些对象
- listing 怎么做
- 对象记录怎么维护

## data

- 真正的对象内容数据
- 以及对象 head/tail、manifest 等内容承载

### 为什么这个拆分特别重要

因为它直接说明：

- RGW 并不是把所有东西都塞成一种 RADOS 对象

而是明确分了：

- 身份与控制元数据
- bucket 维度索引
- 对象内容数据

这就是它能把高层对象语义建立起来的基础。

## 第 10 层：为什么 bucket index 是理解 RGW 的关键，而不是一个实现细节

很多初学者第一次接触 RGW 时，会把 bucket index 当成：

- 一个内部优化结构

其实它远比“优化”重要。

### 为什么

因为对象存储用户天然会问：

- 这个 bucket 里有哪些对象？
- 怎么分页列出？
- 对象新写入后怎样体现在列表里？
- 删除、版本化、多段上传后索引怎么收敛？

这些问题都不是底层 RADOS 自动替你解决的。

RGW 必须自己建立：

- bucket 级索引结构

### 所以最准确的说法是

- bucket index 不是锦上添花
- 它是 RGW 从“裸对象集合”长成“对象存储服务”的关键部件之一

这点一定要立住。

## 第 11 层：为什么对象 head、tail、manifest 这些词在 RGW 里反复出现

一旦进入对象数据层，你会经常看到这些词：

- head object
- tail object
- manifest

这通常会让人疑惑：

- 一个对象为什么还要拆这些层？

### 为什么会这样

因为 RGW 既要对外提供“一个逻辑对象”的体验，又要在内部处理：

- 大对象切分
- 多段上传
- 元数据与数据分离
- 版本化
- 高效读写与恢复

所以一个高层对象，在内部经常不是单一物理对象，而是：

- 一个 head 承载关键元数据和指针
- 若干 tail 承载实际数据段
- manifest 描述整体布局

### 这说明什么

说明 RGW 的对象实现从一开始就是：

- 面向对象服务语义设计的

而不是简单“把上传文件直接存成一个底层 object”。

## 第 12 层：为什么 RGW 本质上是在 `RADOS` 的 objects / xattrs / omap 之上“长出”对象服务

这是整篇最值得反复强调的一条主线。

`RADOS` 原生理解的东西其实很基础：

- object
- xattr
- omap
- pool

但对象存储服务需要的东西远远更多：

- user
- bucket
- listing
- object metadata
- access control
- multipart upload
- versioning
- lifecycle

### RGW 怎么办

它并没有要求底层 RADOS 直接理解这些概念，而是选择：

- 在 RADOS 现有原语之上构建出自己的对象服务模型

### 所以最准确的理解方式是

- RGW 是一个“语义上浮层”

也就是：

- 在底层对象原语之上长出更高层的对象存储语义

这就是它的架构价值。

## 第 13 层：为什么源码里会先看到 `frontend`、`auth`、`process`、`sal` 这几类入口

如果从源码视角去看 RGW，总入口最容易先遇到下面几类东西：

- `rgw_main` / `AppMain`
- `frontend`
- `auth`
- `process`
- `sal`

### 这几个词大致分别在回答什么

## `frontend`

- 请求怎么进来

## `auth`

- 身份怎么识别、权限怎么套用

## `process`

- 一个请求怎样被分发、初始化、执行、完成

## `sal`

- RGW 的抽象存储层怎样对接具体后端

### 为什么这个分法很有价值

因为它几乎就是 RGW 架构的源码版目录：

- 从入口收流量
- 到认证
- 到操作处理
- 到后端对象访问

这条线非常适合后面继续深挖。

## 第 14 层：为什么 `SAL` 这层值得特别注意

如果只看架构总览，这里先立一个直觉就够了：

- `SAL` 不是边角层

### 为什么

因为它说明 RGW 的业务逻辑并不想直接和某一种后端实现死绑。

更准确地说：

- RGW 上层逻辑尽量面向抽象存储接口
- 再由具体 driver 实现，比如 RADOS driver

### 这意味着什么

意味着 RGW 架构上至少在努力做一件事：

- 把对象服务逻辑和底层存储实现解耦

即使今天最核心、最常见的后端仍然是：

- RADOS

这层抽象依然很重要。

## 第 15 层：为什么说 RGW 不是“没有 MDS 的 CephFS”，也不是“HTTP 版 librados”

这是两个特别常见的误解。

### 不是“没有 MDS 的 CephFS”

因为 CephFS 的核心问题是：

- POSIX 文件系统语义
- 目录树
- inode/dentry/caps

RGW 的核心问题则是：

- bucket/object 命名空间
- 对象协议
- 认证与多租户
- bucket index 与对象元数据组织

### 也不是“HTTP 版 librados”

因为 `librados` 只给你底层对象原语，不会帮你解决：

- 用户系统
- bucket
- ACL / IAM
- listing
- multipart
- versioning

### 所以 RGW 真正的位置是什么

- 它是 Ceph 对象服务语义的实现层

这点一定要明确。

## 第 16 层：把 RGW 放回 Ceph 全局图里，它到底像什么

如果把 RGW 放回整个 Ceph 体系里，它最像的是：

- 一个建立在 RADOS 之上的应用网关/服务层

和前面讲过的：

- `RBD` 把块语义建在 RADOS 上
- `CephFS` 把文件语义建在 RADOS 上

是同一种大思路。

只是 RGW 建立的是：

- S3/Swift 对象服务语义

### 所以 Ceph 的一条大主线就很清楚了

- 同一个 `RADOS`
- 上面可以长出块、文件、对象三种完全不同的上层服务

这就是 Ceph 设计特别有代表性的地方。

## 第 17 层：如果只记一张 RGW 总图，应该记什么

如果这一篇只记一张图，我建议记下面这张：

```text
S3 / Swift Client
  ->
HTTP Frontend
  ->
RGW REST 路由
  ->
认证与授权
  ->
RGW 操作执行层
  ->
用户 / bucket / object 逻辑模型
  ->
metadata / bucket index / data
  ->
RADOS objects + xattrs + omap
  ->
OSD / BlueStore
```

这张图里最关键的一句话是：

- **RGW 把对象协议和对象服务语义，翻译成了 RADOS 能承载的底层对象结构。**

## 第 18 层：为什么这一篇要先讲“总览”，而不是一上来就讲一次 PUT/GET 请求

因为 RGW 比很多模块更容易让人一开始就陷进细节。

如果一上来就看：

- 一个 HTTP 请求怎么进前端
- 怎么验签
- 怎么选 `RGWOp`
- 怎么打到 RADOS

你很容易只看到：

- 一条请求处理流水线

却看不到：

- RGW 作为“对象服务层”的整体角色

所以这一篇最重要的目标不是：

- 讲完全部细节

而是先建立这几个关键直觉：

- RGW 在 Ceph 哪一层
- 它不是底层对象存储本体
- 它怎样把协议、认证、元数据和 RADOS 拼在一起

只有这张图立住了，后面再看请求链、bucket index、多站点同步才不容易乱。

## 用一句话重新概括这篇

如果把这篇全部内容压缩成一句尽量准确的话，我会这样说：

**RGW 是 Ceph 建立在 `RADOS` 之上的对象网关与对象服务层：它对外处理 S3/Swift 协议、HTTP 请求、认证和访问控制，对内维护用户、bucket、object、bucket index 和对象元数据这些逻辑模型，再把它们落成底层 RADOS 的对象、xattrs 和 omap；因此 RGW 既不是底层对象存储本体，也不是简单的 HTTP 转发器，而是 Ceph 对象语义真正成立的关键中间层。**

## 把整篇压缩成一条最短骨架

如果你只想记一条骨架，可以记成下面这样：

```text
S3/Swift 请求进入 RGW
  ->
前端和 REST 路由选择协议处理器
  ->
认证和权限系统确定身份
  ->
RGW 在内部处理用户/bucket/object 逻辑
  ->
metadata / bucket index / data 分层落到 RADOS
  ->
最终由 OSD/BlueStore 持久化
```

只要这条骨架记住了，RGW 的总体结构就不会乱。

## 初学者最容易混淆的 10 个点

### 1. 认为 RGW 就是 Ceph 的对象存储底座

不对。底座是 `RADOS`。

### 2. 认为 S3 object 等于 RADOS object

不对。一个是逻辑对象，一个是底层实现对象。

### 3. 认为 RGW 只是 HTTP 转 RADOS 的薄代理

不对。中间还有完整对象服务语义层。

### 4. 认为 S3 和 Swift 在 RGW 里是两套完全独立后端

不对。协议不同，但内部对象服务层和底层存储层尽量统一。

### 5. 认为认证只是附加功能

不对。认证与授权是 RGW 主结构的一部分。

### 6. 认为 bucket index 只是性能优化

不对。它是对象 listing 与对象服务成立的关键结构。

### 7. 认为一个对象一定对应一个底层物理对象

不对。还可能涉及 head、tail、manifest 等布局。

### 8. 认为 RGW 和 CephFS 只是“接口不同”

不对。它们解决的是完全不同的服务语义问题。

### 9. 认为 `librados` 已经足够做对象存储服务

不对。`librados` 不解决 bucket、ACL、IAM、listing 等高层语义。

### 10. 认为理解 RGW 只要盯 HTTP 请求流

不对。还必须先立住它的元数据和底层映射模型。

## 这一篇最应该留下的 5 个直觉

### 直觉一：`RADOS` 是对象底座，`RGW` 是对象服务层

这是第一原则。

### 直觉二：RGW 同时处理协议复杂度和存储映射复杂度

这两层都不能漏。

### 直觉三：认证、ACL、IAM、STS 在 RGW 里是核心结构

不是附属模块。

### 直觉四：bucket index 和对象元数据是 RGW 架构的关键骨架

这点必须立住。

### 直觉五：RGW 的价值在于把对象协议语义稳定地建在 RADOS 上

这是它的核心意义。
